# 수집 소스 관제 화면 + 추가 개선축 (분석/기획, 구현 없음)

## A. "수집 소스" 화면 설계 (대표 제안 — DB에 있는 링크를 보여주고·수정·관리)

### A-1. 문제 (한 줄)
돈 주고 모은 자산(경쟁사 가격 소스 URL 3계층)이 DB에 **이미 있는데**, placeholder 입력칸 뒤에 숨어 회사에서 **아무도 존재를 모른다.** 보이지 않는 자산은 관리되지 않고, 관리 안 되는 소스는 썩는다(URL 만료·구조 변경·수집 실패 방치).

### A-2. 위치·명칭 — `/pricing/gpu?tab=sources` "수집 소스" 독립 탭
경쟁사 탭 내 섹션이 아니라 **독립 탭**. 이유: 대상이 경쟁사 하나가 아니라 3계층(경쟁사 페이지 / 모델별 URL / 시장가 소스) + 향후 공급사까지. 경쟁사 탭에 붙이면 공급사 소스 넣을 곳이 없어 또 갈라진다.

### A-3. 정보구조 — 소스 레지스트리 단일 뷰
기존 3테이블(`competitors.pricing_url` / `competitor_product_mapping.competitor_url` / `market_prices.source_url`)을 **하나의 논리 뷰**로 합쳐 표시(사용자에게 테이블 경계 숨김).

| 컬럼 | 내용 |
|---|---|
| 소스 유형 | 경쟁사 가격페이지 / 모델별 URL / 시장가 소스 (NbBadge) |
| 대상 | 경쟁사명·모델명 (각각 상세로 이동 — 3조) |
| URL | 도메인 축약 + 원본 복사 |
| 마지막 수집 | 일시(KST SSOT) + 결과(성공/실패/변동감지) |
| 상태 | 정상 / 실패 N회 연속 / 미검증 / 삭제됨 (상태색 토큰 SSOT) |
| 자동수집 | on/off 토글(행 단위) |

상단 필터: 소스유형·경쟁사·상태(**"실패만 보기"** 원클릭)·자동수집 여부. 필터·정렬·페이지 URL 상태.

### A-4. 핵심 기능 5
1. **추가/수정/삭제(소프트)** — 소프트삭제+audit 편입(9조), 인라인 편집(2조).
2. **"지금 긁어보기"(링크 테스트)** — market/refresh 엔진 재사용하되 **즉시 덮어쓰기 금지 → 추출 후보→사용자 확인→반영(§5-3 패턴)**. *이 화면의 가장 중요한 설계 결정.* 현재의 무검토 즉시 덮어쓰기가 오염 경로.
3. **자동수집 토글** — 행 단위 + **일괄 on/off(16조)**. 이 토글이 곧 **크론/첫접속트리거의 대상 목록** — 10조 "무엇에 자동수집을 걸지"의 답이 여기서 관리됨.
4. **실패 링크 필터 + 사유** — "왜 실패했나"를 사람 말로(8조): URL만료/파싱실패/차단/타임아웃. 3회 연속 실패 시 배지 승격.
5. **일괄 등록** — 모델 URL 다건 붙여넣기(16조).

### A-5. 이 화면의 진짜 정체 = 수집 파이프라인 관제탑
링크 CRUD는 입구. 자연 흡수 대상: ①스케줄 관제(gcube 크론을 "자동수집 on 소스 전체"로 일반화 시 대상·주기·최근결과를 보는 곳, GcubeSyncBadge 흡수) ②검수 큐(자동수집 켜지면 "긁어온 값 검토 대기열"의 집) ③공급사 확장(suppliers 소스 개념 신설 시 여기에 유형 추가 → 경쟁사·공급사 수집 한 화면 통합).
> **결론: "링크 관리 화면"이 아니라 "가격 데이터가 어디서 오는지에 대한 단일 진실 화면". 안 만들면 10조 자동화는 대상 목록 없는 자동화가 된다.**
> ⚠️ 재수집을 무검토 덮어쓰기로 두면 이 화면이 "오염 가속 버튼"이 됨 — 후보 검토 패턴 결합 필수.

## B. 대표가 아직 안 짚은 추가 개선축 (헌법 비중복 7)

| # | 개선축 | 왜 필요 | 난이도/효과 |
|---|---|---|---|
| 1 | **정합성 린터(데이터 헬스 스코어)** | tier 이중저장·model_name 고아·통화 우회를 매일 자동 스캔→"정합성 점수+위반목록+원클릭 정리". 확인된 이슈 6개+가 전부 사람이 우연히 발견 — 발견 체계 없으면 재발. 골든셋/kst-guard 패턴을 데이터에 적용 | 중/**상** 재발방지 기반 |
| 2 | **소스 신뢰도 등급** | 소스별 성공률·변동빈도·이상치율 누적→A/B/C + 이력 타임라인. "언제 긁었나(신선도)"와 별개로 "이 소스를 믿어도 되나"를 아무도 모름. A-3 위에 컬럼 2개 얹는 수준 | 하~중/상 |
| 3 | **가격 변동 알림/구독** | 관심 GPU watch→경쟁가·시장가·마진 임계 이탈 시 앱내 알림. 지금은 콕핏 들어가야만 앎 — 영업이 견적 내는 날 경쟁사 인하 모르면 진 견적. 시장 이벤트 push | 중/**상** "조회 도구→먼저 알려주는 도구" |
| 4 | **Export·리포트 자동생성** | 콕핏 스냅샷 xlsx/pdf + 주간 가격동향 리포트(주간보고 AI 패턴 재사용). 현재 Export 사실상 없어 대외 공유가 스크린샷 — 신뢰 갉아먹는 신호 | 하~중/중상 |
| 5 | **협업 맥락(last-touched-by + 셀 코멘트)** | 값 옆에 "누가 언제 마지막 수정"+값단위 메모. 감사로그는 사후추적, 이건 보는 순간 신뢰판단. 감사백본(v0.7.289) 있어 표시레이어만 | 하/중 데이터 이미 있음 |
| 6 | **가격결정 회귀 코퍼스 + 롤백 훈련** | 골든셋 eval을 추출→가격결정 전구간 확장, 재설계 전후 동일입력 비교. **이번 재설계 최대 리스크=tier·통화 손대는 순간 기존 가격이 조용히 바뀌는 사고.** 스냅샷 테이블 있으나 복원 실전경험 없음 | 중/**상** 재설계의 보험 |
| 7 | **콕핏 온보딩 + 엑셀형 키보드 UX** | driver.js 온보딩을 콕핏에 재사용 + 방향키/Enter편집/복붙. 콕핏이 가장 복잡한데 안내 0, 담당자 비교대상은 늘 엑셀 — 마우스강제 그리드는 "엑셀보다 느림" 인상 하나로 채택 붕괴 | 중/중 |

### Top 3 우선순위 (단정)
1. **#6 회귀 코퍼스 + 롤백 훈련** — 재설계의 **전제조건**. tier·통화·정렬 건드리기 전 무조건 첫 번째. 없으면 "고쳤는데 가격 바뀜" 사고 통계적 확실.
2. **#1 정합성 린터** — 드러난 이슈 6개+는 빙산의 일각. 발견을 사람 운에 맡기는 구조를 끝내야 재설계 결과가 유지됨.
3. **#2 소스 신뢰도 등급** — A(소스 관제탑) 위에 거의 공짜로 얹힘, 10조 자동수집 품질 좌우. **대표 제안(소스 관제)과 같은 스프린트로 묶기.**

> #3(알림)·#4(Export)는 효과 크나 위 3개로 데이터 신뢰 확보 **다음**. 틀린 값을 push·export하는 것은 화면에 두는 것보다 나쁘다.

## C. 로드맵 반영
- **소스 관제 탭 + #2 신뢰도등급**: Phase 2(인입 재편) 스프린트에 묶음(대표 제안 = 자동화 10조의 선결 화면).
- **#6 회귀 코퍼스**: **Phase 0로 승격**(감사백본과 함께 재설계 착수 전 안전망). tier/통화/정렬 수정보다 먼저.
- **#1 정합성 린터**: Phase 1(데이터계약) 산출물에 편입 — 계약 위반을 상시 감시.
- **#3·#4·#5·#7**: Phase 4(IA통합·표면정리) 이후 부가 가치.

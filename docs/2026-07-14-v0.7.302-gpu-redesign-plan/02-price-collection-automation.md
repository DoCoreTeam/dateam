# 가격 수집 자동화 설계 (제10조 구체 메커니즘) — 분석/기획, 구현 없음

> 대표 지시: "링크를 주면 시스템이 보관했다가 주기적으로 확인(재수집)하라." 이 문서는 그 **"어떻게"** 를 코드 실태 위에서 구체화한다.

## 0. 핵심 결론 — 부품은 이미 있다, 스케줄러만 없다

현재 상태를 3단계로 실측한 결과:

| 단계 | 상태 | 근거 |
|---|---|---|
| ① 링크 저장 | ✅ 있음(경쟁사) | `competitors.pricing_url`, `competitor_product_mapping.competitor_url`, `market_prices.source_url/source_type` |
| ② 링크로 재수집(AI 파싱) | ✅ 있음(수동 트리거) | `POST /api/pricing/gpu/market/refresh` — safe-fetch(SSRF가드)→HTML정리→**Gemini 범용 파싱**→`saveCompetitorPrices` SSOT 저장 |
| ③ 주기 자동 실행 | ❌ **없음** | 이 route를 호출하는 크론/스케줄 0건. "새로고침" 버튼 클릭 때만 동작 |
| ④ 변경 확인 동선 | ⚠️ 약함 | 경쟁사는 즉시 자동 반영 후 무통보(검토 큐 미경유). gcube만 배지 존재 |

> **즉 "링크 주면 확인"의 재수집 엔진(②)은 이미 동작한다. 빠진 건 (③)그걸 매일 자동으로 눌러주는 손과 (④)바뀐 걸 사람에게 알리는 알림뿐이다.** 대표님이 "안 됐다"고 느낀 이유 = 사람이 버튼을 눌러야만 도니까, 그리고 링크 입력칸이 설명 없이 숨어 있으니까.

## 1. 목표 사이클 (링크 등록 → 주기 재수집 → 변경 확인)

```
[사용자] 경쟁사/공급사에 소스 URL 등록 (1회)
   │  (제10-A: "이 링크는 매일 자동 확인됩니다" 명시)
   ▼
[스케줄러] 매일/매시 크론 → 등록된 모든 활성 링크 순회
   │  (기존 market/refresh 엔진 재사용 + safe-fetch SSRF가드)
   ▼
[AI 파서] 페이지 텍스트 → 가격 추출 (Gemini 범용, 이미 존재)
   │
   ▼
[변경 감지] 기존 값과 비교
   ├─ 동일/무변화 → as-of(최근수집일)만 갱신, 조용히
   ├─ 변동 발생 → 검토 큐(review_items)로 "old→new" 카드 (제8조: 무엇이 어떻게 바뀜)
   │             + 신뢰도 높으면 자동확정(제4조 confidence-gate), 낮으면 사람 확인
   └─ 수집 실패 → 버리지 않고 로그+"마지막 성공 N일 전" 배지 (제9조)
   ▼
[화면] 각 링크에 "🟢 자동수집 중 · 마지막 확인 3시간 전" 상태 상시 표시
```

## 2. 재활용할 기존 자산 (새로 만들 게 적다)

| 자산 | 역할 | 상태 |
|---|---|---|
| `market/refresh/route.ts` | URL→AI파싱→저장 엔진 | ✅ 그대로 재사용(크론이 호출만) |
| `lib/gpu/competitor-import.ts` `saveCompetitorPrices` | 저장 SSOT | ✅ |
| `lib/security/safe-fetch.ts` | SSRF 방어 fetch | ✅ 필수 재사용 |
| `.github/workflows/gcube-price-check.yml` | **크론 워크플로 템플릿** | ✅ 복제해 refresh 호출용으로 |
| `competitors.pricing_url` 등 | 링크 저장 스키마 | ✅ 이미 존재 |
| `GcubeSyncBadge` + `status-colors` GCUBE_SYNC | "변경 확인" 배지 패턴 | ✅ 경쟁사 링크에도 확장 |

| `review_items` + restore 백본 | 변경 검토·유실0 | ✅(제9조와 공유) |

## 2-B. 트리거 방식 결정 — 크론 대신 "KST 00시 이후 첫 접속자 구동" (대표 지시)

> **왜 크론이 아니라 첫 접속자 트리거인가**: gcube 크론(`gcube-price-check.yml`, 매일 KST 00시)은 코드상 "돌 수 있는 상태"이나 실제 작동을 신뢰하기 어렵다 — ①GitHub Secrets 실등록 여부 코드로 확인 불가 ②60일 무활동 시 크론 자동 중단(GitHub 정책) ③**실패해도 알림 채널 없음**(조용히 안 돎). 대표 지시대로 **그날 처음 접속한 사용자의 요청이 하루치 수집을 1회 구동**하고, 이미 돌았으면 스킵하는 방식이 이 프로젝트에 더 견고하다.

**재사용 레퍼런스 = 환율 동기화(fx)**: `GpuPricingClient.tsx:185-194` — `fxDate !== KST오늘`이면 그날 첫 접속자가 `POST /api/pricing/gpu/fx` 1회 호출, 서버는 `fx_rates.rate_date` upsert로 멱등 보장(오늘자 있으면 클라가 아예 호출 안 함). **이 패턴이 "첫 접속자 트리거+DB일자 멱등"의 검증된 원형이다.**

**단, fx를 그대로 베끼면 안 되는 3가지 (market/refresh는 fx보다 무거움)**:
| 요소 | fx 상태 | market/refresh 필요 |
|---|---|---|
| 멱등(마지막 수집일 저장) | ✅ `fx_rates.rate_date` | ❌ 없음 → `market_refresh_runs.run_date` 신설 |
| 경합 방지(동시 접속 중복 실행) | ⚠️ 락 없음(fx는 upsert 수렴이라 무해) | ❌ 필수 신설 — URL×AI 중복 실행 방지 `pg_try_advisory_lock` 또는 run_date unique+선점 |
| 비차단(백그라운드) | ✅ 가벼워 무관 | ❌ **필수** — 현재 URL수×(fetch+AI)를 **동기 루프**로 처리(refresh/route.ts:132-178) → 트리거한 사용자 화면이 그대로 멈춤. fire-and-forget+폴링 또는 Vercel `after()` 필요 |
| 권한 | admin 전용 | 첫 접속자가 member일 수 있음 → 트리거는 **서버가 대행**(사용자 권한과 분리)해야 member 접속으로도 구동 |

## 3. 새로 붙일 결손 부품 (최소)

1. **첫 접속자 트리거 배선(③ — 핵심 결손)**: fx 패턴을 복제하되 위 3요소 보강. GpuPricingClient(또는 앱 진입 공통부)에서 `lastRefreshDate !== KST오늘`이면 `POST /market/refresh?auto=1` 1회 발화 → 서버가 **① run_date 선점(경합 방지) → ② 백그라운드로 링크 순회 → ③ 완료 시 run_date 확정**. 트리거는 member 접속도 가능하도록 서버가 서비스롤로 대행. **크론(gcube-price-check.yml)은 폐기하지 않고 "백업 경로"로 병존**(첫 접속자가 늦는 날 대비) — 단 둘 다 같은 run_date 멱등을 공유해 이중 실행 방지.
2. **수집 상태 필드**: 링크별 `last_checked_at`·`last_status`·`last_error` (as-of·성공/실패). gcube의 `gcube_price_checks` 패턴을 경쟁사 링크로 일반화.
3. **변경 → 검토 큐 라우팅(④)**: 현재 경쟁사 refresh는 값을 **즉시 덮어씀**(검토 미경유). 이를 제9조/제4조에 맞춰 "변동분은 review_items로, 신뢰도 높으면 auto-confirm"으로 전환. 조용한 자동 덮어쓰기 → 추적가능한 변경으로.
4. **공급사 소스 링크(신규)**: 현재 링크 저장은 경쟁사만. 공급사도 견적 페이지/포털 URL을 등록·재수집하려면 `suppliers` 쪽 소스 URL 개념 신설(경쟁사 스키마 미러링).
5. **UI 명시(제10-A)**: 링크 입력칸에 "여기 넣으면 매일 자동 확인" 안내 + 링크별 "자동수집 중·마지막 확인 시각" 배지. 지금은 placeholder만 있어 사용자가 기능 존재를 모름.

## 4. 헌법 정합
- **제10조(수집 자동화)**: 본 설계가 제10조의 실행체. gcube 단일 → 사용자 등록 링크 일반화.
- **제9조(유실0)**: 수집 실패도 버리지 않고 로그+배지. never-block.
- **제8조(교정 안내)**: 파싱 실패 시 "어느 URL이 왜 실패(타임아웃/차단/형식)"를 명시, 재시도·URL수정 동선.
- **제4조(AI 자동)**: 변동분 신뢰도 높으면 자동확정, 낮으면 검토 큐 — 사람은 확인만.
- **제12조(마진 가드레일)**: 자동 수집된 경쟁가가 우리 판매가 하한을 침범하면 경고.
- **제14조(파급 통지)**: 자동 수집으로 값이 바뀌면 영향 SKU 마진 재평가+통지.

## 5. 리스크
- ⚠️ 자동 크롤은 **대상 사이트 구조 변경·차단(robots/rate-limit)** 에 취약 → 실패를 조용히 넘기지 말고(제9조) "마지막 성공 N일 전" 가시화가 필수. 실패가 쌓이면 알림.
- ⚠️ AI 파싱은 오추출 가능 → 즉시 자동 덮어쓰기 금지, 검토 큐/자동확정 임계(제4조)로 완충.
- ⚠️ SSRF/과도 요청 → safe-fetch 가드 유지, 주기·동시성 제한.
- ⚠️ 법적/약관: 경쟁사 페이지 자동 수집의 허용 범위는 사업 판단 필요(대표 확인 사항).

## 6. 착수 순서(제안, 구현 아님)
1. 수집 상태 필드(2) + 변경→검토 라우팅(3) — 데이터/안전장치 먼저
2. 스케줄러 배선(1) — gcube 템플릿 복제
3. UI 명시·배지(5)
4. 공급사 소스 링크(4) — 경쟁사 검증 후 확장

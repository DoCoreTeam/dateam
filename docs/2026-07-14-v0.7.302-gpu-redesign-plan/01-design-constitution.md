# GPU 가격관리 설계 헌법 (v2.0 — 16조)

> **전문(前文)** — 기능은 이미 충분히 구현되어 있다. 이 재설계의 목표는 기능 추가가 아니라 **복잡도발 오류의 제거**다. 기존 자산(confidence-gate·audit_log·restore-action·resolveCompetitorId·expiry·unified-cost-pick·impact·golden-set 등)은 **버리지 않고 재배선**한다. 입력부터 가격 결정까지 전 구간이 아래 16개 조문을 예외 없이 지킨다. 모든 설계 결정·코드 리뷰·완료 판정은 이 헌법을 기준으로 한다.
>
> **2부 구성**: 1~9조 = 대표(CEO) 구술 철학 = **입출력의 헌법**(데이터를 잘 받아 잘 보여주고 잃지 않는다). 10~16조 = 🟦 DC-BIZ(fable5) 확장 사고 = **결정의 헌법**(그 데이터로 내린 가격이 지금도 옳고, 왜 옳은지 증명 가능하고, 틀리면 돈 나가기 전에 시스템이 먼저 잡는다). 정식화: 🟦 DC-BIZ(fable5). 조문 개정은 대표 승인 사항.

# 제1부 — 입출력의 헌법 (대표 9원칙)

### 제1조 — 입력 정형화: 선언 없는 인입 금지
- **조문**: 모든 인입은 "무엇을 넣는지"(공급사/경쟁사 등 대상 주체)를 먼저 선언한 뒤 시작. 시스템이 추측으로 분류하지 않는다.
- **왜**: 4단 추측(키워드→AI→화이트리스트→fallback)은 확률 분류. 입구에서 틀리면 하류 전체 오염. 복잡도발 오류 제1 진원.
- **합격**: ①진입점에 대상 주체 선택 **선행 필수 필드** ②분류 추측 코드 경로 **0개** ③preview '→경쟁사' 재분류 버튼 **부재**.
- **현재 위반**: 통합입력 4단 추측, 재분류 버튼=자동판정 불신 물증.

### 제2조 — 없으면 만들게: 빈 화면 금지
- **조문**: 0건 화면은 빈 상태로 끝내지 않고 "만들기/가져오기" 동선 제시.
- **합격**: ①0건 화면은 **공용 EmptyState** + **primary 액션 버튼 1개↑** ②"빈배열→null/빈div" 정적스캔 **0건**.
- **현재 위반**: 공용 빈상태 규칙 부재 → 전 탭 미보장.

### 제3조 — 누르면 바로 이동: 막다른 길 금지
- **조문**: 클릭=즉시 연결. 표시만 되고 아무 일 없는 요소 금지.
- **합격**: ①모든 엔티티 참조(공급사·모델·경쟁사명) 클릭 시 연결, **클릭불가 텍스트 0건** ②모든 리스트 어포던스 4종(검색·정렬·필터·페이지네이션).
- **현재 위반**: 탭마다 제각각. SuppliersTab 정렬 미구현.

### 제4조 — AI 자동 채움: 사람은 확인만
- **조문**: AI가 채울 수 있는 건 AI가. 사람 일은 입력 아닌 확인.
- **합격**: ①confidence-gate 3밴드(auto/review/block) **실처리 배선**(auto=자동확정+감사, review=큐, block=반려). "표시전용" 금지 ②수동검토율 지표화+auto 자동처리율 목표.
- **현재 위반**: 3밴드 표시전용, 100% 수동검토.

### 제5조 — 표시 일관성: 같은 값은 어디서든 같게
- **조문**: 하나의 값은 어느 화면에서든 같은 값·같은 형식.
- **합격**: ①표시 변환 `lib/` **SSOT 1개**, 인라인 복붙 0건 ②동일의미 **이중소스 0쌍**(이중 컬럼은 파생·우선순위 명문화+단일 판정함수) ③두 화면 렌더값 비교 **일관성 테스트**.
- **현재 위반**: tier `gpu_products.tier` vs `supplier_model_tier` 이중, 가격결정 구뷰/신뷰 중복, unified 숨김탭 라우트 생존=이중 IA.

### 제6조 — 입력형태 그대로 표시: 원으로 넣으면 원으로
- **조문**: 원화는 원화로, 달러는 달러로 — 넣은 모양이 기본 표시.
- **합격**: ①기본 표시=`original_currency` 참조, 하드코딩 통화기호 정적가드 **0건** ②환산은 **원본 병기(배지)** 와 함께만.
- **현재 위반**: 공급사모달 하드코딩$, 가격표상세 전역토글. 둘 다 original_currency 미참조.

### 제7조 — 구조화 반영 + 원본 절대 보존
- **조문**: 우리 구조(FK·정규화)로 정리하되 원본은 절대 그대로.
- **합격**: ①원본 필드(`original_*`·raw) **불변** ②구조 반영 **FK 정합**, 텍스트조인 0건 ③모든 자동 변형(AI 포함) **감사로그+되살리기** 가능.
- **현재 위반**: 원본 보존 OK, 구조 반영이 텍스트조인이라 연결 끊김 — 절반 준수.

### 제8조 — 실수를 정확히 교정 안내한다: 에러는 해결 동선이다
- **조문**: 실수 시 ①무엇을(정확한 항목·값) ②왜 ③어떻게 제대로(예시·다음 행동)까지. "오류가 발생했습니다"는 금지어.
- **합격**: 모든 에러가 3요소 — (a)틀린 항목 특정(행·필드·입력값) (b)올바른 형식/예시 (c)원클릭 교정 동선. **필드 미특정 토스트 0건**. 대량 인입 부분실패 시 **행 단위 실패 리포트**(성공N/실패M+행별 3요소), 전체롤백·전체성공 위장 금지.
- **현재 위반**: 통합입력 실패 행·사유 미특정(추정), 화면마다 제각각.

### 제9조 — 절대 멈추지도 버리지도 않는다: 전부 남긴다
- **조문**: 실패·거부·미분류도 (a)저장 안 막고(never-block) (b)버리지 않고 (c)로그·이력·감사로 남긴다. "무엇이 왜 실패했고 어떻게 개선하는지" 항상 추적·복원 가능.
- **정합**: 회사 유실0·마이그146 audit_log·restore-action과 일치. **기존 백본의 GPU 전 화면 적용 확대**(신규 발명 아님).
- **합격**: ①모든 인입·확정·거부·에러 audit 로그 **append(성공·실패 전부)** ②거부/실패도 삭제 아닌 상태보존+소프트삭제, 물리DELETE 0건 ③**되살리기 동선**(항목단위·화이트리스트) ④**로깅 실패가 저장 안 막음**(아웃박스/비차단).
- **현재 위반**: 회사 백본은 있으나 GPU 도메인 배선 부분적(추정).

# 제2부 — 결정의 헌법 (확장 7원칙, fable5)

### 제10조 — 신선도 시효: 가격 데이터는 유통기한이 있다
- **조문**: 모든 원가·시세는 수집 시점을 필수 보유, 시효 경과 시 자동 "부패(stale)" 표시, 부패값은 가격결정 입력에서 자동 강등.
- **왜 위험**: GPU 시장가는 주 단위로 움직인다. 3개월 전 견적이 오늘 견적과 같은 얼굴로 콕핏에 앉으면 대표가 썩은 원가 위에서 판매가 결정. **유실보다 무서운 게 "멀쩡해 보이는 낡은 숫자".**
- **합격**: ①모든 가격 행 as-of 시점 표시 ②시효(시세30일·견적90일 등) 초과 시 자동 배지 ③부패값만으로 산출된 추천가에 경고 ④stale 비율 지표.
- **현재**: 부분 — `expiry.ts`(견적 유효기간) 有, market_prices 시효·콕핏 강등·stale 지표 미확인.

### 제11조 — 가격 결정 근거 스냅샷: 왜 이 가격인지 역추적 *(확장 Top1)*
- **조문**: 판매가(strategic_price) 확정 순간, 결정에 쓰인 원가·경쟁가·환율·마진규칙·결정자를 스냅샷으로 동결 저장, 모든 판매가에서 "왜 이 가격" 원클릭 역추적 가능.
- **왜 위험**: 지금 감사로그는 "값이 바뀌었다"만 기록, "왜 이 값인가"는 없다. 6개월 뒤 "H100 왜 이 가격에 팔았나"에 답 못 하면 협상·손익검증·책임소재가 공중에 뜬다. **근거 없는 숫자는 대표 본인도 못 믿는다.**
- **합격**: ①확정 시 입력값 스냅샷(원가id·경쟁가id·환율·마진%) 동결 ②판매가 클릭→"당시 근거" 화면 ③입력 원본이 이후 수정/삭제돼도 스냅샷 불변.
- **현재**: 없음(추정) — `audit.ts`·`price-breakdown.ts`는 변경기록·현재분해만.

### 제12조 — 마진 가드레일: 바닥 아래로는 시스템이 막는다 *(확장 Top2)*
- **조문**: 판매가는 실효원가(cost_basis) 대비 최저 마진 하한선을 갖고, 하한 미달가는 저장 전 강제 경고+admin 명시 확인 없이 확정 불가.
- **왜 위험**: 우리는 gcube 실매출 사업자. 오타·환율 착오 하나면 **원가 이하 판매가가 그대로 대외 게시**. 입력 오류는 제8조가 잡지만 "숫자는 멀쩡한데 사업적 손실인 가격"은 아무도 안 잡는다. **v0.7.217 cost_min 사고가 정확히 이 유형.**
- **합격**: ①GPU별/전역 최저 마진율 설정 ②하한 미달 저장 시 차단성 경고(사유 입력 후 통과) ③하한 미달 게시중 SKU 목록 상시 조회.
- **현재**: 없음(추정) — `price-signal.ts`는 신호 표시용, 저장 차단 게이트 아님.

### 제13조 — 대표값 결정 규칙 명문화: 여러 값 중 무엇이 진실인가
- **조문**: 같은 GPU에 견적·시세 여럿일 때 대표값 규칙(최신 우선·유효기간 내·지정값 최우선)은 코드 한 곳(SSOT) 명문화, 화면은 대표값과 함께 "왜 이것이 대표인지" 표시.
- **왜 위험**: 규칙이 화면마다 다르면 콕핏=최저가/리포트=최신가로 제5조가 값 차원에서 붕괴. cost_min vs cost_basis 사고의 전조.
- **합격**: ①대표값 로직 `unified-cost-pick`/`unified-price-pick` 단일 모듈에만(정적가드로 우회 차단) ②대표값 옆 선정 사유 배지 ③충돌(동률·모순) 시 자동결정 말고 검수 큐로.
- **현재**: 부분 — pick 모듈 有, 선정사유 노출·우회방지 가드 미확인.

### 제14조 — 파급 재계산과 조용한 드리프트 금지
- **조문**: 원가·환율·경쟁가 변경 시 딸린 마진·가격신호 자동 재평가, 임계 이상 변동은 admin 능동 통지. 값이 바뀌었는데 아무도 모르는 상태 불허.
- **왜 위험**: 원가 15% 올랐는데 판매가 그대로면 마진이 조용히 증발. 알림 없으면 "콕핏 열어봐야만" 발견 — 발견은 늘 손실 누적 후.
- **합격**: ①원가/시세 변경 커밋 시 영향 SKU 마진 자동 재산출 ②마진 변동 임계(±5%p 등) 초과 시 통지 ③"마지막 검토 이후 입력 바뀐 SKU" 목록 상시.
- **현재**: 부분 — `impact.ts`·`revalidate.ts`·GcubeSyncBadge 有, 능동 통지·마진 드리프트 목록 미확인.

### 제15조 — 운영 건강 지표: 파이프라인 자체를 계기판에
- **조문**: 검수 적체·추출 정확도(골든셋 통과율)·자동확정 비율·stale 비율·이상치 건수를 admin 대시보드 한 화면에서 상시 확인.
- **왜 위험**: 데이터 파이프라인은 고장 나도 소리가 안 난다. 검수 큐 200건 적체·추출정확도 하락 시 화면의 모든 가격이 "그럴듯한 오답". **지표 없으면 신뢰 여부를 매번 감으로.**
- **합격**: ①5대 지표 단일 화면 ②골든셋 통과율 시계열 ③검수 큐 체류시간(입력→확정 리드타임).
- **현재**: 부분 — `intake-telemetry.ts`·`golden-set.ts`(측정 원천) 有, 통합 계기판 미확인.

### 제16조 — 대량 작업 1급 시민: 한 건씩만 되면 실무에서 죽는다
- **조문**: 승인·수정·만료·롤백은 단건뿐 아니라 선택 일괄(bulk) 가능, 일괄도 건별 감사로그·건별 되살리기 보장.
- **왜 위험**: 공급사 엑셀 한 장=수십~수백 행. 검수가 한 건씩만 되면 member는 대충 확인(품질붕괴) 또는 안 함(적체). **제2·4조(AI자동+확인) 실효성은 bulk 없이 성립 안 함.**
- **합격**: ①검수 화면 다중선택+일괄 승인/반려 ②일괄이 감사로그에 건별로 남고 건별 되살리기 ③잘못된 일괄확정 원샷 롤백(작업 단위 undo).
- **현재**: 부분(추정) — 감사·되살리기 백본 有, batch 단위 롤백 미확인.

## 확장 원칙 우선순위 Top 3
1. **제11조 근거 스냅샷** — 가격은 이 시스템 최종 산출물=매출 그 자체. 근거 없으면 나머지 다 잘해도 "믿을 수 없는 좋은 숫자". 협상·손익검증·사고조사 3장면 모두 필요.
2. **제12조 마진 가드레일** — 유일하게 "한 번 실수=즉시 현금 손실" 직결 축. 유사 사고 전례(v0.7.217) 있어 재발 확률 입증. 비용 대비 방어가치 최대.
3. **제10조 신선도 시효** — 시세는 자연 부패하는데 현재 견적 유효기간만 부분 커버. 썩은 데이터는 제11조 근거마저 "낡은 근거"로 만듦(11조의 전제조건).

## 대표 9원칙과의 관계
- 제11조=**제9조(로그) 확장**("무엇이 바뀌었나"→"왜 이 값이 됐나"), 제7조 보존 대상을 결정 시점 입력 조합까지 확대.
- 제12조=**제8조(교정안내)의 사업 버전**(형식 오류 아닌 "사업적으로 틀린 값" 교정).
- 제10·14조=**제5조(일관성)의 시간축 확장**("지금도 유효한 값인지"를 같게 보여줌).
- 제13조=제5조가 값 차원에서 성립하기 위한 전제(표시 일관해도 소스 선택이 화면마다 다르면 무의미).
- 제16조=**제4조(AI자동+확인) 실행 가능 조건**(확인 UX 단건뿐이면 확인이 형식 전락).
- 제15조=9원칙 전체 작동 여부를 감시하는 **메타 원칙**.

## 한 줄 총평
> 대표 9원칙 = **입출력의 헌법**(데이터를 잘 받아 잘 보여주고 잃지 않는다). 확장 7조 = **결정의 헌법**(그 가격이 지금도 옳고, 왜 옳은지 증명 가능하고, 틀리면 돈 나가기 전에 시스템이 먼저 잡는다). **9원칙이 데이터를 지키면, 확장조는 마진을 지킨다.**

## 주요 충돌·주의
- **제6조↔제5조**: 원본표시vs일관성 → "같은 값"을 (원본값,환산규칙) 쌍으로 정의. 기본=원본+배지, 비교화면=공통 환산토글+원본 병기, 환산율 SSOT 1곳.
- **제4조↔제7·9조**: 자동확정 틀려도 원본(7)+감사로그(9)로 되돌림. 제9조가 자동화 안전망, 없으면 제4조 가동 금지(Phase0 선행 강제).
- **제4조↔제8조**: 분업. auto밴드는 조용히 채움, review/block밴드는 제8조 3요소로 사람에게. AI 교정 제안도 자동반영 금지(후보 체크리스트).
- **제8조↔제9조**: 남기는 이유가 안내의 근거. 실패 로그는 구조화 필드(에러코드·행·입력값·판정사유), 자유텍스트만 금지.
- **제2+3+8조 = 막다른 길 3형제**: 하나의 공용 동선 규약으로 함께 해결.
- **제9조 주의**: "남긴다"가 "막는다"로 변질 금지 — 로깅 실패가 저장 거부하면 never-block 위반. 무제약 append+동일트랜잭션 트리거+아웃박스. purge조차 감사 남김.
- **제5조 청산 전 제4조 금지**: tier 이중소스 남은 채 자동확정=사람도 모르는 값 자동 전파.
- **제12조 주의**: 하한이 정당한 프로모션까지 막지 않도록 admin 사유 오버라이드(제9조 기록) 병행.

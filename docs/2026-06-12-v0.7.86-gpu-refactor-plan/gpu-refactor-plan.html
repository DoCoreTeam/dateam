보다 통합 표 + 마스터·디테일

왼쪽 목록(보기 프리셋 전환) + 오른쪽 상세 패널(공급원가·시장 비교·변동 이력). 비교 맥락을 잃지 않고 한 건을 깊게 본다.

넣다 통합 입력 + 자동 게이트

텍스트·이미지·URL을 한 번에. 신뢰도 90%↑ 자동 확정 후보 / 그 외만 사람이 검토 → 검토 피로 없이 정합성 유지.

믿다 용어·정합성·정직

계산 SSOT 보존(R1~R7), 약점 정직 노출, 상세 데이터 조회 갭은 "신규 API 필요"로 표기. 시스템 용어로 통일.

01

배경 · 문제 · 목표(정량)

"단순화"를 수치로 정의한다.

배경 / 문제 정의

  • 분산: 같은 GPU의 공급원가·판매가·추종가·시장가·재고가 5개 탭에 흩어져, 한 건 판단에 탭 이동 반복.
  • 입력 혼란: 공급사·견적·경쟁가 등록 경로가 6개 → "어디서 등록?"을 매번 고민, 중복·검증 반복.
  • 상세 단절: 화면은 최저 공급원가·최신 시장가 등 집계만 보여주고, 전체 견적·시세 이력은 조회 경로가 없음.

상세 데이터 — 다 보이나? (냉정)

집계 뒤 전체 견적·시세 이력·변동 이력.

⚠️ 냉정한 답: 지금은 "다" 못 본다. 화면은 최저 공급원가·최신 시장가만. 전체 견적·시세 이력·추출 이력은 조회 라우트가 대부분 없다. 상세 패널을 제대로 채우려면 읽기 전용 신규 API 4개가 필요하다(로드맵 P5).

경쟁사↔시장가 역추적단방향market(경쟁사→가격)역추적 없음(우선순위 낮음)

왜 안 보이나

  • 화면이 의사결정용 집계(최저가/최신가) 위주 설계.
  • append-only 테이블(추출 이력)은 쓰기만 있고 읽기 API 미연결.
  • 시세는 최신 1행만 비교에 사용 — 과거값은 DB엔 있으나 노출 경로 없음.

구현 영향(명확)

  • 상세 드릴다운은 프론트만으로 불가 — 읽기 API 4개 신규.
  • 모두 읽기 전용·RLS read → 위험 낮음, 쓰기 변경 0.
  • DB 변경 없음(기존 테이블 조회) → 마이그레이션 불필요.
  • API 준비 전까지 상세 패널은 "신규 API 필요" 배지로 단계 노출.

09

비기능 요구 · 예외 흐름

성능·반응형·접근성 + 엣지케이스.

비기능 요구(NFR)

  • 성능: 통합 표 초기 로딩 ≤1.5s, 보기 전환은 재요청 없이 컬럼 교체(클라). 목록 서버 페이지네이션 기본(대량 행).
  • 반응형: 모바일(<768)에서 마스터·디테일 → 목록→상세 풀스크린 전환(2-pane 금지). 가로 스크롤 금지(`table-card`).
  • 접근성: 보기 세그먼트 키보드 이동·ARIA, 상태 색에 아이콘/텍스트 병기(색맹), 터치 44px.
  • 로딩/빈/에러 3종 상태 필수(목록·상세·입력 각각).

부정 — 실패 시나리오 (D1~D5)

D1
보기 전환이 인지부하를 올릴 수 있다
컬럼이 통째 바뀌면 "지금 무슨 보기지?" 재인지 비용. → 완화: 기본=판매가 고정, 보기 라벨 상시 표시, 저장된 보기로 자기 뷰 고정.
D2
거대 단일 표가 모바일에서 붕괴
컬럼×상세 패널을 카드로 못 접음. → 완화: 모바일은 목록→상세 풀스크린 전환, 보기는 드롭다운, 핵심 컬럼만.
D3
드릴다운 길찾기 실패
인라인 드로어는 목록을 밀어 맥락 상실. → 완화: 인라인 드로어 폐기 → 우측 고정 상세 패널(목록 유지). 상세 내부는 탭 1개씩.
D4
다중행 검토 피로 → 무검토 일괄 확정
50행 전수 검토 강요는 비현실. → 완화: 신뢰도 자동 게이트(≥90 자동·70~90만 사람), 확정 전 변경분 diff 요약.
D5
권한 분리로 영업이 마스터 단절
공급사 등급·마진을 못 보면 병목. → 완화: 마스터 쓰기만 admin, member 읽기 허용(§6 RBAC).

긍정 — 강한 가치 (P1~P3)

P1
상세 데이터 갭을 숨기지 않는 정직성
"신규 API 필요"를 화면에 1급 노출 → 견적 의사결정의 신뢰 기반. 갭 메우면 배지 자동 소멸.
P2
계산 SSOT 보존, 표현만 재배치
pricing·intake-routing·dedup 등 정합성 핵심 무변경 → 회귀 위험 구조적으로 낮음.
P3
단일 입력 → 자동분류 → 원자 분배
6입력 → 1입력, 어긋날 표면 축소. 자동 게이트로 단순화가 품질저하로 안 번짐.

UX 판단 — 이 도메인에 무엇이 맞나 (정확히)

(a) 보기 7종 — 순수 세그먼트?
권장: 저장된 보기 + 컬럼 토글 하이브리드. 5~7개는 기본 프리셋, 영업이 "내 협상 보기"(판매가+시장 중앙+마진)를 조합·저장.
이유: 가격 결정은 본질이 다축 동시비교 → 세그먼트 단독은 비교 단절. 트레이드오프: 초기 설정 비용 → 기본 프리셋 3개로 상쇄.
(b) 통합 표 정말 1개? — 마스터·디테일
권장: 좌 목록 + 우 상세 패널(마스터·디테일). 영업 실작업="하나 골라 깊게 판단" = 목록 선택→상세 고정의 정석(CRM·이슈뷰 패턴).
"표 1개" 철학 유지하되 상세는 인라인 드로어가 아니라 우측 패널. 모바일은 풀스크린 전환 폴백.
(c) 통합 입력 — 다중행 vs 단건 위저드
권장: 다중행 미리보기 유지(견적은 통째로 옴) + 신뢰도 자동 게이트. 한 메일에 5모델이 흔함 → 단건 위저드는 퇴행.
검토 피로는 자동 게이트로 해소. "전부 확정" 대신 변경분 diff. 자동확정분은 감사에 'auto' 플래그.
(d) 상세 드릴다운 — 인라인 vs 모달 vs 패널
권장: 우측 디테일 패널. 인라인=목록 깨짐, 모달=비교 맥락 차단. 패널은 좌측 비교맥락 유지하며 한 건 깊게.
전체 견적/시세 이력/변동 이력은 동시 3패널 금지 → 상세 내부 탭 1개씩.
🎯 UX 종합 — "단일 표 + 세그먼트 + 인라인 드로어"는 단순함을 의도했으나 비교·드릴다운·검토와 어긋난다. → 저장된 보기 + 마스터·디테일 패널 + 신뢰도 자동 게이트가 이 도메인에 적합. (위 통합 표 목업이 이 권장안을 반영)

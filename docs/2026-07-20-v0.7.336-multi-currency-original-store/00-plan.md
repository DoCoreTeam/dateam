# 다통화 원본저장 + 표시-시점 환산 기획 (미구현)

- 작성: 2026-07-20 · 상태: **기획만 / 구현 금지**
- 발단: 소프트뱅크(엔화) 등 비원화·비달러 페이지의 실제 가격을 "보류"로 버리던 v0.7.333 정책 재설계
- 방향(사용자): **어떤 통화든 받는다 → DB엔 원본 통화 그대로 → 표시할 때 환율 적용 → 어떤 환율을 적용했는지 명시**. 기준통화=원화(KRW).

---

## 1. 현재 상태 (실측)

| 요소 | 현재 | 판정 |
|---|---|---|
| DB 원본통화 저장 | `market_prices.original_currency`·`original_price` 컬럼 존재(024/134), 공급가 프롬프트도 원본 추출 | ✅ 스키마 이미 있음 |
| FX 소스 | 한국수출입은행 `AP01`(oapi.koreaexim) — **전 통화 반환** | ✅ 소스 충분 |
| FX 저장 | `fx_rates(rate_date, usd_krw)` — **USD/KRW 1쌍만** 저장 | ❌ 다통화 미저장 |
| 변환 로직 | `normalize-money.competitorPriceToUsd`·`transcription-to-items` = **비USD/비KRW는 보류(null)** | ❌ 환산 대신 폐기 |
| 표시 | `price_usd`(사전계산값) 기준. 원본은 "보존용" | ⚠️ 원본이 권위 아님 |
| 적용환율 표기 | 없음 | ❌ |

→ **결론: 스키마·FX소스는 이미 있고, "저장은 USD로 환산해 넣고 원본은 참고"인 현재 방향을 "원본이 권위 + 표시때 환산"으로 뒤집는 게 핵심.**

---

## 2. 제안 설계 (사용자 방향 + 정제)

1. **입력**: 어떤 통화든 감지(¥/円→JPY, 元→CNY, €→EUR, $→USD, ₩/원→KRW …). 감지 실패만 검수 플래그.
2. **저장(권위)**: `original_currency` + `original_amount` + `original_unit`(기간·장수 문맥) 을 **진실로 저장**. 환산값은 파생.
3. **FX**: `fx_rates`를 다통화로 확장 — `AP01` 응답에서 USD·JPY·EUR·CNY… 을 함께 저장(한 번의 호출로 전부).
4. **표시(파생)**: SSOT(normalize-money)로 원본→KRW(기준)·USD 환산. **적용한 환율값·환율일자·출처(매매기준율)를 함께 표시.**
5. **감사**: 각 가격행에 "환산에 쓴 환율일자" 참조 저장(재현·감사 가능).

---

## 3. ⚠️ 당신이 놓치고 있는 부분 (핵심 — 여기가 진짜 설계 포인트)

### 놓친 것 #1 — 환율 "시점": 표시-시점 환산은 **가격이 환율따라 움직인다**
"DB엔 엔화, 표시 때 환산"으로만 하면, ¥2,500,000(고정)이 **오늘 ₩X, 다음주 ₩Y** 로 바뀝니다(환율 변동분). 
- "오늘 시장 비교"엔 좋음(항상 현재 환율).
- **하지만 가격 추세/이력 차트가 오염됨** — 실제 가격은 안 변했는데 환율만으로 등락. 
- **정답 설계**: 원본 + **캡처시점 환율 스냅샷** 둘 다 저장. → "지금 비교" 뷰는 현재환율로 재환산, "이력/추세" 뷰는 캡처환율 고정. **이 이원화를 안 정하면 나중에 추세가 거짓말함.**

### 놓친 것 #2 — **통화보다 단위·번들이 훨씬 큰 함정** (진짜 블로커)
소프트뱅크 실데이터: H100 플랜 **¥2,500,000/월** = **8-GPU DGX 번들**(스토리지·InfiniBand·SW 포함). A100 시간제 = **7.2円/분·1장**(이건 깨끗).
- 통화 환산은 쉬운 20%. **"per-GPU·시간"으로 만들려면 ÷8장 ÷730시간 + 번들(스토리지/네트워크) 포함분 분리**가 필요 — 이게 어려운 80%.
- 즉 **환율만 붙여도 "비교 가능한 GPU 시세"가 안 됨.** 번들가는 "번들가(스토리지 포함)" 라벨을 달고, per-GPU 환산은 별도 규칙. **통화에 집중하다 이걸 놓치면 여전히 사과-오렌지 비교.**

### 놓친 것 #3 — JPY는 **100엔 단위 고시** (매매기준율 함정)
한국수출입은행 AP01은 엔화를 **`JPY(100)` = 100엔당 원화**로 줍니다. 그대로 쓰면 **100배 오류.** 다통화 확장 시 통화별 단위계수(100엔, 100원 등) 처리 필수.

### 놓친 것 #4 — 환율 결측/휴일 폴백
매매기준율은 **주말·공휴일 미고시.** 표시-시점 환산인데 그날 JPY 환율이 없으면? → "직전 영업일 환율" 폴백 + "환율 미확인" 상태(빈값·오값 금지).

### 놓친 것 #5 — 기존 데이터 backfill
기존 `market_prices`는 `price_usd`가 권위, `original_currency` 다수 NULL. "원본 권위"로 뒤집으면 **기존 행 해석 규칙**(NULL=USD가정 유지) + 점진 마이그레이션 필요. 과거 파괴 금지.

### 놓친 것 #6 — 표시통화 2개(KRW+USD) 교차환율
기준은 KRW인데 콕핏은 USD도 보여줌. JPY 원본 → KRW(직접 매매기준율) + USD(교차: JPY/KRW ÷ USD/KRW). 교차환율 정합성.

### 놓친 것 #7 — SSOT 통합
현재 환산이 `transcription-to-items`·`competitorPriceToUsd`·`review/route` 3곳 분산. 다통화는 **반드시 `normalize-money` 한 곳**으로 모아 한 번 고치면 전부 반영.

## 4. 제안 단계 (구현은 승인 후)

- **P1 FX 다통화**: `fx_rates`에 통화별 저장(마이그: `fx_rates(rate_date, currency, per_unit, krw_rate)` 또는 컬럼추가) + `AP01` 파서가 전 통화 저장 + JPY 100단위 처리.
- **P2 저장 권위 전환**: 파이프라인이 원본(통화·금액·단위) 저장, 환산은 파생. 감지 실패=검수.
- **P3 표시 SSOT**: `normalize-money`에 `toKrw`/`toUsd(originalCurrency, amount, unit, rateDate)` + 적용환율 메타 반환. 콕핏·표가 이걸로 표시하고 "환율 1 JPY=X원(YYYY-MM-DD 매매기준율)" 병기.
- **P4 시점정책**: 캡처환율 스냅샷 저장 → 이력=고정, 현재비교=재환산.
- **P5 단위·번들**(별 트랙, 더 큼): 번들가 라벨 + per-GPU 환산 규칙. 이건 통화와 분리된 별도 기획.

## 5. 완료 기준(요지)
- [ ] 엔화 페이지 투입 → H100 ¥2,500,000/월 이 **원본 저장 + KRW/USD 환산 표시 + 적용환율 명시**로 나옴(보류 아님)
- [ ] JPY 100단위 정확(100배 오류 0)
- [ ] 이력 차트가 환율 변동으로 거짓 등락 안 함(시점정책)
- [ ] 번들가는 "번들(스토리지 포함)"로 구분 표기
- [ ] 환산 SSOT 단일화, 기존 데이터 무손상

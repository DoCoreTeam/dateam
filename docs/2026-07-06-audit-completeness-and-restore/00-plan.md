# 완벽 이력(유실 0·빈틈 0) + 항목 단위 복구 — 통합 기획

> 상태: **기획만 (절대 구현 금지)** · 2026-07-06
> 요구: ① 아주 미약한 변경 하나도 안 놓치고, 실패·에러까지 정확히 남는 완벽한 이력 (드문 유실도 불가)
>       ② 이력의 각 시점을 **본인·그 항목만** 되살리는 복구 — 다른 사용자·다른 항목은 안 건드림

---

## 0. 원칙
- **유실 0 · 빈틈 0**: 앱의 어느 경로로 바뀌든 모든 상태변경이 반드시 기록.
- **실패·에러도 기록**.
- **복구 = 본인 소유 + 그 항목 단위**. 되살리기도 다시 되돌릴 수 있음.
- **유기적 연계(연결·관계) 보존**.

## 1. 왜 지금(앱단 로깅)으로는 "완벽"이 안 되나 — 실측 근거
- **우회 경로 누락**: `daily_logs`를 바꾸는 코드가 계측한 2파일(daily/dept actions) 밖에도 있음 —
  `calendar/actions.ts`, `meeting-notes/actions.ts`, `api/work/autolink`, `api/deals/activities`,
  `api/ai/analyze-work`, `lib/weekly-report/draft-server.ts`, `lib/work/autolink-run.ts`. → 이력에 안 남음.
- **드문 유실**: 현재 `logActivity`는 best-effort(로그 실패해도 본작업 진행) → 그 활동 로그가 드물게 소실.
- **before 부분 캡처**: 액션별로 전체행이 아닌 일부만 저장(복구 재료 불완전).
- **daily_logs는 완전삭제(hard delete)** — `deleted_at` 컬럼 자체가 없음(라이브 확인). 삭제 복구 시 같은 id·연결 부활 불가.

## 2. 아키텍처 — 하이브리드(트리거=성공, 앱=실패)
### 2-A. 성공 변경 = DB 트리거 (유실 0·빈틈 0·자동 스냅샷)
- 각 도메인 테이블에 `AFTER INSERT/UPDATE/DELETE` 트리거 → 공용 함수 `fn_audit()`가 통합 `audit_log`에 기록.
- `fn_audit()`: `TG_OP`·`TG_TABLE_NAME`·`row_to_json(OLD)`/`row_to_json(NEW)`·`actor=coalesce(auth.uid(), NEW.user_id, OLD.user_id)`.
- **같은 트랜잭션** → 커밋되면 로그가 반드시 존재(유실 0). **앱 코드 어디서 바꾸든 무조건 걸림**(빈틈 0 — 우회 경로 문제 원천 해결).
- **before/after 전체 행이 자동 저장** → 복구 스냅샷을 공짜로·완전하게 확보.
- 대상 테이블(초안): `daily_logs`(일일+부서), `weekly_reports`, `projects`, `work_entity_links`, `project_members`, `daily_log_threads`, `calendar_events`.
### 2-B. 실패/에러 = 앱단 로깅
- **커밋 안 된 쓰기는 트리거가 못 봄**(DB에 도달 못 함). → 실패·에러는 각 액션 catch에서 `audit_log(status=failure, error_json)`로 기록.
- 이건 본질적으로 앱단(트리거로 대체 불가). 실패 로그라 무결성 무게는 성공보다 낮지만 반드시 남김.
### 2-C. ✅ 확정 원칙 — 차단 금지 + 유실 0 을 **동시에** (fallback)
> 사용자 지시: "사용자 저장이 막히는 건 시스템을 못 쓰게 하는 것. 유실은 있으면 안 되지만 **fallback을 해서라도 반드시 저장**되게 하라." → 둘 중 택일이 아니라 **둘 다** 달성한다.
- **성공 감사(트리거)**: `audit_log`를 **제약 없는 append-only 테이블**(FK·UNIQUE·CHECK 최소)로 두고, 원 write와 **같은 트랜잭션**에 기록. → 커밋되면 로그도 반드시 존재(유실 0). 순수 JSON append INSERT는 **현실적으로 실패할 일이 없어** 사용자 write를 막지 않음(테이블이 거부할 제약 자체가 없음). 만약 그 INSERT마저 실패한다면 DB 자체 장애라 원 write도 어차피 실패 — "감사가 업무를 막는" 시나리오가 구조적으로 소거됨.
- **실패 감사(앱단)** 및 만일의 앱단 로그 실패: **아웃박스(outbox) 재시도** — 즉시 기록 실패 시 로컬 durable 큐/아웃박스 테이블에 적재 후 백그라운드 재시도. 사용자 흐름은 절대 안 막고(비차단), 로그는 재시도로 결국 반드시 남음(유실 0).
- 결론: **트리거(동일 트랜잭션·무제약 append) + 아웃박스 재시도** = 차단 0 · 유실 0. 저장을 막는 설계는 채택하지 않는다.

## 3. actor(누가) 식별
- authenticated 경로: `auth.uid()` 정상.
- service_role/AI/크론: `auth.uid()`가 null → `coalesce`로 행의 `user_id`, 그래도 없으면 `'system'`.
- **대리작업**(부서장이 남의 업무 변경): `actor=auth.uid()`, `owner=행.user_id` **둘 다 저장** → 소유자가 자기 이력에서 봄, 누가 했는지도 남음.

## 4. 복구 설계 (본인·항목 단위)
### 4-A. 재료
- 각 `audit_log` 행에 `before_json`(OLD 전체) 존재 → 복구 재료 완비(트리거가 자동 보장).
### 4-B. 동작
- **수정 되돌리기**: 그 항목 id의 현재행을 `before_json`으로 UPDATE. `WHERE id=? AND user_id=auth.uid()` → **본인·그 항목만**.
- **삭제 되돌리기**: 소프트삭제면 `deleted_at=null`(같은 id·연결 부활). 하드삭제였으면 재삽입=새 id(연결 소실) → **soft 전환이 선결**(§5).
- **복구도 트리거가 자동 기록** → 복구의 복구 가능(무한 되돌리기 안전).
### 4-C. 안전장치
- **대상 시점 명시**: 특정 `audit_id`의 before로 복구.
- **동시성**: 현재행이 그 audit 시점 이후 변경됐으면(“그 사이 누가 바꿈”) 경고 후 확인 — 낙관적 잠금(updated_at 비교).
- **권한**: RLS로 본인만. 쿼리 자체가 `id=? AND user_id=auth.uid()` → 남의 것·다른 항목 물리적으로 못 건드림.
- **복구 불가 케이스 명시(UX)**: 트리거 도입 이전 이력, 과거 하드삭제 건은 before가 없어 복구 불가 — 버튼 비활성+사유 표기.

## 5. 소프트삭제 전환 (복구의 전제)
- **daily_logs**: `deleted_at` 없음 → 컬럼 ADD + 모든 삭제경로를 soft로 + **모든 조회에 `deleted_at IS NULL` 필터 추가**(대량·회귀 주의 — 집계·목록·검색 전수 점검).
- **weekly_reports**: `deleted_at` 있으나 RLS(002) 때문에 hard 사용 중(주석 근거) → soft 쓰려면 RLS/RETURNING 재설계.
- **projects**: 이미 soft (복구 준비 완료).
- **cascade 관계**(calendar_events 등 삭제 연동): soft 연동 규칙 필요(원본 soft 시 자식도 soft/복구 동반).

## 6. 기존 것 수렴 (재사용·SSOT)
- 방금 만든 `activity_log`(앱단 daily/dept) + `project_activity` + `weekly_report_activity` + 신규 트리거 `audit_log` → **중복**.
- **트리거 `audit_log`를 성공 이력 SSOT로 수렴**, 앱단 로깅은 **실패 전용**으로 축소. 이력 탭 UNION 단순화(또는 단일 소스화).
- **주간 WIP 스냅샷(마이그144, 미적용·미커밋)** → 트리거 방식과 중복 → 통합 설계로 흡수하거나 폐기 결정.

## 7. 리스크
- **쓰기 성능**(트리거 오버헤드) + **저장 폭증**(매 변경마다 전체행 JSON — 주간보고 리치HTML은 큼) → **보존정책 필수**(버전수/기간 제한, 오래된 것 압축/파기).
- **소프트삭제 전환의 광범위 회귀**(모든 쿼리 필터 누락 시 삭제분 노출).
- **트랜잭션 결합 차단 리스크**(§2-C).
- **마이그레이션 대량**(트리거 N개 + soft 전환 + 백필).
- RLS: `audit_log` 본인만 SELECT, INSERT는 트리거(SECURITY DEFINER)로만.

## 8. 단계적 실행 (구현하게 될 때)
- **P0**: `audit_log` + `fn_audit` 트리거를 `daily_logs`·`projects`·`weekly_reports`에 → 성공 변경 100% 캡처(유실0·빈틈0) + 자동 스냅샷.
- **P1**: 앱단 실패/에러 로깅 통합, 기존 3개 로그 수렴, 이력 탭 소스 단순화.
- **P2**: `daily_logs` 소프트삭제 전환(+조회 필터 전수).
- **P3**: 이력 탭 각 항목 **되살리기(본인·항목 단위)** + 동시성/권한 가드.
- **P4**: 보존정책·성능·복구 동시성.

## 9. 결정지점 (구현 착수 전 승인 필요)
1. ~~유실0 vs 비차단~~ → **확정: 둘 다(트리거+아웃박스 fallback), 저장 차단 없음** (§2-C)
2. **소프트삭제 전면 전환 승인** (복구의 전제, 광범위 변경)
3. **주간 WIP(마이그144)** → **확정: 살려서 통합**(§11 핸드오프로 커밋됨). 트리거 도입 시 주간 스냅샷을 audit_log 방식으로 수렴할지만 추후 결정.
4. **보존정책**: 이력 무한 보관 vs 기간/버전수 제한
5. **복구 범위**: 항목단위만 vs 항목+전체(주차/시점) 둘 다

## 10. 기존 주간 유실0(마이그144) 방식과의 관계
- (B)가 이미 구현한 주간 유실0: **단일 Writer**(draft 확정본 sync 제거) + **삭제 직전 스냅샷 fail-safe**(`snapshot_weekly_report`) + `restoreWeeklyReportSnapshot`(주차 단위 복원) + `WeeklyEditHistory` UI.
- 본 기획(트리거+아웃박스)은 이를 **대체가 아니라 상위 일반화**: 주간에 국한된 스냅샷/복원을 전 모듈 공통 트리거 방식으로 확장. 트리거 도입 시 주간도 audit_log로 수렴 가능(중복 제거) — 단 (B)는 이미 검증·적용됐으므로 **그대로 존치**하고, 트리거 통합은 P1에서 흡수.
- 차이(주의): (B) 주간 복원은 **주차 단위**(그 주 전체 교체) — 사용자가 원한 "항목 단위" 복구와 결이 다름. 전 모듈 트리거 복구(§4)는 **항목 단위**가 기본. 주간도 항목 단위로 맞추려면 트리거 통합 시 조정.

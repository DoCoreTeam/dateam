# GPU 관리 — 1장당 단가 전파 + 재고 수량 + 카테고리 통일 + 후속 (기획 전용)

> 상태: **기획 전용 — 구현 금지** · 2026-06-03 · 근거: 코드 + psql 라이브

## 1. B200 "1장만 금액 들어감" — 원인

### 데이터 (psql)
B200 상품 4개(×1/×2/×4/×8 모두 `pricing_mode=quote`), 견적 5건:
| 구성 | 공급사 | 단가(USD) | 1장당 |
|------|--------|-----------|-------|
| ×1 | **High Reso** | **3.24** | **3.24** ← 사용자가 넣은 1장 견적 |
| ×1 | gcube | 7.0084 | 7.01 |
| ×2 | gcube | 13.8299 | 6.91 |
| ×4 | gcube | 27.1926 | 6.80 |
| ×8 | gcube | 53.591 | 6.70 |

### 원인 (PriceTableTab.tsx 286–328 도출 로직)
- 모델 그룹 내 **1장당 단가(perGpu)는 계산되지만**, `STD_CONFIGS[1,4,8]` 중 **"존재하지 않는 구성"만 합성**(`if (existingCounts.has(n)) continue`).
- B200은 ×1/×2/×4/×8이 **모두 실제 상품으로 존재** → 합성 0건. 각 구성은 **자기 견적의 최저가**만 표시.
- 결과: ×1 = min(3.24, 7.01)=**3.24**(High Reso) / ×2~×8 = gcube 자기 견적(13.83/27.19/53.59).
- → 사용자 눈엔 "1장만 싸게 들어가고 나머진 그대로".

### 사용자가 합의했던 규칙
- **1장 가격 입력 → 모든 구성 = 1장가 × 장수**
- **여러장 가격 입력 → 1장당(=가격÷장수) 환산 후 구성별 적용**
→ 즉 **모델 그룹의 "최저 1장당 단가"를 모든 구성에 전파**해야 함. (지금은 누락 구성만 합성, 실재 구성 미적용 = 버그)

### 설계
1. `lib/gpu/pricing.ts` 신설(공용):
   - `bestPerGpu(variants)` = 그룹 내 모든 견적의 `min(unit_price / gpu_count)` (`toPerGpuPrice` 재사용).
   - `configPrice(perGpu, count)` = perGpu × count.
   - `bestPerGpuSource(variants)` = 최저 1장당 단가를 제공한 견적의 **공급사** (구성별 공급사 표기용).
2. 모델 그룹의 각 구성 표시가 = `min(자기 견적, bestPerGpu × 그 구성 장수)`.
   - 최저 1장당 단가가 곧 min이므로 실질 `bestPerGpu × count`. 공급사도 그 출처로 표기.
   - "추정" 배지: 자기 직접 견적이 아니라 1장당 전파로 산출된 구성은 기존 `추정` 배지 유지(근거 투명).
3. **서버에서 산출 권장**: `/api/pricing/gpu/products` 응답에 `per_gpu_usd`, `effective_unit_price_usd`, `effective_supplier`를 모델 그룹 단위로 계산해 내려주면 → 가격표·시장비교·고객판매가가 **동일 가격**을 보게 됨(불일치 방지). 클라이언트 중복 계산 제거.

### 엣지/리스크
- 1장 공급사가 N장 공급 불가할 수 있음 → "1장가×N(추정)" 배지로 가정 명시. (실제 가용은 재고수량 탭에서 확인)
- bulk 할인(gcube)이 1장당 더 싸면 그게 best → 자동 반영(올바름).

## 2. 재고수량 — 공급사·수량 미표시 원인

### 데이터 (psql)
- `availability_responses` **0행** → `v_fresh_availability`/`v_product_availability_summary` 비어 있음.
- 그래서 InventoryTab은 견적 보유분에 대해 "공급가능 / 수량 확인 필요" 또는 "가용량 정보 없음"만 표시.
- 가용량 등록 POST(`/api/pricing/gpu/availability`)는 있으나 **UI 진입 동선 없음** → 아무도 입력 못 함.

### 사용자 기대
"공급사랑 수량이 나와야 한다" — 확정 견적의 공급사 + 그 공급사의 가용 수량.

### 설계
1. **공급사 표시 즉시 복구(데이터 없이도)**: InventoryTab이 가용량 0건이어도 **확정 견적의 공급사 목록**을 카드에 노출(공급사명 + 단가). 수량은 "미입력" 배지.
2. **수량 인라인 입력**: 각 공급사 행에 "수량 입력" → `availability_responses` POST(resp_qty, status). 입력 즉시 카드에 수량/헬스바 반영.
3. (선택) **견적 등록 시 수량 동시 입력**: QuoteRegisterTab/통합입력에 `available_qty` 옵션 필드 → 견적 확정 시 availability_responses 자동 생성. 한 번 입력으로 가격+수량 동기화.
4. Tier 3는 기존 `direct_pool_stock`(풀 재고) 유지.

## 3. 카테고리 통일 — 공용 컴포넌트화 (핵심 유지보수 요구)

### 현황 (제각각)
| 탭 | 컴포넌트 | 그룹핑 | 접힘 |
|----|----------|--------|------|
| 가격표 | PriceTableTab | **모델별** | 기본 접힘(신규) |
| 시장 비교 | MarketTab | **Tier별** | 기본 접힘(신규) |
| 재고수량 | InventoryTab | **없음(평면)** | — |
| 고객 판매가격표 | catalog/page | **없음(평면, Tier 정렬만)** | — |

### 설계 — 단일 카테고리 스킴 + 공용 컴포넌트
1. **통일 스킴: 2단계 `Tier → 모델`** (모든 탭 공통). 최상위 Tier 그룹, 그 안에 모델 그룹. 둘 다 접기 가능, **기본 접힘**.
   - 가격표는 모델 그룹 안에 구성(×1/×2…) 행 — 기존 구조와 호환.
   - 시장/재고/고객가는 모델 그룹 안에 상품 행.
2. **공용 컴포넌트 신설**:
   - `components/gpu/CategoryGroup.tsx` — 접기 헤더(뱃지/이름/카운트/쉐브론) + 바디 slot. props: `level`(tier|model), `label`, `badge`, `count`, `collapsed`, `onToggle`, `children`.
   - `hooks/useCollapsibleGroups.ts` — `Set<key>` 상태 + `toggle` + `defaultCollapsed` 일괄 초기화(가격표/시장비교의 중복 로직 통합).
   - `lib/gpu/group.ts` — `buildTierModelGroups(products)` 공용 그룹 빌더(정렬·카운트 일관).
3. 4개 탭이 위 3개를 **공유** → 표기·접힘·정렬 동작 100% 동일, 유지보수 1곳.

### 트레이드오프
- 4개 탭 동시 리팩터링 = 회귀 리스크(중간). → **점진 적용**: 공용 컴포넌트 먼저 만들고 탭 1개씩 교체 + 각 탭 Playwright 검증.
- 가격표만 3단계(Tier→모델→구성)라 컴포넌트는 모델 그룹 children을 자유 slot으로 둬 수용.

## 4. 후속 — 공급사 미지정 견적 편집 경로

- `공급사 미지정` 견적 행(가격표 펼침)에 **"공급사 지정" 인라인 셀렉트** → `PATCH /api/pricing/gpu/quotes/[id]`(supplier_id 갱신, 없으면 신규 공급사 생성 — 기존 POST 로직 재사용).
- 지정 즉시 가격표/재고 반영. 기존 10건의 NULL 공급사를 화면에서 정리 가능.

## 5. 권장 구현 순서 (승인 후)
1. **B200 1장당 전파**(pricing.ts + products API) — 가장 체감 큰 버그.
2. **재고 공급사·수량 표시 + 인라인 입력**.
3. **공용 카테고리 컴포넌트** 후 4개 탭 점진 교체.
4. **공급사 지정 인라인 편집**(후속).
- 각 단계 Playwright(UI)+psql(DB) 직접 검증, 단계별 커밋(push 금지).

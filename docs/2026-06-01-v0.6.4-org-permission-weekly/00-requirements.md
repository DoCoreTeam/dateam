# 00 · 요구사항 — 조직도 기반 권한관리 & 주간보고 계층 취합

> 상태: **기획 전용 — 구현 금지** · v0.6.4 · 2026-06-01
> 산출 경로: DOMANGCHA CEO 파이프라인 (DC-ANA ×3 현황 + DC-RES 클로저/RLS + DC-BIZ 롤아웃)

## 1. 한 줄 비전
**조직도(`org_nodes`)를 권한의 단일 진실 소스(Single Source of Truth)로 승격**하고, 그 위에서 주간보고(→이후 캘린더)가 계층적으로 작성·취합·조회되도록 한다.

## 2. 사용자 요구 (원문 정제)
1. 권한은 role 플래그가 아니라 **조직도 위치**에서 파생된다.
2. **부서장**은 어드민이 아닌 **사용자 화면**에서 자기 부서 주간보고를 **취합**할 수 있다. (어드민엔 이 기능 불필요)
3. 취합 결과는 **"팀 전체 취합 주간보고"**로 표시된다.
4. **부서 격리**: 다른 부서의 주간보고는 보이지 않는다.
5. **계층 조회**: 상위 부서장은 모든 하위 부서 + 자기 부서 주간보고를 **조회**할 수 있다.
6. **편집·취합은 자기 부서만**. 하위 부서는 **조회 전용(read-only)**.
7. **대표이사**는 전사를 부서단위/일괄로 조망한다.
8. 캘린더도 동일 권한 패턴으로 움직인다.

## 3. 역할 정의 (권한 주체)
| 역할 | 정의 (조직도 파생) | 주간보고 권한 |
|------|-------------------|--------------|
| 팀원 | `org_nodes(person).user_id` = 본인, 부서장 아님 | 자기 보고 작성·수정·삭제 / 자기 부서 취합본 조회 |
| 부서장 | 어떤 부서 노드의 `head_user_id` = 본인 | + 자기 부서 취합(생성/편집) |
| 상위 부서장(본부장 등) | 상위 부서 노드의 head, 하위 부서를 closure로 관할 | + 하위 부서 취합본·원본 **조회 전용** |
| 대표이사 | 최상위(company/role='대표이사') head 또는 전사 권한 | 전사 전 부서 조망 (= 다부서 대시보드의 최상위 특수 케이스) |
| 어드민 | `profiles.role='admin'` | 시스템 관리 (주간보고 취합 기능은 **사용자화면으로 이관**, 어드민엔 불필요) |

> **핵심 구분**: `profiles.role`(admin/member)은 *시스템 접근*용으로 유지. *조직 권한*(부서장/계층)은 전적으로 `org_nodes`에서 파생 — 두 축을 혼동하지 않는다.

> **★ 1:N 일반화 (수정 반영)**: "상위 부서장"과 "대표이사"는 **별개 등급이 아니라 같은 패턴의 깊이 차이**다. 조직도는 1:1:1이 아니라 **1:N**(한 노드가 자식 부서 N개 보유). 따라서 **자식 부서를 2개 이상 가진 모든 노드의 장(長)** — 본부장·실장·대표이사 누구든 — 은 자기 관할 하위 부서들을 **부서단위 멀티부서 대시보드 + 드릴다운**으로 동일하게 조망한다. 대표이사는 "최상위 노드가 모든 본부를 자식으로 갖는" 특수 케이스일 뿐. 화면 분기는 역할 하드코딩이 아니라 **선택 노드의 자식 부서 수**로 결정(≥2 → 카드그리드 / 1 → 단일취합본 / 0 → 잎). UI 상세는 `02-ui-ux.md` 참조.

## 4. 확정된 아키텍처 결정 (Q&A)
| # | 결정 | 선택 |
|---|------|------|
| D1 | 계층 권한 강제 위치 | **Closure Table(`org_node_closure`) + RLS** |
| D2 | 소속/부서장 정규화 | **`org_nodes` 단일 소스** + 뷰(`v_user_departments`, `v_user_managed_nodes`) — `profiles.department_id` 추가 안 함 |
| D3 | 취합 모델 | **스냅샷 영속(`dept_weekly_reports`) + 부서장 편집**, 상위는 하위 취합본 조회 |
| D4 | 다부서 노드 뷰 + 캘린더 | **부서카드 대시보드 + 드릴다운** (대표이사 한정 아님 — 자식부서 ≥2인 모든 노드 공통, 재귀 적용), 캘린더는 **동일 가시성 매트릭스**로 동등 포함 |

## 5. 현황 제약 (DC-ANA 확인 — 설계가 반드시 반영해야 할 사실)
- `org_nodes`: `parent_id` 인접리스트, **재귀 CTE·path·depth 없음**, 트리는 클라이언트 `buildTree()`로 조립. → **DB에 상위/하위 판정 인프라 전무** (= 클로저 테이블 도입의 근거)
- `profiles`에 `department_id` **없음**. 소속 = `org_nodes(type='person').user_id` + `parent_id`.
- 부서장 = `head_user_id` **또는** role노드 첫 person child(이중 fallback). → **단일화 필요**.
- `weekly_reports`(user_id, week_start[월요일], category, performance, plan, issues): **department_id 없음**, 취합 영속 테이블 없음.
- **현재 RLS = `auth.uid() IS NOT NULL`** (002): 로그인 시 전원 열람. → 부서 격리는 **브레이킹 체인지**.
- 과거 `profiles` RLS 자기참조로 **무한 재귀 발생 → 003에서 단순화**한 이력. → RLS에서 테이블 참조 시 재귀/성능 함정 **반드시 회피**(SECURITY DEFINER plpgsql 함수로 캡슐화).

## 6. 비기능 요구
- **보안 하한선은 DB(RLS)가 보장** (API 우회·직접 쿼리에도 안전). 복잡한 계층 로직은 SECURITY DEFINER 함수에 캡슐화.
- **성능**: 부서 수십·직원 수십~수백 규모. closure JOIN + 인덱스 + `=ANY(ARRAY(select fn()))` 패턴으로 RLS 행단위 함수호출 회피.
- **감사 로그**: "누가 누구 보고서를 언제 열람" 기록(권한 시스템 도입의 정당성·디버깅).
- **반응형**: 모든 화면 모바일 카드 레이아웃, 가로 스크롤 금지(CLAUDE.md 정책).
- **점진 롤아웃**: 피처플래그 `weekly_report_hierarchy_enabled` + Shadow→Soft→Hard 단계 + 30분 내 롤백.

**제외**: 실제 코드·마이그레이션 적용, 캘린더 컴포넌트 상세(권한 매트릭스·원칙만), HR 시스템 연동 구현.

## 8. 성공 정의
- 부서장이 사용자화면에서 자기 부서 주간보고를 취합 → "팀 전체 취합" 표시.
- 타 부서 격리 / 상위→하위 조회 전용 / 자기부서만 편집·취합 — DB 레벨에서 강제됨.
- 대표이사가 전사 부서 현황을 한 화면에서 조망 + 드릴다운.
- 동일 권한 패턴이 캘린더에 재적용 가능한 일반화된 모델.
- 브레이킹 체인지 없이 단계적으로 안전 롤아웃.

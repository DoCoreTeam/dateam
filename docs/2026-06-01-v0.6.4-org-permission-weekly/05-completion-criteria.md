## C. 리스크 등록부
| 리스크 | 심각도 | 완화책 |
|--------|:--:|--------|
| 전원열람→격리 브레이킹 체인지 | 높음 | 피처플래그 + Shadow→Soft→Hard + 30분 롤백 + 사전공지 |
| HR/조직도 마스터 데이터 부정확 | 높음 | Shadow 기간에 부서장지정·소속 정합성 전수점검, 미지정 경고 |
| RLS 무한재귀/성능(003 재발) | 높음 | plpgsql SECURITY DEFINER 캡슐화 + `=ANY(ARRAY(select fn()))` + 인덱스 |
| 부서장 공석 → 취합 마비 | 중 | 상위 자동대행 fallback, 공석 거부 안 함 |
| 조직개편 시 과거 보고서 소급 가시성 | 중 | 작성시점 department 동결 + 경로 보존 |
| 과설계(범용 권한+캘린더 선반영) | 중 | P4는 P2 정착 후 의사결정, 80% 시스템화·20% 운영 |
| 부서장 취합 UI 미사용 | 중 | P3 어드민 취합 병행, 적응 후 제거 |

## E. 핵심 메시지 (요약)
- 이 일의 본질은 **새 권한 시스템 발명이 아니라, 이미 있는 조직도(`org_nodes`)를 권한 소스로 승격**하는 것.
- 최대 난제(상위/하위 판정 인프라 부재)는 **클로저 테이블 + SECURITY DEFINER 함수**로 해결하며, 과거 RLS 재귀 사고를 구조적으로 회피.
- 가장 큰 위험은 기술이 아니라 **"전원열람→격리" 전환 충격** → 단계 롤아웃이 성패를 가른다.
- 욕심을 P1+P2로 묶고, 캘린더·범용화는 검증 후로 미루는 것이 최상의 결과.

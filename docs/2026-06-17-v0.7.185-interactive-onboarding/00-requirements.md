# 00 · 요구사항 — 실습형 인터랙티브 온보딩

> v0.7.185 기획 · 2026-06-17 · **기획 문서(코드 구현 없음)**
> 작성: DOMANGCHA CEO 파이프라인 (🟦 DC-BIZ/RES/OSS/ANA 종합)

## 1. 한 줄 정의

신규 member가 **최초 로그인 시 자동으로**, "설명만 보는 투어"가 아니라 **직접 손으로** 업무를 등록하고 GPU 금액을 확인하며 익히는 **스포트라이트 기반 실습형 온보딩**.

## 2. 왜 (문제 정의)

- newAX는 일일·부서·주간 업무관리 + GPU 가격 콕핏 + AI까지 표면이 넓어 **신규 구성원이 핵심 기능을 자력 발견하기 어렵다.**
- 사내 의무 도구의 진짜 적은 "이탈"이 아니라 **"형식적 1회 입력 후 방치"** → 온보딩 목표는 가입 활성화가 아니라 **첫 주 습관 형성**.
- 기존엔 주간보고에만 `SpotlightOnboarding`(설명형 5스텝, localStorage)이 있고, "직접 해보게" 하는 실습형은 없음.

## 3. 확정된 사용자 결정 (Q&A 결과)

| 항목 | 결정 | 비고 |
|---|---|---|
| 실습 방식 | **진짜 등록 (실데이터)** | ⚠️ 단, 오염 방지 위해 `is_onboarding` 격리 등록으로 전개 (§6 충돌) |
| 트리거 | **최초 로그인 1회 자동** | 완료/스킵 시 재노출 안 함 |
| 실습 범위 | 업무등록(필수) · GPU확인(필수) · AI체험 · 프로필/조직 | 우선순위는 §5 |
| 역할 분기 | **member 중심 단일 플로우** | admin/api_user는 후순위/제외 |
| 스포트라이트 | **driver.js** (경량 OSS) | MIT · 6KB · 의존성0 |
| 진행 저장 | **DB(profiles 컬럼/신규 테이블, RLS)** | localStorage는 깜빡임 방지 캐시만 |
| 이탈 | **언제든 스킵 + 나중에 재진입** | 강제 종료 아님 |

## 4. 기능 요구사항 (FR)

- **FR-1 자동 트리거**: 미완료 member가 `(member)` 진입 시 온보딩 자동 시작. `must_change_password`/이름설정 모달이 떠야 하는 경우 **그것들이 우선**, 온보딩은 그 뒤.
- **FR-2 스포트라이트**: 특정 UI를 강조(딤드 오버레이 + cutout)하고 **강조된 실제 요소를 사용자가 클릭/입력 가능**해야 함(보여주기 only 금지).
- **FR-3 실습 게이팅**: "다음" 버튼이 아니라 **실제 행동 완료(폼 제출 성공 등)**가 다음 스텝 트리거. 행동 전에는 진행 불가(또는 명시적 스킵만).
- **FR-4 멀티페이지**: `/home → /daily → /pricing → /org` 등 **여러 라우트를 가로지르며** 진행, 페이지 전환 후 정확한 스텝에서 재개.
- **FR-5 진행 영속화**: 어디까지 했는지 DB 저장. 기기/브라우저 바뀌어도 이어서.
- **FR-6 스킵·재진입**: 언제든 스킵, 사이드바/도움말에서 "온보딩 다시 하기"로 재시작.
- **FR-7 완료 처리**: 마지막 스텝 후 완료 표시 + 축하/요약 화면.
- **FR-8 SSOT 통합**: 기존 `SpotlightOnboarding`(주간보고)을 **동일 엔진으로 흡수** — 스포트라이트 구현이 2벌 공존 금지.

## 5. 범위 우선순위 (🟦 DC-BIZ 권고)

| 단계 | 실습 | 근거 |
|---|---|---|
| **P0 (필수)** | ① 일일 업무 등록 ② 프로필/조직 둘러보기 | 데일리 핵심 루프 + 권한 컨텍스트 이해(전제) |
| **P1** | ③ AI 기능 1회 체험 | 기존 5-3 패턴 재사용(저비용)이나 첫날 인지부하 → "나중에" 진입점 권장 |
| **P2 (조건부)** | ④ GPU 가격 확인 | member 다수에 관련성 낮음 → 역할/부서 조건부 노출, 전원 강제 금지 |

> 첫날 인지부하 관리: **P0만 강제, P1/P2는 "더 둘러보기" 선택형**으로 두는 것을 강력 권고.

## 6. ⚠️ 핵심 충돌 — 실데이터 오염 (반드시 결정 필요)

사용자는 "진짜 등록(실데이터)"을 선택했으나, 이는 기존 정책과 **정면 충돌**:
- `[[feedback_test_isolation]]` (실데이터 오염 금지)
- `[[feedback_existing_data_protection]]` (상태 플래그/실데이터 보호)
- org-scope 권한상 온보딩 더미가 **상급자 주간보고 롤업·AI 후보추출**까지 흘러감 → 사고.

**CEO 채택안 (양립)**: "진짜 등록"은 유지하되, 온보딩 중 생성 행에 **`is_onboarding = true` 플래그**를 부여하고 **롤업·AI·주간보고 집계에서 제외**. → 사용자는 "진짜로 해봤다"는 체감을 얻고, 운영 데이터는 오염되지 않음. (대안: 비저장 샌드박스 — 체감 약해 비권장.)
→ **이 설계는 04-completion-criteria.md에 박제, 구현 착수 전 사용자 최종 확인 필요.**

## 7. 비기능 요구사항 (NFR)

- **반응형**: 모바일 카드 레이아웃에서 스포트라이트 타겟 좌표 깨짐 점검(브레이크포인트 320/768/1024/1440). 모바일에서 driver cutout 동작 실측 필수.
- **a11y**: 키보드 진행/ESC, `prefers-reduced-motion` 시 애니메이션 off(driver.js 자동대응 없음 → 직접 처리), 색대비 4.5:1, 포커스 복원.
- **성능**: driver.js는 effect 내 동적 import로 초기 청크 분리, app page JS 예산 <300KB(gz) 유지.
- **디자인 토큰**: popover 스타일은 globals.css 토큰 경유(인라인 style 금지), `popoverClass` SSOT.
- **i18n**: 온보딩 카피는 하드코딩 금지 권고(키: `onboarding.*`). 현 프로젝트 i18n 적용 범위 확인 후 정렬.

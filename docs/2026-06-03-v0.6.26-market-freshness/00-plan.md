# 시장 비교 — 만료 제거 + 새로고침 수집 정상화 기획

> 상태: **기획 전용 — 구현 금지** · 2026-06-03
> 근거: 🟦 DC-ANA 코드 분석 + 라이브 데이터 확인

## 1. 문제 (확정)
- **A. 48h 만료로 데이터가 사라짐**: `market_prices` 105행 존재하나 최신 수집이 2026-05-31(약 3일 전). `route.ts:4 MARKET_FRESH_HOURS=48` + `route.ts:139 isFresh` 필터가 48h 초과분을 전부 버려 → `market_min/max/median=null` → 전 행 "데이터 부족", 진영 0/136.
- **B. 새로고침이 데이터를 못 가져옴**: 경쟁사 URL은 일부 등록(pricing_url 8/12, competitor_url 23/50)됐으나, 새로고침(`refresh/route.ts`)이 외부 페이지를 직접 fetch(`fetchUrlText`)→Gemini 분류하는데 **차단/JS렌더/추출 실패 시 조용히 0건 insert** → `recorded_at` 미갱신 → 만료 지속.

## 2. 사용자 요구
1. **만료 개념 제거** — 오래됐어도 마지막 수집 가격을 **계속 보여줄 것**.
2. **새로고침이 실제로 데이터를 가져오게** — 지금은 눌러도 빈손.
3. "이런 설계(48h 만료)는 부적절" → 신선도는 숨김 기준이 아니라 **표시 정보**로만.

## 3. 설계 방향

### A. 만료 필터 제거 → "최신값 항상 표시 + 신선도는 배지"
- **변경 원칙**: `market_min/max/median`을 **경쟁사별 최신 가격(latest per mapping, recorded_at DESC 1건)**으로 항상 계산. 나이(age)로 버리지 않는다.
- `is_fresh`는 **필터가 아니라 표시 플래그**로 강등: 행/배지에 "n일 전 수집" 또는 "최신/오래됨" 표기.
- 코드 영향(설계): `route.ts` — `freshPrices = enriched.filter(c.is_fresh && price!=null)` → `latestPrices = enriched.filter(price!=null)` (나이 무관, 경쟁사별 최신 1건). `MARKET_FRESH_HOURS`는 표시용 임계로만 유지.
- 진영 카운트(`route.ts:218`)도 `market_median!=null`면 정상 집계되므로 자동 복구.
- 결과: 3일 전 데이터라도 화면에 가격·포지션 표시 + "3일 전 수집" 안내.

### B. "데이터 부족" 라벨을 사유별로 분기 (UX)
| 상황 | 현재 | 변경안 |
|------|------|--------|
| 수집 가격 자체가 0건 | "데이터 부족" | "수집 가격 없음 — 가격 등록 필요" |
| 가격은 있으나 오래됨 | "데이터 부족"(만료로 숨김) | **값 표시** + "n일 전 수집" 배지 |
| 공급가 없음(전략탭) | "데이터 부족 — 공급가/시장가 필요" | 유지(별개) |

### C. 새로고침 수집 정상화
근본: 외부 스크레이핑은 본질적으로 불안정(anti-bot·JS 렌더·구조 변경). 현재는 실패가 **조용히 0건**이라 사용자가 원인을 모름.
- **C-1. 실패 가시화(필수)**: 새로고침 응답의 per-URL 결과(`results[]`: 수집 N건/실패 사유)를 **UI에 노출**. "12 URL 중 3건 성공·9건 차단" 식으로. 지금은 토스트만 → 상세 표시.
- **C-2. fetch 견고화**: `fetchUrlText`에 UA/Accept 헤더·리다이렉트·타임아웃 보강. (JS 렌더 사이트는 한계 — 명시)
- **C-3. 수동 입력을 1차 경로로 격상**: "+가격 등록"을 주요 동선으로(스크레이핑은 best-effort 보조). 등록 시 즉시 최신값으로 반영(만료 제거됐으니 계속 유지됨).
- **C-4. last_refresh_attempt + error 기록(선택)**: 마지막 시도 시각·결과를 저장해 "왜 안 들어왔는지" 추적.
- ⚠️ 정직한 한계: 경쟁사 사이트가 봇 차단/JS 렌더면 자동 스크레이핑은 실패할 수밖에 없음 → **수동/주기 입력이 신뢰 소스**, AI 스크레이핑은 보조라는 전제로 설계.

## 5. 엣지/리스크
- 만료 제거 시 **아주 오래된 가격이 "현재 시세"로 오인**될 수 있음 → "n일 전" 배지로 반드시 신선도 노출(혼동 방지).
- "내부 의사결정용·외부 인용 금지" 정책 문구는 유지.
- 스크레이핑 견고화는 외부 의존이라 100% 보장 불가 — 수동 입력 경로가 항상 동작하도록 보강이 핵심.

## 6. 권장 우선순위
1. **A(만료 제거)** — 1줄 수준 핵심, 즉시 화면 복구(기존 105행 표시됨).
2. **B(라벨 사유 분기 + 신선도 배지)** — 혼동 방지.
3. **C-1/C-3(새로고침 결과 가시화 + 수동입력 격상)** — "왜 안 들어오나" 해소.
4. C-2/C-4(견고화·기록) — 후속.

# admin 취합(정상) vs member 조직현황 취합(미동작) 비교 진단 (구현 0)
> 2026-06-08 · "admin 패널 주간보고 취합은 잘 되는데 member 조직현황은 왜 안 되나"

## 한 줄 답
**둘이 완전히 다른 코드 경로를 쓴다.** admin은 검증된 일괄(non-stream) `mergeAndRefineByCategory`를 쓰고, member 조직현황은 별도 SSE 스트림 `aggregate-stream`(취약한 파서·부서 closure 의존·약한 프롬프트)을 쓴다. **게다가 admin과 동일 로직을 쓰는 `aggregateDept()` Server Action이 이미 있는데 member UI가 호출하지 않는 사장 코드다.**

## 경로 비교
| | admin/reports (정상) | member 조직현황 (안됨) |
|--|--|--|
| 호출 | AdminReportsPreview → `GET /api/reports/preview` | OrgWeeklyView → `POST /api/reports/aggregate-stream` |
| AI 방식 | `generateContent`(일괄) + `parseGeminiJson` 1회 파싱(실패 시 throw) | `streamGenerateContent`(SSE) + `extractObjects` 부분파싱(청크 경계 취약) |
| 프롬프트 | `MERGE_BY_CATEGORY_PROMPT`(통합기준 4질문·표준명칭 규칙·3단계) — 정밀 | `HYBRID_PROMPT`(통합 1문장) — 카테고리 통합 부정확 |
| 출력 스키마 | flat `{category,performance,plan,issues}` | 계층 `{category, authors[]}` |
| 데이터 범위 | `createAdminClient` 전체조직 + week 필터 (RLS우회) | `resolveOrgScope` 부서스코프 + `deptMemberUserIds`(closure 의존) |
| 저장 | sessionStorage(영속X) | `dept_weekly_reports` upsert |

## 근본원인 (member가 admin처럼 안 되는 이유)
1. **별도 취약 경로**: member는 `aggregate-stream`의 SSE 부분파싱 사용 → `route.ts:131`이 `authors` 배열 있는 객체만 인정 → AI가 flat로 답하면 **카테고리 무음 드롭**. admin의 일괄 파싱은 견고.
2. **부서 closure 의존**: `deptMemberUserIds`(org-scope.ts:109)가 `org_node_closure` 서브트리로 부서원 산출 → self-row/descendant 누락 시 `[]` → `route.ts:61` "부서원 없음" 400. admin은 members를 직접 in() 필터라 closure 무관.
3. **약한 프롬프트**: HYBRID_PROMPT의 카테고리 통합 기준이 1문장 → 통합 품질 저하. admin은 정밀 기준.
4. **사장 코드**: `org-actions.ts:41 aggregateDept()`가 admin과 **동일한 mergeAndRefineByCategory**를 호출하고 dept_weekly_reports에 저장하도록 이미 구현돼 있으나, **OrgWeeklyView가 이를 안 쓰고 aggregate-stream을 직접 호출**.

## 권장 방향 (구현은 별도 승인)
- **A안(권장): member 조직현황을 admin과 동일 경로로 통일** — OrgWeeklyView가 `aggregate-stream` 대신 사장된 `aggregateDept()`(=admin의 mergeAndRefineByCategory) 호출하도록 전환. 견고한 파싱·정밀 프롬프트·동일 품질 확보. 단 출력이 flat 스키마라 OrgWeeklyView의 authors[] 렌더링도 flat로 교체 필요.
- **B안: aggregate-stream 유지 보수** — HYBRID_PROMPT 통합기준을 admin 수준으로 보강 + route.ts:131에 flat 객체 fallback + 스트림 종료 flush + closure self-row 백필.
- 공통 점검: `org_node_closure` 부서 descendant 등록 확인(취합 시작 자체의 전제).

## 결론
"admin은 되고 member는 안 되는" 이유 = **SSOT 위반(같은 취합 로직 2벌, member는 취약본)**. 재사용 정책상 member도 admin의 mergeAndRefineByCategory(또는 aggregateDept)를 단일 소스로 써야 한다.

# 02 — 집계 → AI 되먹임 (현실적 3경로)

## 집계 (가중치 산출)
주기적/온디맨드로 사용자별(+전역) 패턴 계산:
- **거부율**: ai_split 항목 중 reject 비율. 높으면 과분해.
- **분류 교정쌍 빈도**: (before_type → after_type) 카운트. 예: planned→done 상위면 "이 사용자는 보고를 완료로 본다".
- **표현 교정 패턴**: content before→after에서 반복 규칙(예: 회사명 약어 선호) — n-gram/임베딩 클러스터.
- **일정 오탐율**: schedule_reject / 자동등록 수. 높으면 일정 추출 보수화.
- **저신뢰 거부 상관**: ai_confidence 낮은 항목의 reject율 → 임계 조정 근거.

산출물 = 사용자별 `correction_profile`(JSON): top 규칙 + few-shot 예시 후보 + 임계 보정값.

## 되먹임 경로 (A+B+C 병행)

### A. 프롬프트 컨텍스트 주입 (개인화 규칙·few-shot)
- analyze-work 프롬프트에 **`{USER_CORRECTIONS}` 변수 신설** (기존 {EXISTING_TODAY} 옆).
- 내용: 최근 교정에서 도출한 **규칙 3~5개 + 예시 2~3개**. 예:
  - "이 사용자는 짧은 메모성 문장을 업무로 분해하지 않음(예: '…확인 필요' → note 유지)."
  - "보고/팔로업 류는 보통 entry_type=done로 본다."
  - few-shot: 원본→(사용자 최종 채택) 예시쌍.
- 장점: 모델 변경 없이 즉효. 단점: 토큰↑(상한·요약 필요).

### B. 결정론 후처리 규칙 (코드)
- AI 출력 후 적용하는 SSOT 규칙 모듈(`lib/daily/feedback-rules.ts`):
  - 과분해 억제: 거부율 높은 사용자 → 짧은 단일 문장은 1건 유지.
  - 일정 보수화: 일정 오탐율 높으면 저신뢰 일정은 **자동등록 보류**(드로어 후보로만).
  - 분류 보정: 빈번한 교정쌍을 사후 매핑(신뢰도 조건부).
- 장점: 예측가능·테스트가능(결정론). 단점: 규칙 폭발 주의 → 상위 N개만.

### C. 신뢰도 임계 보정
- 자동 셋팅(특히 캘린더 자동등록)의 임계를 사용자 수용률로 조정.
- 예: 일정 자동등록은 ai_confidence ≥ θ 일 때만, θ는 schedule_reject율로 상향/하향.

## 거버넌스 연동 (기존 재사용)
- `daily-prompt-governance.ts`의 안전 패턴 그대로: **최소 표본**(예: 신호 ≥ 20)·**최근 윈도우**(최근 N일/건)·**다양성 가드**·**쿨다운**·**degraded 시 롤백**.
- 되먹임은 **held(제안) → 검증 → 활성** 단계(전역 자동활성 금지 — 직전 D-8 정책과 동일).

## 평가(되먹임이 실제로 좋아졌나)
- prompt_version별 거부율/교정율 추세(↓이면 개선). `ai_prompt_outcomes`로 환원해 기존 degraded 감지와 통합.
- A/B: 되먹임 ON 사용자 vs OFF의 수용률 비교.

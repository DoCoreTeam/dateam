# 일일업무 AI 피드백-학습 루프 — 기획 (구현 전, 보고용)

> **상태: 기획만. 코드 없음.** 사용자 승인 후 MVP 슬라이스부터 착수.

## 한 줄
사용자가 AI 자동 셋팅 결과를 **수정/삭제/캘린더 취소/그대로 둠** 하는 행동을 **신호로 수집**해 → 패턴(가중치)으로 집계 → AI 분석 **프롬프트와 후처리에 되먹임**해 점점 그 사용자에 맞게 정제되게 한다.

## 핵심 전제 (현실적 정의)
- Gemini는 호스티드 모델 → **사용자별 진짜 파인튜닝은 비현실적**.
- 따라서 "학습 = **교정 신호를 모아 (a) 프롬프트 컨텍스트(개인화 규칙·few-shot)로 주입 + (b) 결정론적 후처리 규칙 + (c) 신뢰도 임계 보정**". 모델이 아니라 **입력·출력 적응**.

## 이미 있는 토대 (재사용)
- `ai_prompt_outcomes`(품질 신호 테이블, mig094) — 신호 적재 기반.
- `daily-prompt-governance.ts` — 자가조정(degraded 감지·롤백·held·다양성·쿨다운). 안전장치 패턴 그대로 활용.
- `ai_prompts`(프롬프트 DB 버전 관리), analyze-work 프롬프트의 치환 변수 메커니즘({EXISTING_TODAY} 등) → `{USER_CORRECTIONS}` 변수 추가 지점.
- `daily_logs`: ai_processed·ai_confidence·source_type·origin_group_id·original_input(전/후 비교 기반).

## 무엇이 새로 필요한가
1. **신호 수집**: AI 파생 항목에 대한 사용자 행동(삭제/수정/캘린더취소/수용)을 잡는 훅 + 저장. (01 문서)
2. **집계/가중치**: 사용자별·전역 패턴 산출(거부율, 빈번한 교정쌍, 일정 오탐율). (02 문서)
3. **되먹임**: 프롬프트 컨텍스트 주입 + 결정론 후처리 + 임계 보정. (02 문서)
4. **안전장치·단계화**: 최소표본·최근윈도우·노이즈/오삭제 방어·프라이버시·MVP 슬라이스. (03 문서)

## 문서
- `01-signal-schema.md` — 수집할 신호 정의 + 저장 스키마
- `02-feedback-mechanism.md` — 집계 → AI 되먹임 방식(현실적 3경로)
- `03-phasing-risks.md` — MVP 슬라이스 + 리스크/안전장치 + 결정 필요사항

# 03 — MVP 슬라이스 + 리스크 + 결정 필요사항

## MVP 슬라이스 (순차)
**Slice 1 — 신호 수집 (먼저, 되먹임 없음)**
- `ai_feedback_signals` 테이블(mig) + RLS.
- 훅 부착: deleteDailyLog→reject, updateDailyLog→correct_*, unlinkDailyCalendar→schedule_reject. (best-effort, 본 흐름 안 막음)
- 가치: 데이터 축적 시작(되먹임의 전제). 사용자 체감 변화 없음·저위험.

**Slice 2 — 집계/가시화**
- correction_profile 산출(서버) + (선택) 어드민 대시보드: 거부율/교정쌍/일정오탐율 추세.
- accept 배치 잡(24h 미수정→accept) 또는 주간 집계.

**Slice 3 — 프롬프트 되먹임 (경로 A)**
- analyze-work에 `{USER_CORRECTIONS}` 주입(요약·토큰상한). **피처 플래그 + held→검증→활성**.
- 평가: prompt_version별 거부율 추세.

**Slice 4 — 결정론 후처리 + 임계 보정 (경로 B·C)**
- `lib/daily/feedback-rules.ts`(SSOT, 단위테스트) + 일정 자동등록 임계 동적화.

## 리스크 / 안전장치
| 리스크 | 완화 |
|---|---|
| 콜드스타트(데이터 부족) | 최소표본 가드, 부족 시 되먹임 비활성(기본 동작) |
| 노이즈(실수 삭제·되돌림) | 명시신호 우선, 단기 되돌림 무시, 최근 윈도우+decay |
| 과적합(경직) | top-N 규칙만, 다양성 가드, 정기 리셋 옵션 |
| 토큰 비용↑(경로 A) | 규칙 요약·상한, 신뢰도 낮을 때만 주입 |
| 프라이버시 | 본인 RLS, 전역은 익명 통계만, 원문은 본인 되먹임 한정 |
| 전역 오염 | held→검증→활성(자동 전역활성 금지, 거버넌스 재사용) |
| 일정 자동등록 악화 | 오탐율 높으면 임계 상향→자동등록 보류(드로어 후보로) |

## 테스트 전략(요지)
- 집계/규칙은 **순수함수 + 단위테스트**(골든셋: 신호 → profile → 규칙).
- 되먹임 ON/OFF 회귀(프롬프트 변수 주입이 기존 분석 안 깨뜨림).
- 신호 수집 훅이 본 저장/삭제 흐름 무영향(best-effort) e2e.

## 사용자 결정 필요사항
1. **수용(accept) 신호**: 24h 미수정=accept 배치로 잡을지 / 명시신호(삭제·수정·취소)만 쓸지.
2. **전역 학습 포함 여부**: 본인 개인화만 vs 전역 익명 패턴도 활용.
3. **되먹임 범위**: 경로 A(프롬프트)만 먼저 vs A+B+C 전부.
4. **착수 슬라이스**: 권장=Slice 1(수집)부터. (데이터 없으면 되먹임 무의미)

## 권장
Slice 1만 먼저 깔아 **수집 가동** → 2~4주 데이터 본 뒤 Slice 2·3 결정. 되먹임은 거버넌스(held→활성)로 안전하게.

# 일일업무 배치 입력 그룹핑(메인-서브) 설계 기획서

> 상태: **기획 전용 — 구현 금지**
> 작성: 2026-06-01 · DOMANGCHA CEO 파이프라인 (🟦 DC-ANA 내부 탐색 기반)

---

## 1. 문제 정의

`/daily`(일일업무)에서 "AI 저장"으로 한 번에 들어온 N개 업무가 서로 연결고리 없이
평면 카드로 나열됨. 사용자 요구: **"한 번에 입력된 것이면 메인 → 서브들이 달리는 형태"**로 묶어서 표시.

## 2. 핵심 진단 — 묶음 데이터는 이미 존재한다

`daily_logs` 테이블(`apps/web/.../022_daily_logs_relations.sql`)에 이미 보유:

| 컬럼 | 의미 | 채워지는 시점 |
|------|------|--------------|
| `origin_group_id` (uuid, FK→`daily_log_origin_groups`) | **"한 번에 입력된 묶음" 식별자** | AI 저장 시 같은 배치 N개 row에 **동일 uuid 자동 저장** (`actions.ts:391`) |
| `parent_log_id` (uuid, FK→`daily_logs`) | 부모-자식 | 현재 스레드 파생 시에만 |
| `source_type` (`manual`/`ai_split`/`ai_derived`/`thread_derived`) | 생성 출처 | AI 분할 항목 = `ai_split` |

**결론**: 신규 데이터 모델 불필요. 그룹핑이 안 보이는 진짜 원인은 두 가지:
1. UI(`daily/page.tsx`의 `LogList`, L787)가 `origin_group_id`를 무시하고 `logged_at` 순서로 평면 렌더링
2. (이번 설계에선 불필요) AI 응답에 메인 개념 없음 — 모든 항목 동등

## 3. 확정된 설계 결정 (사용자 승인)

| # | 결정 | 선택 | 결과 |
|---|------|------|------|
| 1 | 메인의 정체 | **묶음 헤더 (DB 변경 0)** | 메인 = 실제 업무가 아닌 그룹 헤더. `origin_group_id`로 UI 그룹핑만 |
| 2 | 기본 표시 | **접힘 (메인만)** | 기본은 헤더만, 클릭 시 서브 펼침 |
| 3 | 기존 데이터 | **소급 적용** | 과거 배치도 `origin_group_id` 기준 자동 그룹핑 |
| 4 | 단건/수동 입력 | **모두 묶음 형태** | 단건/수동도 1건짜리 묶음 헤더로 통일 |

→ **마이그레이션 0 · AI 프롬프트 변경 0 · 백엔드 변경 0. 순수 프론트엔드 작업.**

## 4. 그룹핑 키 로직

목록을 받은 뒤 클라이언트에서 그룹 단위로 묶는다.

```
groupKey(log) =
  log.origin_group_id  (있으면)          ← AI 배치 → 같은 배치끼리 묶임
  || `single:${log.id}` (없으면)          ← 수동/단건 → 자기 자신만의 1건 묶음
```

- 그룹 정렬: 그룹 내 최소 `logged_at`(가장 먼저 입력된 시각) 기준 — 기존 시간 순서 보존
- 그룹 내 카드 정렬: 기존과 동일(`logged_at ASC`)

## 5. UI 설계

### 5.1 묶음 헤더 (메인 행)

```
▼ 📥 {그룹 대표 라벨}  ({HH:MM} · {N}건)   [완료 n/N]
```

- **대표 라벨 산출 우선순위**:
  1. `daily_log_origin_groups`의 원문 요약/anchor (있으면)
  2. 없으면 그룹 내 첫 항목 `content`를 말줄임(예: `영진기술 AX 실증산단 …`)
  3. 단건 묶음(1건)이면 그 항목 `content` 그대로
- 시각 = 그룹 최소 `logged_at`
- 건수 배지 = 그룹 내 항목 수
- 진행 요약(선택) = 완료(`done`) 개수 / 전체 — 묶음 단위 완료율 미니 표시
- 헤더 좌측 컬러 바: 그룹 내 우선순위 최고 또는 대표 status 색

### 5.2 펼침/접힘

- 기본 **접힘** — 헤더만 노출
- 헤더 클릭/▶▼ 토글로 서브 카드 펼침
- 펼침 상태는 클라이언트 로컬(예: `useState`/`Set<groupKey>`)로 유지. 새로고침 시 초기화(접힘)가 기본 — 영속 저장은 범위 외
- 단건 묶음(1건)은 헤더가 곧 그 항목 — 펼칠 내용이 자기 자신뿐이므로 **헤더 자체가 카드 역할** (헤더에 status 드롭다운·수정·삭제 액션 인라인). "모두 묶음 형태" 결정을 만족시키되 과도한 빈 헤더를 피하는 절충.

### 5.3 서브 카드

- 기존 카드 UI(status 배지·시각·D-day·content·스레드·플로우·수정·삭제) **그대로 재사용** — 들여쓰기 + 좌측 세로선으로 소속 표현
- 서브 카드 마크업/액션은 변경 없음 (회귀 리스크 최소화)

## 6. 데이터 흐름 (구현 시)

```
SWR: GET /api/daily/logs?date=  ──(변경 없음)──▶ logs[] (평면)
                                                   │
                          [NEW] groupLogs(logs)  ◀─┘   클라이언트 그룹핑 유틸
                                   │
                                   ▼
                          GroupedLogList 렌더
                          ├ 묶음 헤더 (접힘 기본)
                          └ (펼침 시) 기존 카드 N개
```

- **서버/쿼리/액션 무변경** — `/api/daily/logs`, `getDailyLogs`, `addMultipleDailyLogs` 그대로
- 그룹핑은 순수 클라이언트 변환 (`logs[] → groups[]`)
- 캐러리(이월) 목록 등 다른 뷰는 이번 범위에서 제외 (현행 유지)

- 들여쓰기는 모바일에서 과하지 않게 축소(좌측 세로선 + 소폭 패딩)

## 9. 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| `origin_group_id` NULL인 과거 수동 항목 | `single:${id}` 키 → 1건 헤더(= 자기 자신 카드) |
| 묶음 중 일부만 다른 날짜(`target_date`)로 조회됨 | 현재 쿼리는 `log_date OR target_date`로 가져옴 → 그날 화면엔 그룹의 부분 집합만 보일 수 있음. 건수 배지는 **그 화면에 로드된 항목 수** 기준으로 표기(혼동 방지 문구 검토) |
| 묶음 내 카드 개별 삭제로 0건 됨 | 그룹 자동 소멸(클라이언트 그룹핑이라 자연 처리) |
| status 변경 시 묶음 완료율 | 클라이언트 재계산 |

# 04 — 리스크 + 사용자 결정 필요 사항

## 리스크
| # | 리스크 | 완화 |
|---|---|---|
| R1 | 원본+AI 이중 표시로 화면 복잡도↑ | 드로어 **기본 접힘** + 요약 칩(분해N·중복N·일정N·메모N) |
| R2 | 캘린더 **자동** 등록 오탐 → 신뢰 훼손 | P2로 격리 + **사용자 확인형**(자동 INSERT 금지) |
| R3 | 중복 감지 오탐(노이즈) → AI 불신 | 결정론+임계 보수적, "제안만", 채택률 지표로 감시 |
| R4 | original_input 행별 중복 저장(현행) | P0는 표시만(영향 없음). SSOT 정리는 선택·후속(마이그레이션 동반) |
| R5 | 과거 manual 항목 호환 | origin_group 없으면 단일 카드 폴백, 마이그레이션 불요 |
| R6 | AI 입력에 HTML 혼입(주간보고 전례) | daily는 plain 기본, 인용 경로 html→plain 점검 |
| R7 | 현행 "즉시 분해 저장" 흐름 변경 시 회귀 | P0는 표시 레이어만, 저장 로직 무변경 → 회귀 최소 |

## 사용자 결정 필요 사항 (이 답이 있어야 P0 착수)
1. **저장 시점 정책**
   - (A) 현행 유지 — "확인 저장" 시 분해 항목 저장, 원본은 origin_group에 보존 → 드로어는 저장된 묶음 표시. **(권장, 변경 최소)**
   - (B) 변경 — 분해 항목은 확정 전 미저장(순수 후보), 원본만 먼저 저장.
2. **원본 카드 표시 단위**
   - origin_group(입력 1회) 단위 묶음 카드 **(권장)** vs 현행처럼 분해 항목 평면 나열 유지.
3. **이번 범위 확정**
   - (권장) **P0만 먼저** → 검증 후 P1 → P2. vs P0+P1 동시. (P2 캘린더 자동화는 별도 스프린트 권고)
4. **중복 정리 강도**
   - 감지·표기·"병합 요청"까지(자동삭제 X) **(권장)** vs 자동 병합까지.
5. **캘린더 반영 방식(P2)**
   - 후보 체크 → 사용자 확정 등록 **(권장)** vs 일정 추출 시 자동 등록.

## 권장 결론 (DC-BIZ 정합)
- **P0(원본표시+드로어분리+누락메모) 우선** — 데이터 이미 존재, 저비용·고신뢰.
- 중복=제안까지(P1), 캘린더=확인형·별도 스프린트(P2).
- 전 구간 **AI는 제안 / 사용자 확정**, 원본 비파괴 불변.

## 성공 지표(측정)
- 원본 드로어 펼침률 / AI 후보 채택률 / 일일 작성 지속률(분리 전후).

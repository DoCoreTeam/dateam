# 02 — 설계 (2-레이어: 원본 보존 + AI 정제 펼침)

## 2-1. UX 구조
타임라인의 한 "입력"을 **origin_group 단위 카드**로 렌더:

```
┌ 원본 카드 (origin_group) ───────────────────────────────┐
│  [원본]  "영진기술 대표님 컨택하고 시티큐브 PASS 진행,    │  ← daily_log_origin_groups.original_input
│          내일 제주 교육 일정 잡기"                         │     사용자가 쓴 그대로(plain). 절대 비파괴.
│  작성 15:21 · AI 3건 분해 · 중복1 · 일정1 · 메모1   ▾ 펼침 │  ← 요약 칩(접힘 기본)
├─ ▾ 펼치면 AI 분석 드로어 ───────────────────────────────┤
│  ◦ 업무 분해(확정됨)                                      │  ← origin_group_id로 묶인 daily_logs(이미 저장)
│     - 영진기술 대표 컨택  [예정·D-day]                     │
│     - 시티큐브 PASS       [예정·D-15]                      │
│  ◦ ① 중복 의심                                            │  ← P1: 유사 항목 표기(제안)
│     - "시티큐브 PASS" ≈ 오늘 기존 "시티큐브 용인" ? [무시][병합요청]
│  ◦ ② 일정 후보 → 캘린더                                   │  ← P2: 후보 체크 → 사용자 확정 시 calendar_events
│     - ☑ 제주 교육 일정 (내일) → 캘린더에 추가              │
│  ◦ ③ 놓친 메모 환기                                       │  ← P0: 기존 메모 시스템 연계
│     - 확인 안 한 메모 2건이 이 입력과 연관                  │
└──────────────────────────────────────────────────────────┘
```

원칙: **원본 카드는 항상 보임 / AI 드로어는 기본 접힘**(인지부하·신뢰). 모바일은 드로어가 세로 스택.

## 2-2. 데이터 설계 (대부분 재사용)
- **원본 SSOT** = `daily_log_origin_groups.original_input`. 카드 상단은 이 값을 표시.
- **분해 항목** = `daily_logs WHERE origin_group_id = ?` (이미 연결됨) → 드로어 "업무 분해" 섹션.
- **표시 조회** = 기존 `getOriginGroupLogs()` / `groupDailyLogs()` 재사용. 신규 테이블 불필요(P0).
- (선택·후속) `daily_logs.original_input` 행별 중복 저장을 origin_group FK 참조로 정리 → 저장 SSOT화. **데이터 마이그레이션 동반이라 P0 제외, 선택사항.**
- **중복 표기(P1):** 신규 컬럼/테이블 없이 조회 시 계산(파생)하거나, 확정 병합 시에만 `daily_log_relations`(related/duplicate 엣지) 기록.
- **캘린더(P2):** 후보는 비저장(드로어 표시만), 사용자 확정 시 기존 `calendar_events` INSERT(link_kind='daily_log', link_id=해당 분해 항목). 신규 테이블 불필요.

## 2-3. AI 파이프라인 설계 (기존 analyze-work 확장)
현행 흐름 유지 + 출력에 "정제 신호" 추가. **원본 비파괴, 후보 제시 표준 유지.**

| 신호 | 산출 방식 | 저장 |
|---|---|---|
| 업무 분해 | 현행 그대로(SSE) | 확정 시 daily_logs(현행) |
| ① 중복 의심 | **결정론 우선**(제목 정규화 + 유사도; note는 기존 embedding 재사용, task는 문자열/토큰 유사도) → 임계 초과만 "후보" | 미저장(표시). 병합은 사용자 확정 시 relations |
| ② 일정 후보 | AI가 날짜/시간 표현 추출(scheduled_at/target_date) → "캘린더 추가?" 후보 | 미저장. 확정 시 calendar_events |
| ③ 누락 메모 | 기존 메모(memo_status='new') 중 이 입력과 연관/유사(embedding)한 것 환기 | 기존 메모 시스템 |

- 프롬프트는 `ai_prompts` 거버넌스로 버전 추가(현행 교체 아님, 출력 스키마 확장). 토큰 로깅·outcomes 그대로.
- HTML→plain 변환 경로 점검(주간보고 source_quote 사고 전례) — daily는 plain 기본이라 저위험이나 인용 경로 확인.

## 2-4. 비파괴 보장 (가장 중요)
- AI 어떤 신호도 **원본 텍스트를 수정/삭제하지 않음**.
- 중복 "병합", 캘린더 "등록", 메모 "처리"는 모두 **사용자 확정 액션**에서만 DB 변경.
- 실패/오탐 시 원본 대조 가능(원본 항상 표시) → 책임 추적성 확보.

## 2-5. 기존 화면 호환
- 현재 타임라인은 분해 항목을 평면 표시. 신규는 origin_group 카드로 묶어 표시 → `groupDailyLogs()` 이미 존재해 전환 용이.
- **과거 데이터**: origin_group_id 없는 옛 수동 항목(source_type='manual')은 원본=본문 자체로 간주해 단일 카드로 폴백(드로어 없음). 마이그레이션 불요.

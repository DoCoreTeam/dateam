# 기획 — 일일=캡처 / 부서=승격 / 그룹핑·관여도 대시보드 (분석·기획 전용, 미구현)

## 사용자 통찰 (검증 결과: 맞음)
1. **방향이 거꾸로다** — 부서업무에 남긴 게 일일에 역류 노출됨.
2. **AI 정제의 본질 = 그룹핑** (프로젝트별·고객별) + "내가 뭘 얼마나 하는지" 한눈에.
3. **주간보고 같은 "구분 구조"가 없어 일회성 느낌, 관리되는 느낌 없음.**

## 🟦 DC-ANA — 왜 그렇게 보이나 (코드 근거)
- **근본 원인 = 필터 누락(버그)**: `daily_logs` 한 테이블에 일일·부서업무가 같이 산다(`task_kind`). 부서업무 화면은 `.eq('task_kind','dept_task')`로 거르지만 **일일 API 3곳은 task_kind 필터가 없다**:
  - `api/daily/logs/route.ts`(user_id만), `api/daily/week/route.ts`(assignee까지 포함), `api/daily/carryover/route.ts` → RLS 통과 행(부서업무 포함) 전부 노출.
- **그룹핑 재료**: `linked_account_id`(AI가 일부 채움)·`daily_log_tags`·`work_entity_links`(autolink, 방금 추가·콜드스타트). **프로젝트(deal) 축은 사실상 공백**(전용 컬럼 없음, work_entity_links kind='deal' 미적재). **category 필드 없음**(주간보고엔 있음).
- **대시보드**: `/dashboard`는 `/home`으로 리다이렉트 — 실질 없음. 집계 재료(getMonthLogSummary·dept summary)는 있으나 그룹별 뷰 없음.
- **주간보고가 가진 "구조"**: category(분류) + performance/plan/issues(성과/계획/이슈) + week_start(기간 프레임). 일일엔 이 셋 다 없음 → 일회성 느낌의 실제 원인.
- **재사용 자산**: `DeptTaskSuggestPanel`(후보 제안→체크→일괄등록 = 승격 패턴 이미 존재), `groupDailyLogs`, org-scope, autolink/work_entity_links.

## 🟦 DC-RES — 정석 IA/UX
> "일회성 느낌"의 진짜 원인 = **지속되는 그룹 엔티티가 없어서**. 정석 = ①지속 그룹 엔티티 → ②자유 로그를 거기에 연결 → ③롤업해서 보여줌. **AI는 군집을 만들지 말고, 사용자가 정의한 엔티티(고객/딜)에 "연결을 제안"만**(CRM 활동 롤업 + Notion/Linear group-by + GTD inbox→project).
- **그룹 축 우선순위**: ①엔티티 링크(고객/딜 — 이미 보유, 영구·정확) > ②명시 태그 > ③AI 자동군집(보조·제안만, group key로 쓰면 매번 흔들려 "관리 안 됨" 악화).
- **승격 UX**: 캡처(자유)→정제("오늘 정리" 시 AI 후보 제안+체크)→승격(복제 아님, **참조 링크**). §5-3 추출형 패턴 그대로.
- **대시보드(시간 아님, 건수/비중)**: ①관여 분포(도넛 Top5+기타) ②그룹별 진행 롤업 ③활동 추세(주간 스파크). 1화면 3위젯, 과설계 금지.

## 🟦 DC-BIZ — 방향·가치·우선순위
- **방향 재설계: 강한 YES.** 일일=저마찰 개인 inbox / 부서=선별 승격된 상위 레이어. 섞으면 둘 다 오염. → 미결이던 `dept-task-management` 설계를 **ⓐ(daily 재사용)+승격 링크**로 확정 권고.
- **그룹핑 8/10**: "업무관리+영업" 제품 → 일일 로그를 고객/딜 축으로 묶으면 곧 **영업 활동 이력/계정 타임라인**. 축이 이미 데이터로 존재(accounts/deals/work_entity_links).
- **"관리되는 느낌" = 별도 기능 아님**, 승격+그룹핑의 **부산물**. 대시보드 선제 구축 = 과설계.
- **리스크 직언**: AI 자동그룹핑 오분류=신뢰 붕괴(→제안만), 축 과다=죽은 기능(→고객/딜 단일축 시작), 빈 대시보드 데드락(→승격 1클릭, AI로 마찰↓).

## 결론 — 한 기능으로서의 전체 구성 (단계로 쪼개지 않음, 통째 설계)
이 기획은 "1단계만 먼저" 식이 아니라 **하나의 업무체계 재설계**다. 아래 4개 구성요소는 함께 작동해야 의미가 있다.

| 구성요소 | 내용 |
|---|---|
| **A. 일일=캡처 / 부서=승격** | 일일 API 3곳 `task_kind` 필터로 부서업무 역류 제거 + 일일→부서/프로젝트 **승격(참조 링크, 복제X)**. 일일은 개인 inbox, 부서는 선별 승격된 상위 레이어 |
| **B. 그룹핑(고객/딜/프로젝트)** | 확정 엔티티 링크(autolink/work_entity_links)를 그룹 키로 한 group-by. 링크 없는 일일엔 AI가 후보 **제안→체크 확정**(자동확정 금지) |
| **C. 워크로드 대시보드** | 건수/비중/추세 — 관여분포·그룹별 진행 롤업·활동추세. "내가 뭘 얼마나" 한눈에. (시간트래킹 없음→시간차트 금지) |
| **D. 임시저장 + Ctrl+Z (전 영역 공통)** | 작성 중 내용 새로고침 유지(localStorage/IDB) + 되돌리기/다시실행. 일일뿐 아니라 모든 입력면. `lib/forms/useFormCore` 단일 훅. → **상세: `01-draft-and-undo-everywhere.md`** |

> 사용자 지시 반영: "1단계 이런 거 하지 말랬지" — 순서 강제·부분 납품 프레이밍 제거. 위는 **함께 완성되어야 할 한 묶음**. 구현 착수 시 작업 분해는 내부 일정일 뿐, 사용자에겐 통째 기능으로 전달.

## 핵심 설계 원칙 (불변)
- 일일=캡처(자유, 마찰0), 부서=승격된 것만. **승격은 참조 링크(SSOT, 복제 금지).**
- 그룹 키 = **확정된 엔티티 링크(진실)**. AI = 후보 제안만(§5-3), 자동 확정 금지.
- 그룹 축 기본 1개(고객), 토글 최대 2개. 자유태그 무한증식 주의.
- 대시보드 = 건수/비중/추세(시간 트래킹 없음). 3위젯·1화면.
- 미분류 일일도 정상 — 정제 강제 금지.

## 재사용 (신규 최소)
autolink/work_entity_links(그룹 축), DeptTaskSuggestPanel(승격 후보 UI), §5-3 추출형 패턴, accounts/deals, org-scope, getMonthLogSummary·dept summary(집계). 신규 = "personal vs dept 분리 + 승격 참조 + group-by 뷰 + 대시보드 위젯". 신규 테이블은 사실상 불필요(work_entity_links로 충분).

## 미결 결정 (착수 시 Q&A)
1. "프로젝트" = deals를 그대로 쓸지 / 별도 project 엔티티 신설할지.
2. 부서업무 = daily 승격 참조(ⓐ) 확정 vs 별도 테이블(ⓑ).
3. category(분류) 필드 도입 vs daily_log_tags를 분류축으로.
4. 승격 권한(본인만 vs 부서장 큐레이션).

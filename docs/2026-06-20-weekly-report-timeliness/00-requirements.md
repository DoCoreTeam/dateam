# 주간보고 지연 추적·증빙 — 요구사항 (기획 전용 / 구현 미진행)

> 사용자 지시: **기획해서 알려줘 + 절대구현하지마.** 본 문서군은 설계 확정용이며 코드/마이그레이션은 적용하지 않는다.

## 1. 배경 / 인사이트
- 전 구성원의 **주간보고 작성 타이밍**을 데이터로 추적할 필요가 생김.
- 단순 "썼다/안 썼다"가 아니라 **완성도·마감 준수**의 문제.
- 핵심 룰(사용자 정의): **"취합 시점 이후에 작성·수정되는 건 지연"**.

## 2. 목표
1. 주간보고 작성·수정 시각을 **변경 불가하게 기록**(평가 증빙).
2. **취합(consolidation) 기준선** 대비 **지연 자동 판정·표시**.
3. 미작성·지연자에게 **작성 안내**(독촉 아님, "쓰라고" 톤).
4. admin이 **정시율/이력을 증빙으로 export**.

## 3. 확정 요구사항 (Q&A 반영)
| # | 항목 | 확정 내용 |
|---|---|---|
| R1 | 마감 기준선 | **취합(confirm) 이벤트 + 캘린더 2단계 백스톱** |
| R2 | 취합 앵커 | `dept_weekly_reports`(부서 주간보고 취합). 부서장 취합 = 관리자가 부서 골라 하는 취합 = **동일 테이블/이벤트** |
| R3 | 캘린더 백스톱 | **토요일 00:00 = 1차 지연 체크**, **(다음 주) 월요일 00:00 = 최종 지연 체크** |
| R4 | 재취합 | 최신 `confirmed_at`을 취합선으로 쓰되 **토/월 캘린더선은 고정**(재취합으로 지연 우회 불가) |
| R5 | 취합 후 수정 | 취합 이후 원본 수정 = **지연**. 기존 `source_hash`(재취합 필요 감지)와 동일 자리 → 재사용 |
| R6 | 작성시각 진실 | 저장이 `replace_weekly_report`=DELETE+INSERT라 `created_at`이 매 저장마다 리셋됨 → **append-only 활동로그**가 단일 진실원 |
| R7 | 독촉 | 톤 = "작성 안내". **인앱 배지 + 로그인 시 모달**("이번 주 주간보고를 작성해주세요") |
| R8 | 증빙 범위 | **admin만** 정시율/이력 export·조회 |
| R9 | 권한(표시) | 보고서 가시성과 동일(같은부서/관할서브트리/전사) — `lib/org-scope.ts` 재사용 |

## 4. 지연 판정룰 (확정)
주차 W(월요일 시작), 멤버 U의 상태:

| 상태 | 조건 | 배지 |
|---|---|---|
| 정시 | 최종 작성·수정이 **취합 이전** AND **토 00:00 이전** | 🟢 |
| 지연(1차) | **토 00:00 ~ 월 00:00** 사이 최종 작성, **또는 취합 이후 수정** | 🟡 |
| 지연(최종) | **다음 주 월 00:00 이후** 작성·수정, 또는 그때까지 미작성 | 🔴 |
| 진행중 | 토 00:00 전 + 취합 전 (판정 보류) | ⚪ |

### 캘린더선 정의 (확정)
- **토요일 00:00 = 주차 W의 토요일**(week_start+5일) → 1차 지연 체크.
- **월요일 00:00 = 다음 주 시작**(week_start+7일) → 최종 지연 체크.
  → 금요일까지 정시 / 주말 = 1차 지연 / 다음 주 진입 = 최종 지연.

## 5. 범위
- 포함: 활동로그(불변), 취합시각, 지연 판정, 상태 보드/타임라인/배지, 작성안내 모달, admin 증빙 export, 권한.
- 제외(이번 기획): 코드 구현, 마이그레이션 적용, 이메일 발송 인프라(필요 시 별도).

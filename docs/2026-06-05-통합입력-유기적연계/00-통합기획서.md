> 작성: 2026-06-05 · 상태: **기획 확정(결정 반영 완료, 구현 대기)** · 기준 버전: v0.6.85 · 구현 시 등급: **MINOR → v0.7.0**(D1)
> 확정 결정: D1 MINOR · D2 활성화권한=관리자 · D3 **AI 자가갱신 자동반영+사람 롤백** · D4 백필 안 함 · D5 담당자 PII 블랙리스트 제외(노출 허용)

## 0. 배경 — 왜 이 기획이 필요한가

사용자 지시(반복):
> "통합입력은 **어떤 상황에서든** 체크되어 **정확한 수준**으로 데이터가 들어가야 한다. **DB 활용 프롬프트로 확장성 있게**, **우리 스키마를 완전히 파악**하게. 추출된 데이터는 **가야 할 모든 테이블에 유기적으로 연계**되어야 하고, **사람이 같은 걸 여러 번 실수로 넣어도 정합성에 맞게** 들어가야 한다. AI가 프롬프트가 안 맞으면 **자가 변경**할 수 있어야 하고, 그게 **도는지 관리자에서 확인·CRUD**되어야 하며, **AI가 왜·어떻게 바꿨는지 감사로그로 전부** 보여야 한다."

### 현황 분석으로 드러난 핵심 결함 (실측 근거)
| 결함 | 증거 |
|------|------|
| **재고 증발** — AI는 `quantity.resp_qty`를 추출하나 저장 경로가 버림 | active 프롬프트 v2.0에 `{quantity}`×11·`{resp_qty}`×3 존재 / `review/[id]`·`stream`·`commit` 전부 `availability_responses` INSERT **0건** |
| **스키마 부분 파악(~35%)** — 화이트리스트 8개 하드코딩 | `058_schema_digest_fn.sql:18-21` ARRAY 고정. `availability_responses·direct_pool_stock·suppliers·accounts·contacts·gpu_audit_logs` 비가시 |
| **AI 자가갱신 반쪽** — draft 생성만, 가시화·검수·승격 루프 없음 | `extract-helpers.ts:157` draft(active=false) 저장 / 관리자 CRUD·드래프트 큐 UI 전무 |
| **감사 부재** — 누가/AI가 왜·어떻게 바꿨는지 기록 구조 없음 | `ai_prompts`에 updated_at·updated_by·revision 이력 없음 |

### 근본원인 1줄
> **추출 스키마(프롬프트)와 영속 경로(저장 핸들러)가 분리되어, 계약 불일치를 잡는 게이트가 없고, 스키마 가시 범위가 하드코딩 화이트리스트로 고정되어 확장성·완전성이 구조적으로 막혀 있다.**

## 1. 설계 원칙

1. **증발 불가(no-silent-drop)** — 추출된 필드는 반드시 저장 대상이 있고, 없으면 빌드가 차단.
2. **자동 확장(opt-in by metadata)** — 새 테이블 연계는 코드 수정이 아니라 메타데이터(테이블 코멘트) 한 줄.
3. **단일 라우팅 SSOT** — 모든 AI 진입점이 하나의 추출↔저장 라우터를 거친다.
4. **멱등 정합** — 같은 입력 N회 = 결과 1건(자연키 UPSERT/supersede).
5. **AI 자율 + 사람 통제** — AI는 draft 제안까지, 활성화는 사람 승인. 모든 변경은 감사로그.
6. **무회귀** — 가격 SSOT(`supply_quotes`) 로직 동결, 신규는 additive.

## 2. 8축 아키텍처

### 축 1 — 스키마 완전 파악: 화이트리스트 폐기 → 테이블 코멘트 자동 스캔
- `get_schema_digest()`를 ARRAY 하드코딩 대신 `COMMENT ON TABLE … IS 'ai:intake'` 태그 테이블 자동 포함.
- 새 연계 = 코멘트 한 줄, SQL/TS 수정 0 (확장성).
- 대상 태깅: `availability_responses·direct_pool_stock·suppliers·accounts·contacts`.
- **민감테이블 강제 제외(G1)**: `profiles·auth.*·api_keys`만 블랙리스트 차단. **담당자(`accounts·contacts`)는 노출 허용(D5=no)** — 유기적 연계 위해 AI 가시.
- `schema-contract.ts` 정적본은 폴백, 런타임 다이제스트가 SSOT(드리프트 테스트 감시).

### 축 2 — 추출↔저장 라우팅 SSOT: `lib/gpu/intake-routing.ts`
- 추출 JSON 섹션 → 대상 테이블 writer 매핑을 한 파일에 선언:
  `price→supply_quotes` · `availability→availability_responses` · `account→supply_quotes.account_id` · `contact→contacts`.
- `review/[id]`·`commit`·구형 `review`·`recheck`·`reanalyze` **전 경로가 이 라우터만 호출**(단일구현).
- 매핑 계약은 `ai_prompts.output_schema`(G6)에서 파생 — 추출 선언과 저장 매핑이 같은 출처.

### 축 3 — confirm 다중 테이블 트랜잭션 저장: DB RPC `commit_intake()`
- 가격 + 재고(+account_id 백필 + 담당자)를 한 트랜잭션으로 저장. 부분성공/유령데이터 방지.
- 재고: `quantity.resp_qty` 있으면 `availability_responses` upsert(freshness·expires_at 갱신).
- **부분 커밋(G2)**: 섹션별 검증 — 가격 유효·재고 누락이면 가격만 저장 + 재고 skip 사유 기록(증발 아님).
- **무회귀**: 가격 INSERT 블록 동결, 재고/account는 별도 함수로 분리.

### 축 4 — 계약 불일치 게이트 (어떤 상황에서든 체크)
- 빌드타임 테스트: 추출 스키마(`output_schema`) 선언 필드 중 `intake-routing`에 저장 대상 없는 필드 = **FAIL**.
- → 프롬프트가 새 필드를 뽑기 시작하면 저장 라우팅 없을 시 코드리뷰 전 차단(재고 증발 재발 불가).
- H1 검증 게이트를 재고 필드(resp_qty 음수·범위·is_total_capacity)까지 확장.

### 축 5 — 중복 입력 멱등 정합 (사람 실수 흡수)
- **자연키 UPSERT만**(INSERT 금지):
  - `supply_quotes`: product+supplier+term+pricing_model → 내용동일=no-op / 값변경=기존 supersede+신규 active.
  - `availability_responses`: product+supplier → 최신 upsert(누적 X).
  - `accounts/contacts`: 존재 시 재사용.
- **3중 방어**: ①추출 dedup(`lib/gpu/dedup.ts` 재사용) ②영속 라우터 자연키 판단 ③DB unique index(`uq_active_quote_per_supplier` 등) ON CONFLICT(G4 동시성).
- 재고는 "현재 가용량"이라 덮어씀(더하지 않음).

### 축 6 — 프롬프트 CRUD + AI 자가갱신 자동반영 + 사람 롤백: `/admin/ai-prompts` 신설
> `admin/settings/actions.ts`(작업중) 미접촉 — 독립 신규 페이지.
> **결정(D3): AI 자가갱신은 자동 반영(active)이 필수. 사람의 역할은 사전승인이 아니라 사후 롤백.**
- **6-1 CRUD**: 전체 `ai_prompts` 목록(조회→수정 모달), content 편집, active 토글, 버전 생성. 같은 key active 1건 강제(부분 unique index). 수동 편집·롤백 권한 = 관리자(D2).
- **6-2 AI 자가갱신 자동반영 루프 (사전승인 없음)**:
  ```
  AI "안 맞음" 판단(트리거: 추출0건·저신뢰·게이트 다수차단)
    → 새 버전 생성 → [자동 게이트] 골든셋 eval 통과 시 즉시 active 전환(자동반영)
                                    └ 미통과 시 draft 보류 + 관리자 알림(폭주 방지)
    → 라이브 모니터: 활성 후 추출0건/게이트차단 급증 감지 시 → [자동 롤백] 직전 버전 복귀 + 알림
    → 관리자: 변경 이력 타임라인에서 언제든 [수동 롤백](원클릭, 직전/임의 버전 복원)
  ```
  - **안전망(자동반영의 대가)**: ①활성 전 골든셋 자동 eval 게이트(나쁜 버전 자동활성 차단) ②활성 후 라이브 품질 급락 시 자동 롤백 ③직전 active 버전 항상 보존(즉시 복원). → "자가갱신 필수"를 살리되 catastrophic 자동활성은 구조적으로 차단.
  - **비용 가드(G5)**: 입력해시 dedup(존재) + 레이트리밋 + 보류 draft TTL.
- **6-3 감사로그(왜·어떻게 전부 추적)**:
  - 이벤트 6종: `ai_prompt_auto_activated`(AI 자가갱신 자동반영) · `ai_prompt_auto_rolled_back`(라이브 급락 자동롤백) · `prompt_rolled_back`(사람 수동롤백) · `prompt_edited`(사람 편집) · `prompt_activated/deactivated` · `ai_prompt_held`(eval 미통과 보류).
  - 각 로그: 왜(trigger·reason·evidence) + 어떻게(version_from→to·revision_id·diff_summary).
  - 저장: `ai_prompt_revisions`(append-only 전문 스냅샷) + `gpu_audit_logs`(메타, 기존 재사용).
  - 관리자 "변경 이력" 탭: 타임라인 + before/after diff 뷰어 + actor/trigger 필터 + 데이터품질 카드 점프.

### 축 7 — 스키마 가시화 (`/admin/ai-prompts` 탭)
- `get_schema_digest()` 결과 그대로 표시 = "AI가 지금 보는 스키마".
- 태그된 테이블·컬럼·제약 표시(축1 직결 — 태그 추가하면 즉시 반영). 태그 안 된 테이블은 "AI 비가시" 경고.

### 축 8 — 보안·견고성 (자가감사 보강)
| 코드 | 항목 | 처리 |
|------|------|------|
| G1 | PII/민감테이블 AI 노출 | 민감 블랙리스트 강제 제외(profiles·auth·api_keys) |

## 3. 전체 흐름 (8축 통합)

```
입력(메일·이미지·URL·혼합 — 어떤 상황)
 → [축1] 스키마 완전파악(코멘트 자동스캔, 민감 제외)으로 AI가 모든 대상 인지
 → AI 추출 + [축5①] 입력내 dedup
 → [축2] 단일 라우터 ──[축5②] 자연키 UPSERT/supersede 판단──┐
 → [축3] 트랜잭션 다중저장(가격·재고·account·담당자, 부분커밋)   │
      └ [축5③] DB unique 제약 = 최종 안전망(ON CONFLICT) ──────┘
 → [축4] 계약 게이트: 추출했는데 저장대상 없는 필드 = 빌드 차단
결과: 무엇이·몇 번 들어와도 → 정확히 1건, 올바른 모든 테이블에, 무손실·무중복

[운영/신뢰 평면]
 [축6] 프롬프트 CRUD + 자가반영 검수루프 + 감사로그(왜·어떻게)
 [축7] 스키마 가시화   [축8] 보안·견고성
```

## 5. 마이그레이션 안전성
- 전부 **ADD → backfill → 신규경로부터 사용** (additive, 롤백 가능). 기존 데이터 0건 손실.
- 가격 경로 동결로 회귀 0 보장. 과거 데이터 백필(account_id·누락 재고)은 별도 선택 작업으로 분리.

## 6. 검증 전략
1. **계약 드리프트/커버리지 테스트** — 추출필드 전부 저장대상 보유(축4).
2. **골든셋 eval 100%** — 재고 반영 케이스 + 멱등 케이스(동일 3회→1건) + 부분커밋 케이스.
3. **E2E**: ①재고 포함 견적 확정→재고탭 자동반영 ②같은 입력 2회 확정→수치 무변동 ③미준비 입력→AI draft 제안→감사로그 기록→테스트비교→승격→반영.
4. **회귀**: 가격표 SSOT 무변경.
5. **보안**: 스키마 다이제스트에 민감테이블 미포함 확인, 프롬프트 활성화 권한 검증.

## 8. 완료 정의 (Definition of Done)
- [ ] 통합입력 재고 포함 견적 확정 → 재고탭 자동 반영(E2E)
- [ ] 같은 입력 N회 → 가격·재고 1건 유지(멱등 E2E)
- [ ] 추출했는데 저장대상 없는 필드 → 빌드 차단(계약 게이트)

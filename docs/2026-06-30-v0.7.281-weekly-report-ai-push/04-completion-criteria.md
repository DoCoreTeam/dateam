# 04 · 완료 기준 + 단계별 롤아웃 (기획)

> 🟦 DC-BIZ 판정 = **CONDITIONAL GO**. 아래 단계/조건은 그 판정을 반영한다.
> 본 문서는 기획 완료 기준이며, **구현 착수는 사용자 승인 후 별도**.

## A. 단계별 롤아웃 (DC-BIZ 우선순위 권고 반영)

### Phase 1 — 자동초안(push) + 자동/수동 분리 ★최우선
가치의 80%, 재사용 자산으로 최저비용 검증. 단독 출시 가치 있음(사용자가 직접 확정).
- [ ] 화면 진입 시 저장본 없으면 AI 자동초안 생성·표시 (FR-1)
- [ ] 카테고리(구분) → 성과/계획/이슈 2층 배치, origin=auto, confidence 마커 (FR-2)
- [ ] 카테고리 참조계층: 개인 지난주 → (확장) 부서
- [ ] 자동항목 체크박스(포함/제외)·X 삭제 + 수동 에디터 영역 공존, origin=manual (FR-3)
- [ ] 다음 주 계획 = +1주 캘린더 미래 일정 (FR-4)
- [ ] 초안 저장·캐시, 재진입 AI 재호출 없음 (FR-5)
- [ ] **멱등/락 가드**: user×week 1회 생성, 입력 토큰 압축 (DC-BIZ 조건 ③)
- [ ] 데이터 0건 graceful degrade + 기존 pull/기존 데이터 하위호환 (FR-8)
- [ ] 첫 진입 "초안입니다 — 감수·가감하세요" 신호 명확(방치→오확정 방지)

### Phase 2 — 출처 기반 취합
- [ ] 부서 취합이 auto/manual origin 태그 보존하며 동등 취합 (FR-7, 엔진B 재사용)

### Phase 3 — 기한 자동확정 ★마지막·신중 (DC-BIZ 조건 ①②)
> 리스크 최고·가치기여 최소. **공정성 분리 설계 확정 전 착수 금지.**
- [ ] **timeliness 집계에서 actor=system 자동확정분 분리** (정시 실적 비포함 또는 별도 수치) — **선결조건**
- [ ] 사전알림 실제 도달 보장 후에만 자동확정 (통보 없는 박제 금지)
- [ ] cron(주) + lazy(조회/취합 시 백업) 혼합 실행기, actor=system 불변로그
- [ ] (대안) 초기에는 자동확정 대신 "강한 리마인드 + 수동확정 유지"로 시작 가능

## B. DC-BIZ CONDITIONAL GO 조건 (박제)

1. ✅ Phase 1(①자동초안 + ②origin분리)을 1차로 먼저 출시, ③자동확정은 후속 분리.
2. ✅ 자동확정분은 timeliness/인사 집계에서 actor=user와 분리 (정시실적 비포함/별도수치) — ③ 선결.
3. ✅ 화면열때생성에 멱등/락 가드 + 입력토큰 압축으로 비용통제.
4. ✅ 신규 스키마는 weekly_reports 호환 보존, 기존 상태플래그 무단 덮어쓰기 금지.

## C. 횡단 완료 기준 (모든 Phase)

- [ ] datetime 전부 `lib/datetime/kst.ts` SSOT (캘린더 ±9h 사고 방지)
- [ ] 표시 HTML은 `RichText`, AI 입력 인용은 `html-to-plain` SSOT
- [ ] 신규 테이블 RLS default-deny + 본인 스코프
- [ ] 분류·생성 로직 `lib/weekly-report/` 단일구현(SSOT), FE/API import만
- [ ] 실제 렌더 경로(`?tab=mine`) Playwright 실측 통과
- [ ] 디자인 토큰·공용 컴포넌트(input-field/label/NbButton 등) 준수, `pnpm design:check` 통과
- [ ] 테스트 파일 `apps/web/package.json` test 목록에 등록

## Q1. 여러 화면 확인 = 렌더링 코드가 우후죽순?

**아니다.** "전수 확인"은 아키텍처 품질과 무관한, 완전성 감사(이 버그가 다른 곳에도 있나?)의 본질이다.
React/Next App Router 앱은 **라우트마다 독립 컴포넌트 트리**를 갖는 것이 표준 설계다. 화면이 많아 컴포넌트가 많은 것은 정상 분해(decomposition)이지 산개(fragmentation)가 아니다.

## Q2. 실제 구조는?

### 측정값 (정량)
| 항목 | 수치 |
|------|------|
| 클라이언트 컴포넌트 `'use client'` (app/) | 117 파일 |
| `useEffect` 사용처 | 134곳 / 81 파일 |
| SWR 데이터 훅 `useSWR/useSWRInfinite` | 109곳 / 35 파일 |

### 공용화(SSOT) 인프라 — 잘 갖춰짐
- 공용 UI: `components/ui/*` (NbButton·NbCard·MobileShell·RichText·모달) — 화면별 재구현 안 함
- 공용 로직: `lib/*` (datetime/kst, gpu/dedup·tier·normalize, html-to-plain) — CLAUDE.md "재사용·단일구현 정책"으로 강제
- 데이터층: `SWRProvider` + `lib/gpu/swr-keys.ts`(키 SSOT)
- 디자인: `globals.css :root` 토큰 단일 소스 + `design:check` CI 가드

→ **렌더 뷰 계층은 화면별 분해(정상), 공용 요소는 중앙집중(양호).** 우후죽순 아님.

## 진짜 이슈 — 데이터 패칭 idiom 이원화 (사용자 직감의 절반이 맞음)

렌더는 깔끔하나, **데이터 가져오기 방식이 2갈래로 공존**하는 것이 이번 깜빡임 버그의 구조적 배경이다.

| 방식 | 규모 | 루프 안전성 |
|------|------|-------------|
| ✅ SWR (중앙집중·권장) | 109곳/35파일 | 캐싱·dedup·`isValidating` 가드로 무한루프 구조적 차단 |
| ⚠️ 직접 fetch (컴포넌트 내 `useState+useEffect+fetch`) | 다수 컴포넌트 | loading/empty/error 각자 손코딩 → 공용 가드 없음. **DailyTaskSelector가 이 방식**, 여기서 루프 발생 |

**근본 원인: 공용 `useFetch`/`useAsync` 훅 부재.** 손코딩 fetch마다 로딩·빈결과 로직을 재구현 →
(1) 버그가 특정 컴포넌트에만 숨을 수 있고 (2) "다른 데도 있나?" 검증 시 손코딩 fetch를 하나씩 다 열어봐야 함(= 사용자가 느낀 검증 비용).

## 권고 (미구현 — 방향만)
1. 신규/수정 시 데이터 패칭은 **SWR로 일원화** (이미 인프라 존재). 손코딩 fetch 신규 도입 지양.
2. SWR가 과한 소형 위젯용으로 **공용 `useAsync` 훅 1개**를 `lib/`에 두고 loading/empty/error·중복요청 가드를 캡슐화 → 손코딩 재발 방지.
3. 기존 손코딩 fetch 컴포넌트는 점진적으로 위 둘 중 하나로 수렴(대규모 리팩터링 불필요, 신규부터 규율).

> 결론: 구조는 "우후죽순"이 아니라 **정상 분해 + 양호한 공용화**. 유일한 실질 갭은 **데이터 패칭 패턴 이원화(공용 fetch 훅 부재)**이며, 이것이 검증을 여러 화면에 걸치게 만든 원인이다. 코드는 일절 수정하지 않았다.

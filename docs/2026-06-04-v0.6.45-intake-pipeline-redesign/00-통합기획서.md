## 0. 한 줄 요약

통합입력(`/intake`)이 **"코드가 판단하고 AI가 빈칸만 채우는"** 경직 구조 + **"입구(카피)–과정(처리)–출구(결과)" 가시성 부재**로 사용자 혼란을 일으킨다.
→ **AI가 판단하고 코드는 자원조달·계약강제만 하는 Schema-Aware 다단 파이프라인**으로 전환하고, **자동분류 약속 카피 → SSE 실시간 진행 → 커밋상태(LIVE/PENDING) 결과 모달**까지 한 흐름으로 일관화한다.

## 1. 문제 정의 (대화에서 도출된 7개 근거)

| # | 문제 | 근거(실측) |
|---|------|-----------|
| P1 | URL 멀티 미지원·SPA fetch 무력 | `route.ts:334` 정규식 비-global / `fetchUrlText` 정적 fetch — NHN SPA는 본문 0 |
| P2 | 사용자 지시문·2번째 URL 소실 | `route.ts:340` `contentText = fetched` 원문 덮어쓰기 |
| P3 | "기존 스펙 추론" 미구현 | review 경로에 `gpu_specs`/`gpu_products` 스펙 주입 코드 전무 |
| P4 | 코드 하드분기(경쟁사/공급사 2지선다) | `CLASSIFY_PROMPT` 코드 하드코딩(L116), 단발 추출 |
| P5 | 결과 불투명 — "내 입력이 뭘 했나" 모름 | 응답이 "결과값"만, 신규/갱신·테이블·환산 미반환 |
| P6 | 경로 의미 비대칭 미고지 | 경쟁사=`market_prices` 즉시 LIVE / 공급가=`review_items` pending(무변경) |
| P7 | 카피 프레이밍 갭 | 카피 100% "견적 전용" vs 기능=경쟁사·공급가·URL 멀티 |

**인과 사슬:** P7(입구 오안내) → P1~P4(처리 깨짐) → P5/P6(출구 불투명) — 한 문제의 입구·과정·출구.

## 2. 설계 원칙 (사용자 확정 사항)

1. **프롬프트 코드 하드코딩 금지** — 전부 `ai_prompts`(DB). 이미 레포 표준(`analyze-work` 주석 "하드코딩 대신 DB 로드").
2. **판단은 AI, 계약(schema)은 코드** — 자유도는 AI, 안정성은 Zod 스키마 강제.
3. **고정 스키마 전제** — 마이그레이션/신규 테이블은 런타임에 안 함. 9개 고정 테이블이 타겟. → 신규 레지스트리 테이블 불필요.
4. **스키마 매개체는 SSOT 자동파생 + drift 게이트** — 손 관리 금지(깜빡=오염).
5. **RAG는 스키마용 아님, 엔티티정합용** — 스키마는 통째 주입(수 KB). RAG는 모델명/공급사명 모호 매칭에만(기존 pgvector 재활용).
6. **실시간 스트리밍(SSE)으로 과정 가시화** — 기존 6곳 패턴 재활용.
7. **결과는 커밋상태(LIVE/PENDING)로 명시** — 공급가 pending 게이트(안전장치)는 유지, 가시성만 추가.
8. **반응형·i18n·인라인 style 금지** — CLAUDE.md 정책 준수.

## 3. 타겟 아키텍처 — Schema-Aware 다단 파이프라인

```
[입구] 자동분류 약속 카피 (P7 해소)
   │  사용자가 무엇이든 붙여넣음 (경쟁사/공급가/URL 혼합 가능)
   ▼  SSE 채널 오픈 ── 이하 모든 단계 실시간 중계
┌─ PASS 1: 이해(Understand) ──────────────────────────────┐
│ 입력 원문 전체 + [스키마 계약서] 주입                     │
│ AI 판단: URL 몇 개? 경쟁사/공급가/혼합? 사용자 지시?      │
│          모델 단서? 기존 스키마 매핑 가능?                │
│ 📡 "URL 2개 감지 · 경쟁사 가격 추정"                      │
└──────────────────────────────────────────────────────────┘
   │ 코드(얇은 오케스트레이터, 판단 안 함): plan 읽고 자원만 조달
   │   • urls[] 전부 fetch(병렬)  • needs_specs → gpu_specs/products SELECT
   │   • 매핑 모호 → [Layer2 RAG] 엔티티 후보 검색(선택)
   ▼
┌─ PASS 2: 구조화(Extract) ───────────────────────────────┐
│ (DB 프롬프트) + 스키마 계약서 + 조달자원으로 최종 JSON    │
│ competitor_items[] + supplier_items[] 동시 산출           │
│ enum·정규화·단위·FK 규칙 준수(계약서 근거)                │
│ 📡 "H100 80GB $2.39 추출 · A100 기존행 매칭"             │
└──────────────────────────────────────────────────────────┘
   ▼ 코드: 저장 (기존 saveCompetitorPrices / review_items 재활용 — 무수정)
   │   처리명세(manifest) 수집: 대상·테이블·신규|갱신·환산·커밋상태
   ▼
[출구] 결과 요약 모달 (P5/P6 해소)
   🟢 경쟁사: 시장가 즉시 반영(LIVE) + 되돌리는 법
   🟡 공급가: 검토 대기(아직 무변경) + "확정해야 반영" CTA
   📡 "완료 — 3건 처리" → 스트림 종료
```

### 미준비 입력 대응(자기적응)
PASS1이 "기존 스키마 매핑 불가" 선언 시 → AI에 메타호출로 **추출 프롬프트 합성** → `ai_prompts`에 `status='draft'` 격리 저장(자동 active 금지) → 1회 사용 + 사람 검수 후 승격. **출력 스키마는 고정** — 합성은 "입력 읽는 법"만.

## 4. 스키마 매개체 + RAG 2계층

```
Layer 1 (필수·항상 주입): 스키마 계약서
  - 9개 테이블 × [컬럼·타입 | enum 허용값 | 정규화규칙 | 단위변환 | FK흐름 | 샘플행]
  - SSOT(마이그/information_schema)에서 자동 파생 → 수 KB → 프롬프트 1회 주입
  - drift 게이트: 마이그 추가 시 재생성 강제(CI/훅), 불일치 시 빌드 차단
  - 저장 위치: 정적 산출물 또는 기존 ai_prompts/org_content 행 (신규 테이블 0)

Layer 2 (선택·모호할 때만): 엔티티정합 RAG
  - 기존 gpu_products/competitors 행 임베딩 → 입력 모델명/공급사명 유사도 매칭
  - 인프라 재활용: 042 pgvector + ivfflat + gemini-embedding-001(768d)
  - 효과: 중복행 방지("4090"="RTX 4090"), 정확 FK 연결
  - 마이그: Layer2 갈 때만 임베딩 컬럼 소량 추가
```

## 5. 작업 분해 — 무중단 루프 실행 명세 (Ralph-style)

> 각 STEP: **작업 → 자가검증(통과조건) → 통과 시 다음 STEP 자동진입 / 실패 시 동일 STEP 내 자동수정 후 재검증**. 멈춤 없음. 3회 연속 실패만 에스컬레이션.

| STEP | 작업 | 자가검증(통과조건) | 산출물 |
|------|------|-------------------|--------|
| S1 | 스키마 계약서 생성기 (SSOT 파생) | 9개 테이블 enum·정규화 규칙 누락 0, 파일 생성됨 | schema-contract |
| S2 | drift 게이트(CI/훅) | 마이그 변경 후 계약서 미갱신 시 빌드 fail 재현 | gate script |
| S3 | 프롬프트 DB 이관 (P4) | `CLASSIFY_PROMPT` 등 코드 하드코딩 0, `ai_prompts` 행 존재 | ai_prompts seed |
| S4 | `route.ts` SSE 전환 (과정 가시화) | 단계 이벤트 스트림 수신 확인 | stream route |
| S5 | PASS1 이해 패스 (멀티URL·혼합·지시문 보존) — P1/P2 | URL 2개·지시문 모두 plan 반영 | pass1 |
| S6 | 자원조달 오케스트레이터 (URL 병렬 fetch + 스펙 주입) — P3 | 기존 스펙이 프롬프트에 주입됨 | orchestrator |
| S7 | PASS2 구조화 (competitor+supplier 동시, 계약 준수) | enum/정규화 위반 0, Zod 통과 | pass2 |
| S8 | 처리명세 반환 보강 (saveCompetitorPrices) — P5 | action(created/matched)·테이블·환산·커밋상태 포함 | manifest |
| S9 | 결과 요약 모달 (LIVE/PENDING 구분) — P5/P6 | 경쟁사=LIVE·공급가=PENDING+CTA 표기, 반응형 | modal |
| S10 | 자동분류 약속 카피 + i18n — P7 | 견적전용 표현 제거, 경쟁사·URL 노출, H1중복 제거 | copy/i18n |
| S11 | (선택) Layer2 엔티티 RAG | 중복행 생성 테스트 통과 | rag |

**완료조건(전체):** P1~P7 전부 자가검증 ✅ + 회귀 0(기존 저장부 무수정) + **§5.1 브라우저 UI 테스트 PASS + §5.2 DB 검증 PASS**.

## 5.1 브라우저 UI 테스트 (필수 — 루프 종료 게이트)

> **모든 STEP은 코드 통과만으로 완료 불가.** 실제 사용자 UI 흐름을 브라우저(Playwright/MCP)로 구동해 PASS해야 다음 STEP 진입. 단위/타입 통과는 필요조건이지 충분조건 아님.

| TC | 시나리오 (사용자 관점) | 통과 기준 |
|----|----------------------|-----------|
| T1 | `/intake` 진입 → 카피 확인 | 경쟁사·공급가·URL 멀티 용도가 화면에 노출, H1 중복 없음 |
| T2 | 경쟁사 텍스트 붙여넣기 → 분석 | SSE 진행 단계 실시간 표시 → 결과 모달 **🟢 LIVE** 표기 |
| T3 | 공급가 견적 붙여넣기 → 분석 | 결과 모달 **🟡 PENDING + "확정해야 반영" CTA** 표기 |
| T4 | 멀티 URL(KR1·KR3) 붙여넣기 | URL 2개 모두 처리(plan 반영), 지시문 보존 |
| T5 | 모델 모호 입력 | "AI가 GPU 모델 인식 못함" 무한반복 아님 — 추론/안내로 귀결 |
| T6 | 모달 → 바로가기 CTA 클릭 | 시장비교/검토대기 정확한 위치로 이동 |
| T7 | 반응형(375·768·1440) | 모달·결과표 가로스크롤 없음, 카드 변환 |

검증 방식: `mcp__playwright__*` 또는 `apps/web/e2e/*.spec.ts`로 실제 클릭·입력·스냅샷. 콘솔 에러 0.

## 5.2 DB 검증 (필수 — 루프 종료 게이트)

> UI가 "됐다"고 말해도 **DB 실제 반영을 SQL로 확인**해야 PASS. 화면 메시지 ≠ 데이터 진실.

| DC | 확인 대상 | 통과 기준 |
|----|----------|-----------|
| D1 | 경쟁사 경로 | `competitors`·`competitor_product_mapping`·`market_prices` 행 실제 생성, 중복행 없음 |
| D2 | 공급가 경로 | `review_items` status=`pending` 생성, **가격 SSOT(supply_quotes) 무변경** 확인 |
| D3 | 단위 변환 | `market_prices.price_usd` 환산값 정확, `notes`에 원본 기재 |
| D4 | enum·정규화 | `pricing_model` enum 위반 0, `memory` 정규화 일관(중복 모델 없음) |
| D5 | 처리명세 일치 | 모달이 보여준 신규/갱신·테이블 = DB 실제 상태 일치 |

검증 방식: `mcp__claude_ai_Supabase__execute_sql` 또는 `list_tables`로 실제 조회.

## 5.3 실패 시 자동 해결 (루프 — 멈춤 없음, 필수 포함)

```
테스트(§5.1/§5.2) 실행
   │
   ├─ PASS → 다음 STEP 자동 진입
   │
   └─ FAIL → 🔧 근본원인 진단 → 수정 → 동일 테스트 재실행 (조용히 반복)
              └ 3회 연속 실패 시에만 [ESCALATION] 근본원인+대안 3가지 보고
```
**원칙:** 테스트에서 이상 발견 = **반드시 해결까지 동일 루프 내 처리.** "발견했다"로 멈추거나 다음 스프린트로 미루기 **금지**(EXEC-002). 해결 후 전체 회귀 재실행.

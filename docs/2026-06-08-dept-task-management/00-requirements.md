# 부서 업무 관리 — 요구사항 정의서 (기획)

> 작성일 2026-06-08 · 상태: **기획(구현 0)** · 대상 버전 제안 v0.8.0
> 본 문서는 설계 합의용 기획서다. 코드/마이그레이션/커밋은 일절 포함하지 않는다.

## 1. 문제 정의 (Why)

현재 newAX의 업무 기록은 **개인 단위**다.
- `daily_logs`(일일업무) = 사용자가 *자기* 업무를 기록·관리. 작성자=소유자 고정.
- `weekly_reports`(주간보고) = 주 단위 개인 회고를 부서 취합으로만 굴림.

→ **부서장이 "부서의 업무가 지금 어디까지 진행됐는지"를 실시간으로 알 수 없다.** 개별 일일업무를 일일이 들춰보거나 주간보고를 기다려야 한다. 부서 공동 과제에 담당자를 지정하고 진행을 추적하며, 담당자·부서원이 진행상황을 댓글로 남기는 **협업 프로세스가 없다.**

## 2. 해결 목표 (What)

부서장(및 부서원)이 **부서 단위 업무**를 등록 → 담당자 지정 → 상태/진행률로 추적하고,
담당자·부서원이 **댓글로 진행상황을 누적 기록**하며, 그 진행이 **주간보고에서 확인**되는 프로세스.

## 3. 절대 설계 원칙 (사용자 지시 — 최상위 제약)

1. **기존 일일업무(daily_logs)·기존 프로세스를 최대 재사용한다. 완전히 새로운 프로세스를 만들지 않는다.**
2. 조직 계층 권한(`org_nodes`/`lib/org-scope.ts`/`private.my_*` 함수/v0.7.38 계층플래그)을 **그대로 재사용**한다 — 새 권한 모델 신설 금지.
3. 과설계 금지: 협업툴(멘션·실시간·간트·첨부·알림)을 MVP에 넣지 않는다.
4. 일일업무(개인) ↔ 부서업무(공동)의 **개념·화면을 명확히 분리**한다(혼동 방지).

## 4. 사용자 & 역할

| 역할 | 권한(재사용: org-scope) |
|------|------|
| 부서장(`head_user_id`) | 관할 부서·하위 서브트리 부서업무 등록·담당자 지정·수정·삭제, 진행 추적, 댓글 |
| 부서원(person, 소속) | 부서업무 **등록 가능**(Q&A 확정), 본인 담당 업무 상태/진행 갱신, 댓글, 부서 업무 조회 |
| 상위 부서장/임원 | 하위 부서업무 열람(계층 재귀 — `my_readable_dept_ids`) |
| admin | 전체 |

> 담당자 **지정/삭제**의 최종 권한은 부서장(=`my_editable_dept_ids`)으로 한정(부서원은 등록은 가능하나 타인 지정은 부서장 승인 범위) — 02 아키텍처에서 RLS로 확정.

## 5. 사용자 스토리 (MVP)

- US-1 부서장으로서, 부서 업무를 제목·설명·마감일과 함께 등록하고 담당자를 지정한다.
- US-2 부서원으로서, 내가 담당한 부서 업무의 상태(예정/진행/완료/보류)와 진행률을 갱신한다.
- US-3 담당자/부서원으로서, 업무에 **댓글**로 진행상황을 남기고 서로의 댓글을 본다.
- US-4 부서장으로서, 부서 업무 목록을 상태별로 보고 어디까지 진행됐는지 한눈에 본다.
- US-5 업무 생성 시 **선택적으로** 체크리스트를 달아 세부 항목을 관리한다.
- US-6 주간보고 작성 시, 진행 중인 부서 업무가 **자동 집계로 표시**되고 원하면 특정 업무를 **수동 인용**한다.

## 6. 범위 (MVP vs 후속) — 🟦 DC-BIZ 권고 반영

| 영역 | **MVP (v0.8.0)** | 후속(v0.9+) |
|------|------|------|
| 업무 객체 | 제목/설명/담당자/마감/상태/진행률(0–100) | 서브태스크, 업무 간 의존성 |
| 협업 | **댓글(스레드 재사용)** | 대댓글·멘션(@)·이모지 |
| 체크리스트 | 단순 체크리스트(선택, 생성 시) | 템플릿, 항목별 담당자 |
| 권한 | 기존 org-scope 재사용 | 외부 공유/게스트 |
| 주간보고 연동 | 자동 집계 표시 + 수동 인용(읽기 전용) | 진척률 그래프 |
| 첨부/알림 | ❌ (인앱 배지 정도) | Storage 첨부, 이메일/카톡 알림 |
| 뷰 | 리스트 + 상태 필터 | 칸반/간트 |

> **대댓글**: 사용자 요지에 있으나 🟦 DC-BIZ는 MVP에서 1-depth 댓글 권고. → 02 문서에서 "댓글 컨테이너는 대댓글 가능 구조로 두되 MVP UI는 1-depth"로 절충(데이터는 미래 대비, UI는 단순).

## 7. 비범위 (이번에 하지 않음)

- 모든 코드 구현·DB 마이그레이션 적용·커밋 (이 작업은 **기획서 산출까지**).
- 알림/멘션/첨부/실시간 협업/칸반.
- 일일업무·부서업무 화면 통합(분리 유지).

## 8. 성공 지표 (측정 가능 — DB 카운트로 산출)

1. 부서업무 페이지 부서장 WAU ≥ 등록 부서장 수의 70% (실제 본다)
2. 부서업무 등록 후 7일 내 상태 변경률 ≥ 60% (방치 업무 역지표)
3. 주간보고 작성 시 부서업무 연동 섹션 채택률 ≥ 50% (기존 워크플로 통합)

## 9. 핵심 결정 (사용자 확정 — 2026-06-08) ✅

- **D-1 데이터 표현 = ⓐ `daily_logs` 확장.** 신규 테이블 0, daily_logs를 substrate로 재사용(`task_kind`로 개인/부서 분리).
- **D-2 댓글 저장소 = 기존 `daily_log_threads` 확장.** 범용 polymorphic `comments`는 후속 SSOT 과제로 보류.
- **D-3 담당자 지정/변경 = 부서장만.** 부서원은 부서업무 *등록*은 가능하되 `assignee_user_id` 지정/변경은 부서장(`my_editable_dept_ids`)·admin만. 부서원 등록 시 담당자는 비워두거나 본인으로만 설정 가능.

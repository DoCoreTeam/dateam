# 00 · 요구사항 — 주간보고 작성분 영구 유실 0 (Zero-Loss)

> 상태: **기획(구현 전)**. 대상 버전 v0.7.287. 작성일 2026-07-06.
> 트리거: 이도현 주간보고(2026-06-29주) "또 날아갔다" 신고 → 포렌식 결과 구조적 유실 위험 확정.

## 1. 최상위 원칙 (불변)

- **P0 — 사용자가 작성한 데이터는 어떤 경우에도 영구 유실되지 않는다.** 로직 버그·경쟁 저장·잘못된 동기화가 발생하더라도, 작성분은 **항상 복원 가능**해야 한다.
- **P1 — 편의(AI 초안·자동 반영 등)는 P0를 절대 침해하지 않는 선에서만 제공한다.** 편의 기능이 확정본을 덮어 유실을 유발하면 그 기능이 틀린 것이다.

## 3. 사용자 확정 결정 (Q&A)

1. **안전망 = 사용자 버전 히스토리.** 저장 직전 전체 확정본 스냅샷 + 사용자 화면에서 이전 버전 조회·복원.
2. **저장 경로 = 단일 Writer.** 확정본에 쓰는 유일 경로는 "사용자 폼 저장". AI초안·메모·일일반영은 화면 폼 state에만 반영 → 사용자 검토 후 저장. draft PUT의 확정본 자동 sync 제거.
3. **현재 06-29 갈라짐 즉시 안전조치**를 본 기획에 포함.

## 4. 기능 요구사항

### FR-1 단일 Writer (원인 ② 제거)
- `weekly_reports`에 INSERT/DELETE/UPDATE 하는 앱 경로는 **오직 `upsertWeeklyReport`(+명시적 삭제 액션)** 로 한정한다.
- `draft/route.ts` PUT은 `weekly_report_items`만 저장하고 `replace_weekly_report` 호출을 **제거**한다.
- AI 초안/메모/일일 반영 UX는 "폼에 반영(client state)" → 사용자 저장으로 통일한다(기존 UX 흐름 유지, 백그라운드 확정본 쓰기만 차단).

### FR-2 저장 전 스냅샷 + 사용자 복원 (P0 보루)
- 신규 append-only 테이블 `weekly_report_snapshots`에 **매 확정본 저장 직전 전체 상태**를 보관한다.
- 스냅샷은 확정본 저장 RPC **동일 트랜잭션** 내에서 DELETE 전에 적재(원자성).
- 주간보고 화면에 주차별 **"편집 이력"** 패널: 스냅샷 목록(시각·항목수·사유) + **[복원]** (해당 시점 rows를 폼으로 로드 또는 확정본 복원).
- 복원 자체도 하나의 저장 → 복원 직전 상태가 또 스냅샷됨(복원의 되돌리기 보장).

### FR-3 감사로깅·부서·content_hash 복원 (원인 ③ 되돌림)
- `replace_weekly_report`에 마이그 120의 `weekly_report_activity`(create/edit) 기록 + `department_id` 세팅을 **복원**하고, 141의 `seq` 부여를 **유지**한다(120+141 병합).
- `content_hash`를 채워 저장 간 실질 변경을 식별한다.

### FR-4 즉시 안전조치 (현재 06-29 및 동일상황)
- 배포로 draft 확정본 sync가 제거되면 06-29 덮어쓰기 경로가 원천 소멸.
- 배포 시 **현재 확정본 1건을 스냅샷 시딩**하여 초기 안전망을 확보한다(무손상, 추가만).

## 5. 비기능 요구사항
- **데이터 무손상 마이그레이션**: 신규 테이블 추가 + `CREATE OR REPLACE FUNCTION`만. 기존 행 갱신/삭제 없음. 상태플래그 덮어쓰기 금지(프로젝트 원칙).
- **RLS 필수**: 스냅샷 SELECT=본인, INSERT=본인(RPC invoker), UPDATE/DELETE 정책 없음(append-only 불변).
- **성능**: 스냅샷은 저장당 1행(jsonb). 주차별 조회는 인덱스(user_id, week_start, taken_at DESC).
- **KST**: 표시 시각은 `lib/datetime/kst.ts` 경유(taken_at은 UTC 저장).
- **디자인 토큰/공용 컴포넌트**: 편집이력 패널·복원 모달은 토큰+공용 컴포넌트(NbButton/모달 표준) 준수.

## 6. 완료 정의 (개요, 상세는 04)
사용자가 어떤 순서로 저장/AI반영/복원을 해도 **작성분이 영구 소실되지 않으며**, 소실 정황 발생 시 사용자가 스스로 이전 버전을 복원할 수 있다.

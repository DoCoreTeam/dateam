# 03 · 테스트 전략

> v0.7.211 · 2026-06-20 · 기획 확정(구현 전)
> 러너: node:test (apps/web). 신규 *.test.ts는 apps/web/package.json `test` 목록에 **수동 추가** 필요.

## 1. 단위 테스트 — `lib/admin/daily-monitoring.ts`

감사·평가 신뢰성이 걸린 핵심. 데이터 정확성 최우선.

| 테스트 | 검증 |
|--------|------|
| 월 집계 — 날짜별 작성인원수 | 같은 user가 하루 여러 건 써도 **distinct user 1명**으로 카운트 |
| 미작성자 차집합 | 활성 멤버 − 작성자 = 정확한 미작성 명단. 퇴사/비활성 제외 |
| is_onboarding 제외 | 실습행이 집계·리스트·카운트 어디에도 안 섞임 (NFR-5) |
| task_kind 구분 | personal/dept_task 필터가 정확히 분리 |
| "수정됨" 판정 | created_at≠updated_at → 수정됨 true, 같으면 false (NFR-2/3) |
| 시각 포맷 | logged_at → KST 표시 일관, DST 무관 (NFR-4) |
| 카운트=리스트 일치 | 요약 KPI의 작성 N명 == 리스트 distinct user 수 (NFR-1) |

## 2. 통합 테스트 — 쿼리 경계

| 테스트 | 검증 |
|--------|------|
| 날짜 없는 달(작성 0건) | 빈 캘린더·빈 리스트, 에러 없음 |
| 2000건 초과 월 | 페이지네이션으로 누락 없이 전수 접근 (LIMIT 2000 회귀 방지) |
| 검색 ilike | 특수문자/한글 키워드 sanitize, injection 불가 |
| 정렬 화이트리스트 | 허용 외 sort 파라미터 → 기본값 폴백(임의 컬럼 정렬 차단) |
| 권한 | 비admin 접근 → /dashboard 리다이렉트 |

## 3. E2E (Playwright) — 실제 렌더 경로 기준

> CLAUDE.md "실제 렌더 경로에서 검증" 정책. throwaway admin 계정 + is_test 데이터로 격리.

| 시나리오 | 단계 |
|----------|------|
| 캘린더 → 날짜 클릭 → 리스트 | 셀 인원수 보임 → 클릭 → 하단 작성자 리스트 + 미작성자 노출 |
| 작성일시 표시 | 리스트 첫 컬럼에 날짜·시각, 수정 로그엔 "수정됨" 뱃지 |
| 검색/정렬/필터 | 키워드 입력 시 행 축소, 헤더 클릭 시 정렬, URL 반영 |
| 반응형 | 320/768/1024 — 캘린더·리스트 카드 변환, 가로스크롤 없음 |

## 4. 비회귀 — 감사 신뢰성 가드

- **카운트 불일치 금지**: 요약 숫자와 실제 리스트 길이가 다르면 FAIL (감사 근거 신뢰 깨짐)
- **행 은닉 금지**: 필터 OFF 상태에서 그날 모든 작성 로그가 100% 보여야 함
- 실데이터 오염 금지: 운영 daily_logs로 테스트하지 않음 (메모리 정책)

## 5. 커버리지 목표

- lib 집계/포맷 함수 90%+ (감사 핵심)
- 화면 컴포넌트는 E2E·시각검증으로 보완 (브리틀 마크업 단언 지양)

# 00 · 요구사항 — 관리자 일일업무 모니터링 재설계

> 버전 기준: v0.7.211 · 작성일: 2026-06-20 · 상태: **기획 확정(구현 전)**
> 사용자 지시: **"절대구현하지마 / 기획해서 보고"** — 본 docs는 승인 시 구현 입력물

## 1. 배경 — 현재 무엇이 문제인가

현 `/admin/daily-logs`(서버 컴포넌트 단일 파일 `app/admin/daily-logs/page.tsx`)는
**"한 사람의 하루"를 보는 단건 조회**다. 관리자가 원하는 "조직 전체 작성 현황을
기간 단위로 모니터링"하는 도구가 아니다.

확정된 한계 (DC-ANA 진단, 파일·라인 근거):

| # | 문제 | 코드 근거 |
|---|------|----------|
| 1 | 단건 날짜만 조회 | `page.tsx` L65 `.eq('log_date', selectedDate)` |
| 2 | 작성일/시각 표시 미흡 (멤버 카드 그룹만) | L63-69 |
| 3 | 내용 검색 없음 | — |
| 4 | 정렬 고정(user_id→logged_at) | L67-68 |
| 5 | 페이지네이션 없음, 하드 LIMIT 2000 | L60 |
| 6 | "전체 탐색" 개념 부재 (날짜 없으면 공백) | L65 |
| 7 | task_kind(개인/부서) 혼재 표시 | L63-69 |
| 8 | 디자인 이탈(인라인 style, input-field 미적용) | L245-248 |

## 2. 사용자 요구 (대화 누적)

1. **달력에 작성된 날짜 표시** — 캘린더 셀에 그날 작성 여부/인원 표시
2. **몇 명이 썼는지 리스트 확인** — 날짜 클릭 시 작성자 명단
3. **전체 탐색 / 검색 / 정렬이 명확** — 하루씩 넘기지 않아도 됨
4. **작성일자 표시** — 언제 작성했는지 명확
5. **미래 용도 = 감사로그(audit) · 인사평가(evaluation)** → 데이터 신뢰성 필수
6. **"의도와 상관없이 명확하게 잘 보이면 된다"** — 가독성·명료성 최우선

## 3. 기능 요구사항 (FR)

| ID | 요구 | 우선순위 |
|----|------|---------|
| FR-1 | 월간 캘린더 뷰 — 각 날짜 셀에 **작성 인원수** 뱃지 | P0 |
| FR-2 | 셀에 **블로커 발생일** 표시(▲) | P1 |
| FR-3 | 셀 색 농도로 **미작성 많은 날** 시각화 | P1 |
| FR-4 | 날짜 클릭 → 하단에 **그날 작성자 리스트**(작성일시·부서·타입·내용) | P0 |
| FR-5 | 같은 화면에서 **미작성자 명단** 노출 | P0 |
| FR-6 | 리스트 **내용·이름 검색** | P1 |
| FR-7 | 리스트 **정렬**(작성일시/멤버/부서/타입, 헤더 토글) | P1 |
| FR-8 | 리스트 **필터**(부서/타입/task_kind/블로커만) | P1 |
| FR-9 | 리스트 **서버 페이지네이션**(LIMIT 2000 폐기) | P1 |
| FR-10 | **작성 vs 수정 구분 표시**(created_at≠updated_at → "수정됨") | P0(감사) |
| FR-11 | 상태(검색/정렬/필터/선택일) **URL 동기화** | P1 |
| FR-12 | (Phase3) **CSV 내보내기**(감사·평가 근거 보존) | P2 |

## 4. 비기능 요구사항 (NFR) — 감사·평가 대응

| ID | 요구 | 근거 |
|----|------|------|
| NFR-1 | **완전성**: 조회 결과에서 행을 임의로 숨기지 않음. 카운트와 리스트가 항상 일치 | 감사 신뢰 |
| NFR-2 | **시각 정확성**: 작성시각(`logged_at`/`created_at`)과 수정시각(`updated_at`) 구분 표기 | 평가 공정 |
| NFR-3 | **변조 인지**: 수정된 로그는 "수정됨" 표시로 사후 변경을 드러냄 | 감사 |
| NFR-4 | **시간대 일관**: 모든 시각 KST 기준 표시, 포맷 SSOT(Intl) | 일관성 |
| NFR-5 | **온보딩/실습행 제외**: `is_onboarding=true` 행은 집계·리스트에서 기본 제외 | 정확성 |
| NFR-6 | **명료성**: 10px 미만 폰트 금지, 금액 외 텍스트도 디자인 토큰 준수 | 사용자 명시 "잘 보이게" |
| NFR-7 | **성능**: 월 집계 1쿼리 + 선택일 리스트 1쿼리. 기존 인덱스 활용 | 반응성 |
| NFR-8 | **권한**: admin 전용. `requireAdmin` 패턴 통일 | 보안 |

## 5. 제외 범위 (이번 기획에서 다루지 않음)

- 로그 **수정 이력 전체 버전 보관**(감사 강화) — 현재 `daily_logs`는 updated_at만 보유.
  완전한 audit trail이 필요하면 별도 `daily_logs_audit` 테이블이 필요 → **Phase4 후보**, 본 기획 범위 밖.
- 평가 점수 산출 로직 — 본 화면은 "근거 데이터 표시"까지. 평가 알고리즘은 별도.
- 멤버 본인용 화면 변경 — admin 화면만 대상.

## 6. 감사·평가 용도에 대한 사실 고지 (과장 없음)

- 미작성자 명단 노출은 **사실 기반 운영 지표**다. 평가 활용은 조직 정책의 영역이며,
  본 화면은 데이터를 **정확·완전·명료**하게 보여주는 책임만 진다.
- 현재 스키마로는 "수정됨" 여부까지는 알 수 있으나 **무엇을 언제 어떻게 고쳤는지**까지는
  알 수 없다(버전 보관 부재). 감사 강도가 높아지면 NFR 상향(Phase4)이 별도 필요함을 명시.

## A. 근본 원인 — 2개 축, 6개 결함

### 축 1: "데이터가 일부만 추출됨" = 다층 상한(cap) + 표구조 조기 손실

| # | 결함 | 위치 | 값 | 영향 |
|---|------|------|-----|------|
| R1 | **URL 본문 15K자 절단** | `lib/gpu/extract-helpers.ts:55` | `.slice(0, 15000)` | 긴 가격표 하단 행이 AI 입력에 아예 안 들어감 |
| R2 | **HTML 태그 전체 제거 → 표 구조 파괴** | `extract-helpers.ts:51-54` | `replace(/<[^>]+>/g,' ')` | `<table>`이 word-soup가 됨 → 열정렬 소실 → "Contact us"·"—" 모호행 AI가 버림 |
| R3 | **추출 결과 50건 하드컷** | `app/api/pricing/gpu/review/stream/route.ts:164` | `.slice(0, 50)` | AI가 더 찾아도 50건에서 잘림(조용히) |
| R4 | **카탈로그(xlsx/csv) 레거시: 첫 시트만·평면표 가정** | `lib/gpu/catalog-parse.ts` (첫시트), 행 1,000컷 | `MAX_CATALOG_ROWS=1000`, 첫 시트 only | 다중 시트·다중 블록 가격표 후반 전부 누락 |
| R5 | **USAI(좌표보존 엔진) 적용 불완전** | `catalog/route.ts:170` 만 플래그 체크. `review/stream`은 USAI 미적용 | `GPU_USAI_INGEST` 기본 OFF | 통합입력의 이미지/PDF/URL은 항상 Vision/word-soup 경로 → 구조손실. USAI는 카탈로그 업로드에만, 그나마 기본 OFF |

> 주의(추정 배제): 분석 중 "MAX_INPUT_TEXT 200K와 URL본문이 경합", "Gemini 우측 절단"은 코드상 단정 불가.
> 실제 코드는 `contentText = rawInputText + merged`로 200K 재절단을 하지 않음(`stream/route.ts:100-101`).
> **확정된 지배 원인은 R1(15K)·R2(표파괴)·R3(50건)** 세 가지. R4·R5는 카탈로그/대용량에서 추가 가중.

### 축 2: "경쟁사인데 공급가로 분류됨" = 결정적 경로 부재

| # | 결함 | 위치 | 내용 |
|---|------|------|------|
| C1 | **분류 100% AI 추정, 사용자 의도 강제경로 없음** | `stream/route.ts:112-122` + DB `gpu.input-classify` | "경쟁사 등록"이라 써도 AI가 다시 판단. competitor 아니면 전부 supplier 폴백 |
| C2 | **채널 드롭다운(own/mail/…)이 분류와 무관** | `QuoteRegisterTab.tsx` 채널 select → `review/commit` `channel` 컬럼 | 저장 추적용일 뿐, 공급가/경쟁사 판정에 0 영향 |
| C3 | **이미지/PDF는 분류 단계 자체를 스킵 → 무조건 supplier** | `stream/route.ts:114` `if(!hasImages)` | 경쟁사 가격표 스크린샷도 100% 공급가로 저장 |
| C4 | **경쟁사 화이트리스트가 코드 상수가 아닌 DB 프롬프트 문구** | `review/route.ts:202` 프롬프트 내 나열 | RunPod/Lambda/AWS… 가 프롬프트 텍스트로만 존재 → 결정성·테스트 불가. Nebius는 목록에도 없음 |

**한 줄 핵심**
- 추출: *DoS/메모리 방어용 상한*들이 **정상 입력에도 무차별 적용**되고, **표 구조를 입력 단계에서 파괴**해 AI가 일부만 건짐.
- 분류: **사용자 의도를 강제하는 결정적 분기가 없고**, 이미지/PDF는 분류조차 안 함.

## B. 연관 이슈 (샘플 외 — 동일 뿌리에서 파생, 재현 가능)

1. **다중 URL일수록 더 잃음** — URL N개를 넣어도 각 URL이 15K로 잘려 병합(`stream/route.ts:99`) + 최종 50건컷. "많이 넣을수록 일부만".
2. **엑셀 다지역 가격표(예: 8지역)** — 레거시는 첫 시트만(R4), USAI ON이어도 1,500셀 압축컷(`grid-compress.ts:20`)으로 후반 블록 누락.
3. **경쟁사 스크린샷 업로드** — C3로 전량 공급가 오적재 → 시장비교(market_prices) 0건, 검토대기만 쌓임.
4. **신규 경쟁사(Nebius 등) 미인식** — C4 화이트리스트 부재로 목록에 없는 신생 클라우드는 supplier로 흐름.
5. **조용한 절단(silent truncation)** — R1·R3는 사용자에게 "N건 잘림" 고지가 없음 → "다 들어간 줄" 착각(가장 위험).

## C. 수정 설계 (구현은 승인 후 — 여기선 설계만)

설계 원칙: **상한은 "보안용 상한"과 "데이터 손실"을 분리** + **표 구조 보존** + **분류에 사용자 결정권 부여**. 전부 SSOT(`lib/gpu/`)로.

### Phase 1 — 추출 무손실화 (축1)
- **P1-1 표 구조 보존 파서 도입(SSOT)**: URL HTML은 태그 제거(R2) 대신 `<table>`을 행·열 보존 텍스트(파이프/탭 구분)로 변환. 경량 파서(cheerio/htmlparser2 후보, OSS 평가 필요) → `lib/gpu/html-table-extract.ts`.
- **P1-2 상한 재설계**: R1(15K)·R3(50건)을 *데이터 손실용*에서 *보안 상한*으로 승격 — 값 대폭 상향(예: 본문 200K+, 결과 무제한 또는 페이지드) + **초과 시 사용자 고지(잘림 배너)** 필수. 2MB(safe-fetch)는 보안상 유지하되 "표만 추출"로 효율↑.
- **P1-3 USAI를 통합입력 경로에도 연결**: `review/stream`의 이미지/PDF를 USAI 좌표보존 파이프(이미 구축됨, flag OFF)로 라우팅. 1,500셀 압축컷은 멀티패스(블록 단위 반복호출)로 무손실화 검토.
- **P1-4 silent truncation 제거**: 모든 cap 도달 시 SSE `progress`로 "N건/N자 잘림" 송출 + 미리보기 상단 경고.

### Phase 2 — 분류 결정성 부여 (축2)
- **P2-1 사용자 의도 override(결정적)**: 입력창에 "경쟁사/공급가" 명시 토글(또는 키워드 파싱)을 두고, 있으면 **AI 분류를 덮어씀**. 채널 드롭다운과 별개의 "분류" 컨트롤.
- **P2-2 provider 화이트리스트 코드 SSOT화**: `lib/gpu/provider-registry.ts`에 경쟁 클라우드 목록(Nebius·RunPod·Lambda·CoreWeave·AWS·GCP·Azure·NHN·NAVER…) 상수화 → 분류 1차 결정 + 프롬프트는 보조. 테스트 가능.
- **P2-3 이미지/PDF 분류 스킵 제거(C3)**: Vision 추출 후 동일 분류기/화이트리스트를 태움.
- **P2-4 미리보기 인라인 정정 UI(C5)**: 각 행에 공급가↔경쟁사 토글 → 확정 시 올바른 테이블로 라우팅.

### Phase 3 — 회귀 방어
- 기존 `review/route.ts`(POST) 무수정 보존 정책 유지하되, 신로직은 SSOT 모듈로 두고 stream·catalog가 import.

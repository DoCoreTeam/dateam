# 04 · 테스트 전략 & 완료 기준 & 리스크

## A. 테스트 전략
### 등록/CRUD
- 날짜 클릭 → 모달 → 저장 → 캘린더 즉시 반영(월/주). 편집·삭제 반영.
- 종일/시점/기간(start~end) 각각 올바른 셀 배치.
### 자연어
- "내일 3시 A사 미팅 1시간" → 시작=내일 15:00, 종료 16:00(또는 추정), 제목 추출. 모호 입력은 폼에서 보정.
- 날짜 없는 입력 → 시작 미설정으로 두고 사용자 입력 유도(빈 일정 저장 방지).
### 업무 연계
- 일정에 daily/weekly/memo 연결 → 상세에서 원본 점프. 원본 삭제 시 link 정리(또는 '연결끊김' 표시).
### AI 추천
- 규칙 후보(planned/이월/plan/memo) 정확 도출. Gemini 추천 근거(reason) 노출. [등록] 시 source='ai'+link 자동.
- 비용: 매 페이지 로드 자동호출 0(수동/주1회만).
### 권한 (조직 계층 — 기존 매트릭스 재사용)
| 주체 | 본인 일정 | 관할 팀원 일정 | 타 부서 |
|------|:--:|:--:|:--:|
| 팀원 | 쓰기 | ✕ | ✕ |
| 부서장 | 쓰기 | 조회(🔒) | ✕ |
| 상위/대표 | 쓰기 | 조회 | (대표=전사) |
- 플래그 OFF → 본인+admin만(기존 동작). RLS·API 양쪽 검증.
### 반복
- rrule 저장 → 조회 범위 내 인스턴스 전개 정확(주/월). 성능(월뷰 1쿼리+전개).
### 브라우저 E2E
- 등록(클릭+자연어), 연계 점프, 추천 등록, 계층 조회, 반응형 320/768/1440, 콘솔에러 0.

## C. 리스크
| 리스크 | 완화 |
|--------|------|
| polymorphic link(link_kind/link_id) 무결성 | 앱 레벨 검증 + 원본 삭제 시 정리 트리거(선택), 'none' 허용 |
| 반복 일정 복잡도(예외/수정 인스턴스) | 1차는 단순 rrule만, override는 2차. rrule.js 표준 사용 |
| AI 추천 비용/품질 | 규칙 1차 필터 → Gemini 1회, 수동 트리거, 근거 노출 |
| daily_logs와 이중 표시 혼란 | 색·아이콘 범례로 일정/업무 구분, 통합 렌더 |
| 계층 RLS 회귀 | 기존 헬퍼 재사용(신규 권한로직 0), 플래그 OFF=기존동작 보존 |
| 과설계(구글동기화·알림 선반영) | 범위 외로 명시, 컬럼 여지(external_id)만 |

## E. 핵심 메시지
- 캘린더 '등록 불가'는 **모델·API·UI 3중 부재** — 신규 `calendar_events`로 정공법.
- 핵심 차별점은 **"업무적 등록"**: 일정이 daily/weekly/memo와 링크되고, 그 데이터로 **AI가 다음 일정을 추천**.
- 권한·자연어·AI는 **이미 만든 자산(조직 계층 RLS, analyze-work, Gemini)을 재사용** → 신규 비용 최소.
- P1(등록 MVP)만으로 "캘린더에서 일정 등록" 즉시 해결, P2~P3에서 연계·AI로 확장.

# 디자인 변경 커버리지 진단 (구현 0 — 분석 전용)
> 2026-06-08 · "디자인 변경 시 모든 부분이 한 곳에서 통제되는가?" 정직 점검

## 한 줄 답
**색(color)은 99% 통제됨(토큰 1곳=전 화면). 그러나 "치수(보더 두께·모서리 radius·그림자 형태)"는 아직 약 850곳이 리터럴이라 한 곳 통제 불가.**
즉 "색 바꾸는 디자인 변경"은 완성, "두께·모서리·그림자 바꾸는 변경"은 미완성.

## 정량 측정
| 영역 | 상태 | 수치 |
|------|------|------|
| 색상 토큰화 | ✅ 통제됨 | var(--) 3,550 / 인라인 hex 잔여 13 (핑크 등 의도) = **99%** |
| 테마 전환 | ✅ 작동 | nb↔classic `[data-theme]` 4블록, SSR 주입 |
| **보더 두께** | ❌ 갭 | 인라인 `Npx solid` 리터럴 **577** + globals **89** = 666 (`--border-w` 미사용) |
| **모서리 radius** | ❌ 갭 | 인라인 `borderRadius:'Npx/rem'` 리터럴 **180** (`var(--radius)` 미사용) |
| rgba 하드코딩 | ⚠️ 부분 | hover/오버레이 등 **149** (토큰 아님) |
| 재발 방지(lint) | ❌ 없음 | eslint 인라인 hex/리터럴 차단 룰 0 |
| globals 모듈화 | ❌ 안 됨 | globals.css **3,780줄** 단일 평면 |
| status 색 SSOT | ⚠️ 고정 | status-colors.ts는 hex(테마 공통, 의도적) |

## 의미 (무엇이 통제되고 무엇이 새는가)
### ✅ 지금 1곳에서 통제되는 변경
- 브랜드색/텍스트/배경/보더"색"/상태색 변경 → 토큰 1줄
- 테마 추가/전환 → globals 블록 1개 + 레지스트리 1줄
- 사이드바 색 → `--sidebar-*`

### ❌ 아직 여러 곳 만져야 하는 변경
- **보더를 더 두껍게/얇게** → 666곳 리터럴(`2px/3px solid`)이 `--border-w` 무시 → classic 테마에서도 두께 안 줄어듦
- **모서리를 더 둥글게/각지게** → 180곳 `borderRadius` 리터럴이 `--radius` 무시
- **그림자 스타일 일괄 변경** → 인라인 일부 리터럴/rgba
- **새 색 재유입 차단** → lint 없어 누구나 하드코딩 가능

## 갭이 생긴 이유
토큰화 작업이 **색 중심**이었음(hex→토큰). 보더 두께·radius·그림자 오프셋 같은 **치수 리터럴**은 codemod 범위 밖이라 그대로. 색은 통일됐지만 "형태 토큰"은 부분만 적용.

## 권고 (구현 시 — 별도 승인)
1. **치수 토큰화 2차 codemod**: `Npx solid`→`var(--border-w) solid`, `borderRadius:'Npx'`→`var(--radius)` (단 의도적 예외: 원형 50%, 특수 두께 제외). 단일 실행 명세 동일 패턴.
2. **ESLint 가드**(Phase 1): 인라인 hex/치수 리터럴 신규 차단(warn→error) → 재발 방지.
3. **globals.css 모듈화**(Phase 4): gpu/calendar/daily CSS 분리 → 3,780→~700줄.
4. (선택) rgba 하드코딩 토큰화, status 색 테마화 여부 결정.

## 결론
디자인 **색** 변경은 "모든 부분 파악·통제" 됨. 디자인 **형태(두께·모서리·그림자)** 변경은 **아직 50% 수준** — 치수 토큰화 2차 + lint를 하면 "디자인 변경 = 토큰 몇 줄"이 완성됨.

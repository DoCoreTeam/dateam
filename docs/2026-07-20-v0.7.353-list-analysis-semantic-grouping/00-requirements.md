## 1. 배경 — 왜 세 번 실패했는가

이 기능은 세 세대에 걸쳐 재설계됐고, 세 번 모두 같은 이유로 실패했다.

| 세대 | 최상위 계약 (문서 원문) | 결과 |
|---|---|---|
| v1 (v0.7.304) | "추출 시 항목을 생략·축소·요약·병합·절단하지 않는다" | 무의미 파편 대량 |
| v2 (v0.7.32x) | "유실0/생략0 — 커버리지 게이트가 최종 판정" | 무손실을 더 강하게 기계화 |
| v3 (미병합 브랜치) | "무손실은 목표가 아니라 부작용이었다" | 방향은 옳으나 평탄화 추출은 유지한 채 템플릿만 증축 |

세 세대 모두 이전 계약을 **폐기하지 않고 그 위에 쌓았다.**

### 1-1. 실패의 진짜 원인 — 발화의 오역

사용자가 "생략하지 마라"고 요구한 것의 실제 의미는 **신뢰의 문제**였다:

> "내가 중요하다고 생각하는 것을 AI가 멋대로 빼먹지 마라"

그러나 시스템은 이를 **구현 계약**으로 번역했다:

> "마커(`-`, `1.`, `①`)로 시작하는 모든 줄을 항목으로 보존하라"

이 번역 오류가 세 세대에 걸쳐 승계됐다. 세 번의 재설계가 모두 "어떻게 하면 더 완벽하게 무손실할 수 있는가"를 풀었고, 번역 자체를 의심하지 않았다.

### 1-2. 실측 증상 (v0.7.352, localhost 재현)

기획서 1건 입력 → **141개 항목** 추출.

```
141 = AI 추출 129개 + 결정론 파서 결과 중 글자 완전일치 실패 12개
```

추출된 항목에 `문서 버전: v0.1.0`, `작성일: 2026-07-20`, `상태: 초안(Draft)` 등 문서 메타데이터가 포함됨.

원인 (코드 확정):
- `lib/ai-chat/list-extract.ts:16-46` — 판정 기준이 "줄이 마커로 시작하는가" 단 하나. 내용 의미 미판정.
- `analyze/actions.ts:111-118` — 추출 프롬프트가 *"하나도 빠짐없이 … 애매하면 누락시키지 말고 포함한다(누락보다 중복이 낫다)"*. 의미 조건 0건.
- `lib/ai-chat/list-extract.ts:2-3` — 병합 계약 *"AI 보정은 '추가'만 하고 '축소'하지 않는다"*. 축소 경로 부재.
- **필터링·제외 로직은 전 파이프라인에 존재하지 않음.**

### 1-3. 가장 치명적인 결함 — 지시가 추출에 도달하지 못한다

| 확인 항목 | 실측 |
|---|---|
| 추출 요청 payload | `text` 또는 `file` **뿐**. command 미포함 (`AnalyzeClient.tsx:70-74`) |
| 서버 `extractItems` | `text`/`file`만 읽음. 프롬프트는 상수 고정 (`actions.ts:184-185`) |
| `command` 실사용처 | `analyze-core.ts:47-56` — **항목별 분석 단계 전용** |
| 지시 입력 UI 렌더 시점 | `step === 'review'` — **이미 쪼갠 뒤** (`AnalyzeClient.tsx:233-235`) |

즉 사용자가 무엇을 지시하든 **항목 나누기 방식에는 아무 영향이 없다.** 이는 버그가 아니라 "무손실 결정론 추출" 계약의 필연적 귀결이다 — 지시가 개입하면 계약 위반이 되기 때문.

### 1-4. 병행 장애 — 마이그레이션 161 미적용

`161_ai_analysis_v2.sql`이 실DB에 미적용. 161만 건너뛴 채 162~165(GPU 작업분)가 적용됨.

- 저장 실패: `session-persist-actions.ts:48` → `command` 컬럼 부재
- 목록 실패: `session-list-actions.ts:93` → `phase`, `synth_status` 컬럼 부재
- `ai_analysis_sessions` 실제 행수 **0** — 세션 저장 성공 이력 없음
- 두 지점 모두 Postgres 에러 원문을 로그 없이 폐기 (`:52`, `:111`) → 원인 은폐

이 건은 재정의 방향과 무관하게 선행 처리해야 한다 (Phase 0).

## 2. Job-to-be-done

> **"AI(또는 사람)가 만든 1차 산출물을 믿을 수 없다. 뭉개지고 생략된 부분을 찾아내서, 내가 지시한 형태의 더 완전하고 깊은 2차 문서로 만들어 달라. 단, 중요한 것을 빼먹었는지 내가 확인할 수 있게 하라."**

핵심 전제: **입력 문서는 이미 한 번 고민해서 작성된 산출물이다.** 따라서 구조는 이미 문서 안에 존재하며, 시스템의 일은 구조를 발명하는 것이 아니라 **복원하는 것**이다.

## 3. 계약 교체 (이 문서의 핵심)

### 계약 A — 단위

| | |
|---|---|
| (구) | 마커로 시작하는 모든 **줄**이 항목이다 |
| **(신)** | **원문의 의미 블록(그룹)이 단위다.** 그룹 = 제목 + 그에 속한 하위 내용 전체 |

예시 — 요구사항정의서:
```
(구) 항목 #37: "소셜 로그인 지원"
     항목 #38: "2FA 옵션"
     항목 #39: "세션 만료 30분"

(신) 그룹 #1: "요구사항 1. 사용자 인증"
       ├ 소셜 로그인 지원
       ├ 2FA 옵션
       └ 세션 만료 30분
```

### 계약 B — 수량

| | |
|---|---|
| (구) | 유실 방지를 위해 최대한 많이 |
| (오답) | 상한 30개 등 고정 캡 ← **채택하지 않음** (잘림 사고 유발) |
| **(신)** | **수량은 파라미터가 아니라 결과값이다.** 문서 구조와 지시가 결정한다. 상한 없음 |

### 계약 C — 무손실

| | |
|---|---|
| (구) | 모든 줄을 독립 항목으로 보존 → 파편화 |
| **(신)** | **묶는 것이므로 유실이 원천적으로 발생하지 않는다.** 모든 원문 줄은 어느 그룹엔가 소속되거나 문서 메타로 분류된다. 소속처 없는 줄 = 0 |

**중요**: 무손실 요구는 폐기되지 않는다. 충돌의 원인은 "무손실"이 아니라 **"평탄화(flatten)"**였다. 그룹핑은 무손실을 유지하면서 파편화만 제거한다.

### 계약 D — 컨트롤

| | |
|---|---|
| (구) | lens 칩 5개(요약/리스크/실행계획/근거/비교) + 별도 UI 컨트롤 |
| **(신)** | **자유 지시 한 곳이 전 단계를 지배한다.** 유형 판정·그룹 절단 레벨·재가공 형태 모두 지시로 제어. 별도 슬라이더/레벨 선택기 신설 금지 |

### 계약 E — 1급 객체

| | |
|---|---|
| (구) | 세션이 1급 관리 대상 (목록·검색·필터·휴지통) |
| **(신)** | **결과 문서가 1급 객체.** 실행은 문서의 상태다. 세션 = 문서당 1개 (그룹이 세션이 되지 않는다) |

## 4. 기능 요구사항 (FR)

### FR-1. 문서 유형 판정
전체 원문을 읽고 문서 유형을 판정한다 (요구사항정의서 / 회의록 / 리스크 레지스터 / 기획서 / 로드맵 / 기타). 사용자 지시에 유형이 명시되면 지시가 우선한다.

### FR-2. 구조 트리 복원
원문이 이미 가진 계층을 복원한다. 신호: 마크다운 헤딩 레벨, 번호 체계(`1.` → `1.1` → `-`), 들여쓰기 깊이, 표 구조, 빈 줄 구분.

### FR-3. 지시 지배 그룹 절단
문서 유형과 사용자 지시에 따라 **트리의 어느 레벨에서 자를지** 결정한다.

| 유형 | 기본 절단 레벨 |
|---|---|
| 요구사항정의서 | 요구사항 1건 |
| 회의록 | 안건 1건 |
| 리스크 레지스터 | 리스크 1건 |
| 기획서/로드맵 | 기능·단계 블록 |

지시로 재정의 가능: `"카테고리 단위로 크게 묶어"`, `"세부 요구사항까지 쪼개"`, `"P2 부분만"`.

### FR-4. 그룹 구성
각 그룹은 다음을 보유한다:
- 제목 (원문 헤딩 또는 AI 생성 요약 제목)
- **소속 원문 전체 (슬라이스 그대로, 재작성 금지)**
- 문서 내 위치 (경로/순서)
- 계층 레벨

### FR-5. 문서 메타 분리
버전·작성일·상태·문서명·작성자 등은 **그룹이 아니라 문서 속성(`doc_meta`)으로 분리 보관**한다. 삭제하지 않는다 (유실 0 유지).

### FR-6. 유실 0 검증
원문의 모든 줄이 "어느 그룹" 또는 "문서 메타" 또는 "미분류"에 귀속됨을 **결정론 코드로 검증**한다. 미분류가 존재하면 UI에 명시 노출한다. AI 자기보고에 의존하지 않는다.

### FR-7. 재그룹핑 (재지시 루프)
그룹 결과를 보고 재지시하면 **원문은 그대로 두고 그룹 구조만 다시 잡는다.** 새 리비전으로 기록하며 이전 리비전은 보존한다. 심화 실행 전이므로 비용이 낮다.

### FR-8. 그룹별 재가공
확정된 그룹 각각에 대해 지시를 적용한다 (심화 / 요약 / 다른 양식으로 변환). 입력에는 **문서 유형 + 전체 맥락 요약 + 해당 그룹 원문 전체**가 포함된다.

### FR-9. 넓이 패스 (빠진 것 발굴) — 승계
원문이 언급하지 않았으나 해당 문서 유형이라면 있어야 할 항목을 후보로 제시한다. **자동 반영 금지 — 사용자 확인 후 추가.** (미병합 브랜치 `breadth-pass.ts` 자산 승계)

### FR-10. 완성 문서 조립
그룹별 결과를 결정론 코드로 조립한다. 전 그룹 누락 0 검증, 미결 질문 섹션 포함. (미병합 브랜치 `assemble-template.ts` 자산 승계)

### FR-11. 산출물 목적지 (4경로 전부)
1. 파일 내보내기 (md/txt/docx/pdf) — 기존 유지
2. **앱 내 문서 라이브러리** — 결과 문서가 1급 객체로 저장, 재열람·재가공
3. **업무 흐름 연계** — 주간보고·부서업무·프로젝트·회의노트로 전달
4. **AI 채팅으로 이어가기** — 결과를 채팅 컨텍스트로 승계

### FR-12. 실행 전 계약 표시
실행 버튼 근처에 항상 표시: `N개 그룹 · 예상 호출 M회 · 예상 소요 시간`.

### FR-13. 진행·실패 가시화
429 등 오류를 사용자 언어로 노출한다 (예: "AI 사용량 한도로 대기 중, 자동 재개됩니다"). 에러 원문은 서버 로그에 반드시 남긴다.

### ⚠️ 마이그레이션 번호 충돌 — 조용한 사고 예약

`migrate.sh`는 번호 prefix로 적용 여부를 추적한다. 브랜치의 `163_ai_analysis_templates.sql`을 그대로 병합하면 **"163 이미 적용됨"으로 판정해 에러 없이 건너뛴다.** 161 사고(에러라도 났음)보다 더 위험하다.

→ **166번 이후로 재할당한다.** 별건으로 전체 마이그 적용 감사가 필요하다 (동일 함정 3회째).

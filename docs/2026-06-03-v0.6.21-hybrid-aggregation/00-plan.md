# 부서 취합 — 하이브리드 정렬/병합 규칙 기획

> 상태: **기획 전용 — 구현 금지** · 2026-06-03
> 배경: 현재 취합은 "카테고리 병합 + 작성자 제거"라 같은 카테고리 안에서 A·B 항목이 AI 임의로 섞임(A1·B1·A2·A3·B2). 입력 정렬도 없음.

## 1. 사용자 요구 (정제)
- **기본은 작성자별 순서** — A가 쓴 것 → B가 쓴 것 순.
- **단, 카테고리가 같으면 한데 맞춰서(같은 섹션으로) 정렬**.
- 즉 **하이브리드**: 카테고리로 묶되, 같은 카테고리 안에서는 작성자 순서를 보존.

## 2. 핵심 규칙 (HYBRID RULE)
> **정렬 축 = 카테고리(섹션) → 그 안에서 작성자 순서.**
> 같은 카테고리는 한 섹션으로 모으고(요구: "카테고리가 같다면 맞춰서"), 섹션 내부는 **작성자 블록 단위로 순서 유지**(요구: "작성자별로"). AI가 작성자 경계를 넘나들며 섞지 않는다.

```
[카테고리: 평택강소형]
  ├ (A 작성자 블록)  A의 항목 1, A의 항목 2, A의 항목 3
  └ (B 작성자 블록)  B의 항목 1, B의 항목 2
[카테고리: 광주지역거점]
  ├ (A 블록) …
  └ (C 블록) …
```
- **현재(섞임)**: 평택강소형 → A1·B1·A2·A3·B2 (작성자 경계 깨짐)
- **하이브리드(목표)**: 평택강소형 → [A: 1·2·3] 다음 [B: 1·2] (작성자 블록 보존, 같은 카테고리는 한 섹션)

## 3. 정렬 결정 규칙 (확정 필요 항목 = CEO 기본안)
| 축 | 규칙 | 기본안 |
|----|------|--------|
| 카테고리 순서 | 섹션 자체의 나열 순서 | **첫 등장 작성자 순서**(= 작성자 정렬을 따름). 즉 A가 처음 쓴 카테고리가 먼저 |
| 작성자 순서 | 섹션 내·카테고리 등장 모두의 기준 | **조직도 순서(`display_order`) → 이름** (안정적·결정적). 폼 제출 시각 아님 |
| 같은 작성자·같은 카테고리 내 항목 | 작성자가 쓴 원본 순서 유지 | 입력 bullet 순서 보존 |
| 중복 | A·B가 동일/유사 항목 작성 시 | **완전 동일만 1개로 병합**(작성자 블록 경계는 유지, 유사는 각자 유지) |
| 작성자 표기 | 섹션 내 작성자 블록 구분 | 옵션: 작성자명 라벨 표시 / 미표시(블록 구분만). **기본=라벨 표시**(누가 썼는지 보이게) |

## 4. 구현 변경점 (설계 — 적용 X)
### 4-1. 입력 정렬 (결정성 확보) — `aggregate-stream/route.ts`
```
weekly_reports 조회 시 ORDER BY 추가:
  작성자 순서(org display_order/name) → category → bullet 원본순
→ AI에 "이미 정렬된" 입력을 넘겨 비결정성 제거
```
입력 구성 시 각 행에 `authorOrder`(정수) 부여해 AI가 순서를 알 수 있게.

### 4-2. 프롬프트 규칙 교체 — `MERGE_BY_CATEGORY_PROMPT`
현재 2단계 "작성자 이름 제거 + 모두 나열"을 **하이브리드 지시**로 교체:
```
- 카테고리(섹션)는 의미상 같은 것끼리 통합(기존 1단계 유지)
- 단, 각 카테고리 내부는 입력의 작성자 순서를 보존하여
  [작성자 A의 항목들] → [작성자 B의 항목들] 순으로 나열
- 작성자 경계를 넘어 항목을 재배열·교차 정렬 금지
- 완전히 동일한 항목만 1개로 병합(유사 항목은 각 작성자 것 유지)
- (옵션) 각 작성자 블록 앞에 작성자명 표기
- 출력은 카테고리별 1행, 단 내부 bullet 순서는 작성자 블록 단위
```
출력 스키마는 유지(`{category, performance, plan, issues}`)하되, 각 필드 HTML 안에서 **작성자 블록 순서가 보존**되도록.

### 4-3. (선택) 출력 스키마 확장안
작성자 블록을 데이터로 명확히 하려면:
```
{category, authors: [{name, performance, plan, issues}], ...}
```
→ 화면에서 작성자별 소그룹 렌더 가능. 단 기존 table(카테고리 1행) 구조 변경 필요 → 범위 큼. **1차는 4-2(프롬프트+정렬)만으로 "블록 보존" 달성 권장.**

## 5. 엣지 케이스
| 케이스 | 처리 |
|--------|------|
| 한 작성자가 같은 카테고리에 여러 항목 | 원본 순서 유지 (한 블록) |
| A·B가 완전 동일 항목 | 1개로 병합(어느 블록에 둘지: 먼저 등장한 작성자 블록) |
| A·B가 유사하지만 다른 항목 | 병합 안 함, 각 블록에 유지 |
| 카테고리명이 의미상 같음(오타·약어) | 1단계에서 한 카테고리로 통합 후, 내부는 작성자 순 |
| 작성자 1명뿐 | 그대로(블록 1개) |
| 빈 필드 | 빈 문자열 |

## 6. 결정 확정 (사용자 승인)
1. **작성자 순서 기준 = 직급 우선 → 그다음 이름** ✅
   - 정렬: `profiles.rank`(직급)의 서열 → 동일 직급 시 `name` 가나다.
   - 직급 서열은 자유 텍스트가 아니라 **`org_ranks.display_order` 마스터**로 매핑(회장→…→연구원). 미등록 직급은 맨 뒤.
   - 정렬 SQL(개념): `weekly_reports JOIN profiles JOIN org_ranks ON profiles.rank=org_ranks.name ORDER BY org_ranks.display_order NULLS LAST, profiles.name, category, 원본순`
2. **작성자명 표기 = 표시** ✅ (확장 스키마에서 작성자 소그룹 헤더로 노출)
3. **범위 = 스키마 확장(4-3)** ✅ — 작성자 소그룹을 데이터로 명시, UI에서 작성자별 블록 렌더.

### 확정 스키마 (4-3 채택)
```
{
  category: string,
  authors: [
    { name: string, rank: string, performance: string, plan: string, issues: string }
  ]
}
```
- authors 배열 순서 = **직급 서열 → 이름** (정렬 보존, AI가 재배열 금지).
- 화면: 카테고리 섹션 안에 작성자별 소블록(작성자명·직급 라벨 + 성과/계획/이슈). 완전 동일 항목만 병합(먼저 등장 작성자 블록에).
- 기존 `{category,performance,plan,issues}` 단일 행 → authors 배열로 변경 = **dept_weekly_reports.body 구조 변경**(기존 취합본은 마이그레이션/재취합 필요 — 엣지로 관리).

## 7. 권장
- **1차**: 4-1(쿼리 정렬) + 4-2(프롬프트 하이브리드 규칙)만 — 기존 화면/스키마 그대로, "작성자 블록 보존 + 같은 카테고리 한 섹션" 달성. 회귀 최소.
- **2차(선택)**: 4-3 스키마 확장으로 작성자별 소그룹을 명시적 UI로.
- 핵심: **"섞임"은 프롬프트가 작성자 경계를 무시하기 때문 → 프롬프트에 '작성자 블록 보존' 규칙 + 입력 정렬**이면 해결. 화면 구조 변경 없이 가능.

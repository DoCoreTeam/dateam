# 00-requirements — GPU 모델 그룹핑 + 완전중복 병합 (v0.7.377)

## 발단
사용자: "H100이 다 다른거야? 종류만 4종류인데 fable5로 최신 데이터 확인해서 병합이 필요하면 병합진행. 다른 모델들도 확인해봐"

## 조사 결과 (DC-ANA, 실DB 조회)
GPU 관리 목록(`SpecsTab`, SSOT `/api/pricing/gpu/specs`)이 `model_name` **원문 문자열**로 그룹핑 →
`H100`·`H100 SXM`·`H100 PCIe`·`H100 NVL` 4개 문자열이 4개 카드로 뜸 = 사용자가 본 "4종류".

- **SXM/PCIe/NVL은 실존 SKU** (최신 NVIDIA 기준: SXM5 80GB HBM3 / PCIe 80GB HBM2e / NVL 94GB 페어) → 병합 금지.
- **문제의 본질**: 폼팩터 없는 "맨 H100"(레거시 gcube 시드, ×1/2/4/8 사다리·일부 확정가 보유)과
  폼팩터 SKU(신규 intake, ×1만)가 **같은 모델인데 별도 카드**로 분리 → 혼란.
- **별개로 진짜 완전중복 29그룹** 존재: model+form_factor+memory+gpu_count가 완전 동일한데 2~3행
  (주로 소비자 RTX 20/30/40/50 + T4). 참조 경미(확정가 대부분 null).

## 사용자 결정 (AskUserQuestion)
- **A**: 완전중복 29그룹 → **병합 실행(승인)**. 생존행 1개로 참조 재연결 후 나머지 소프트삭제.
- **B**: "폼팩터 없는 방식이 우리 스타일. 폼팩터 있는 것(SXM/PCIe/NVL)은 없는 것(H100) **아래로 그룹핑**되고,
  그 녀석들도 **×1/2/4/8 수량별 계산 시스템**을 동일하게 탄다."

## 요구사항
- R1. GPU 관리에서 `H100`은 **1종**으로 뜨고, 하위에 폼팩터(generic/SXM/PCIe/NVL)가 전개된다.
- R2. 각 폼팩터가 **×1/2/4/8 수량 사다리**를 동일하게 가진다. (수량은 v0.7.240 정책대로 **표시계층 파생** — 새 DB행 금지)
- R3. 완전중복 29그룹 병합 — 무손실(참조 재연결, 소프트삭제).
- R4. 병합은 **재사용 가능한 SSOT RPC**(트랜잭션)로. 경쟁사 병합 RPC(133)와 동형.
- R5. GPU 목록을 공유하는 화면(가격표·시장비교·재고·고객가격표·스펙) 회귀 0.

## 제외
- 폼팩터 SKU에 물리 ×2/4/8 행 생성(정책 위반) — 표시 파생만.
- 레거시 맨-H100/A100의 물리 ×N 행 물리적 붕괴(A100 확정가 유실 위험) — 이번 차수 보류.
- 가격 결정 로직 변경.
